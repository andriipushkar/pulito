# 16. База даних та Prisma

## PostgreSQL через Docker

PostgreSQL запускається автоматично разом з проєктом:

```bash
docker compose up -d
```

За замовчуванням PostgreSQL доступний на порту **5432**. Параметри підключення задаються в `.env`:

```env
DATABASE_URL=postgresql://user:password@localhost:5432/poroshok?schema=public
```

## PgBouncer — пулінг з'єднань

### Що це і навіщо

PgBouncer — це легкий проксі для PostgreSQL, який керує пулом з'єднань до бази. Замість того, щоб кожен запит відкривав нове з'єднання (що дорого по пам'яті), PgBouncer перевикористовує існуючі.

### Навіщо потрібен на дешевому VPS

Дешеві VPS мають обмежену RAM (1–2 GB). Кожне з'єднання PostgreSQL забирає ~5–10 MB. Без пулінгу при 50 одночасних запитах це вже 250–500 MB тільки на з'єднання. PgBouncer вирішує цю проблему, тримаючи 5–10 реальних з'єднань замість 50.

### Режим роботи

Використовується режим **transaction** — з'єднання виділяється тільки на час транзакції, потім повертається в пул. Це найефективніший режим для веб-додатків.

### Налаштування пулу

```env
# Мінімальна кількість з'єднань у пулі (завжди відкриті)
DATABASE_POOL_MIN=2

# Максимальна кількість з'єднань у пулі
DATABASE_POOL_MAX=10
```

**Рекомендації:**
- VPS 1 GB RAM: `MIN=2`, `MAX=5`
- VPS 2 GB RAM: `MIN=2`, `MAX=10`
- VPS 4+ GB RAM: `MIN=5`, `MAX=20`

## Prisma — команди

### Генерація клієнта

```bash
npm run db:generate
```

Генерує Prisma Client на основі `schema.prisma`. Потрібно запускати після кожної зміни схеми та після `npm install`.

### Створення міграції (dev)

```bash
npm run db:migrate
```

Порівнює поточну схему з базою, створює SQL-файл міграції та застосовує її. Prisma запитає назву міграції — давайте зрозумілу назву, наприклад: `add_order_status_field`.

### Застосування міграцій (production)

```bash
npm run db:migrate:prod
```

Застосовує всі наявні, ще не застосовані міграції. **Не створює нових** — тільки виконує існуючі SQL-файли з `prisma/migrations/`.

### Push схеми без міграції (dev only)

```bash
npm run db:push
```

Синхронізує схему з базою напряму, **без створення файлу міграції**. Зручно для швидкого прототипування. **Не використовуйте в продакшні.**

### Скидання бази

```bash
npm run db:reset
```

> **УВАГА: Знищує всі дані!** Видаляє базу, створює заново, застосовує всі міграції та запускає seed.

Використовуйте тільки в розробці, коли потрібно почати з чистого аркуша.

### Seed (заповнення початковими даними)

```bash
npm run db:seed
```

Запускає скрипт `prisma/seed.ts`, який створює:
- **Адмін-користувача** (логін і пароль з `.env`)
- **Категорії товарів** (базова структура каталогу)
- **Зразкові товари** (для тестування інтерфейсу)

### Візуальний перегляд бази

```bash
npm run db:studio
```

Відкриває Prisma Studio у браузері (порт 5555) — візуальний інтерфейс для перегляду і редагування даних у базі.

## Робочий процес міграцій

### Додати нове поле або таблицю

1. Відредагуйте `prisma/schema.prisma`:

```prisma
model Product {
  // ... існуючі поля
  weight Float? // <-- нове поле
}
```

2. Створіть міграцію:

```bash
npm run db:migrate
```

3. Prisma запитає назву — введіть, наприклад: `add_product_weight`

4. Перевірте згенерований SQL у `prisma/migrations/<timestamp>_add_product_weight/migration.sql`

5. Закомітьте зміни:

```bash
git add prisma/schema.prisma prisma/migrations/
git commit -m "Add weight field to Product"
```

### Застосувати на продакшні

```bash
# На сервері
git pull
npm run db:migrate:prod
npm run db:generate
pm2 restart all
```

## Відкат міграцій

Prisma **не має вбудованого механізму відкату** (rollback). Якщо міграція зламала базу:

1. **Відновіть з бекапу** (див. [11-backups.md](11-backups.md))
2. Видаліть проблемну міграцію з `prisma/migrations/`
3. Виправте `schema.prisma`
4. Створіть нову міграцію

> Тому **завжди робіть бекап перед міграцією на продакшні:**

```bash
pg_dump -U user -d poroshok > backup_before_migration.sql
```

## Усунення проблем

| Проблема | Рішення |
|----------|---------|
| `P1001: Can't reach database server` | Перевірте, що PostgreSQL запущений: `docker ps` або `systemctl status postgresql` |
| `P1003: Database does not exist` | Створіть базу: `createdb poroshok` або запустіть `npm run db:push` |
| `P2002: Unique constraint failed` | Дублікат даних. Перевірте seed або міграцію. |
| `Migration failed to apply cleanly` | Бекап → видаліть останню міграцію → виправте → створіть нову |
| Prisma Client застарів | Запустіть `npm run db:generate` після зміни схеми |
| Studio не відкривається | Перевірте, що порт 5555 не зайнятий: `lsof -i :5555` |
