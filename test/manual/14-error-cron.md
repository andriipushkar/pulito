# 14 — Сторінки помилок, offline, health check та cron jobs

---

## 14.1 Сторінки помилок

### TC-EC-1.1 Сторінка 404
**Передумова:** Відкрити браузер
**Кроки:**
1. Перейти на `/nonexistent-page` (неіснуюча URL)
2. Перевірити HTTP статус код — 404
3. Перевірити наявність:
   - Повідомлення "Сторінку не знайдено" або аналогічне
   - Посилання на головну сторінку
   - Пошук (якщо є)
   - Хедер та футер сайту
**Очікуваний результат:** Відображається кастомна сторінка 404 з навігацією та посиланням на головну
**Статус:** ⬜

### TC-EC-1.2 Сторінка 500
**Передумова:** Доступ до серверного оточення або можливість викликати серверну помилку
**Кроки:**
1. Спробувати викликати серверну помилку (якщо є тестовий endpoint або спосіб)
2. Перевірити HTTP статус код — 500
3. Перевірити наявність:
   - Повідомлення "Внутрішня помилка сервера" або аналогічне
   - Посилання на головну
   - Відсутність стек-трейсу у production режимі
**Очікуваний результат:** Відображається кастомна сторінка 500 без технічних деталей, з навігацією
**Статус:** ⬜

### TC-EC-1.3 Maintenance mode
**Передумова:** Увійти як admin
**Кроки:**
1. Перейти в адмін-панель → увімкнути maintenance mode (режим обслуговування)
2. Відкрити сайт в іншому браузері (без авторизації)
3. Перевірити що відображається сторінка "Сайт на обслуговуванні"
4. Перевірити що admin все ще має доступ до сайту
5. Вимкнути maintenance mode
6. Перевірити що сайт знову доступний для всіх
**Очікуваний результат:** При увімкненому maintenance mode звичайні користувачі бачать сторінку обслуговування, адміни мають доступ
**Статус:** ⬜

## 14.2 Offline сторінка

### TC-EC-2.1 Offline сторінка
**Передумова:** Відкрити сайт, дочекатись завантаження Service Worker
**Кроки:**
1. Перейти на `/offline` напряму або вимкнути мережу в DevTools
2. Спробувати перейти на будь-яку сторінку
3. Перевірити що відображається offline сторінка з повідомленням
4. Увімкнути мережу → оновити сторінку
**Очікуваний результат:** Відображається сторінка `/offline` з повідомленням про відсутність з'єднання, після відновлення мережі сайт працює
**Статус:** ⬜

## 14.3 Health check

### TC-EC-3.1 Health check endpoint
**Передумова:** Доступ до API
**Кроки:**
1. Виконати GET запит на `/api/v1/health`
2. Перевірити HTTP статус код — 200
3. Перевірити тіло відповіді:
   - `status`: "ok" або "healthy"
   - Інформація про версію (якщо є)
   - Статус підключення до БД
   - Час роботи (uptime)
**Очікуваний результат:** HTTP 200 з JSON, що містить status info (status: ok, db connection, uptime)
**Статус:** ⬜

## 14.4 Cron jobs

### TC-EC-4.1 Auto-cancel замовлень
**Передумова:** Існує замовлення зі статусом "Нове" створене давніше за таймаут автовідміни
**Кроки:**
1. Створити тестове замовлення (або знайти старе зі статусом "Нове")
2. Дочекатись запуску cron job auto-cancel (або запустити вручну)
3. Перевірити що замовлення змінило статус на "Скасовано"
4. Перевірити запис в логах
**Очікуваний результат:** Замовлення автоматично скасовується після спливання таймауту
**Статус:** ⬜

### TC-EC-4.2 Cleanup tokens
**Передумова:** В БД є прострочені токени (refresh, reset password, email verification)
**Кроки:**
1. Перевірити наявність прострочених токенів у БД (або створити тестовий)
2. Дочекатись запуску cron job cleanup-tokens (або запустити вручну)
3. Перевірити що прострочені токени видалені з БД
4. Перевірити що валідні токени залишились
**Очікуваний результат:** Прострочені токени видалені, валідні — збережені
**Статус:** ⬜

### TC-EC-4.3 Analytics digest
**Передумова:** Налаштована email-розсилка дайджесту аналітики
**Кроки:**
1. Дочекатись запланованого часу відправки дайджесту (або запустити вручну)
2. Перевірити пошту отримувача (admin email)
3. Перевірити що лист містить: KPI за період, графіки або посилання на дашборд
**Очікуваний результат:** Digest email надіслано з актуальними даними аналітики
**Статус:** ⬜

### TC-EC-4.4 DB backup
**Передумова:** Налаштований автоматичний бекап БД
**Кроки:**
1. Дочекатись запланованого часу бекапу (або запустити вручну)
2. Перевірити директорію бекапів (або S3/cloud storage)
3. Перевірити що файл бекапу створено з актуальною датою
4. Перевірити розмір файлу (не порожній)
5. (Опціонально) Спробувати відновити з бекапу на тестовому середовищі
**Очікуваний результат:** Файл бекапу БД створено з коректною датою та непорожнім розміром
**Статус:** ⬜
