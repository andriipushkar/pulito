1. Загальні відомості
   1.1. Мета проєкту
   Створення веб-платформи для гуртово-роздрібного продажу побутової хімії, миючих засобів, порошків та супутніх товарів. Платформа орієнтована переважно на гуртових клієнтів із можливістю роздрібних продажів. Система повинна забезпечити зручний каталог продукції, диференційовану цінову політику залежно від типу клієнта, а також ефективну комунікацію через месенджери.
   1.2. Цільова аудиторія
   • Гуртові покупці (магазини, дистриб’ютори, підприємства) — основна аудиторія
   • Роздрібні клієнти — другорядна аудиторія
   • Менеджери компанії — адміністрування платформи
   1.3. Обмеження поточної фази
   На поточному етапі НЕ реалізуються: інтеграція з платіжними системами (LiqPay, Monobank тощо), інтеграція зі службами доставки (Нова Пошта, УкрПошта, Meest), програма лояльності та накопичувальні знижки, а також система аналітики та звітності. Ці модулі заплановані для подальших фаз розробки.
2. Каталог товарів
   Каталог товарів є основною частиною сайту. Усі сторінки каталогу мають єдину структуру: хедер, контентна частина, футер.
   2.0. Загальна структура сторінки (Layout)
   Кожна сторінка сайту побудована за єдиним каркасом: фіксований хедер зверху, контентна область по центру, футер внизу. Хедер та футер однакові для всіх сторінок і рендеряться як спільні компоненти:
   2.0.1. Хедер сайту
   Хедер фіксований (sticky) і відображається на всіх сторінках сайту:
   • Верхня смуга (top bar): телефон компанії (клікабельний tel:), графік роботи, посилання на Telegram/Viber/Instagram
   • Основна частина: логотип (посилання на головну), рядок пошуку (з автодоповненням), іконки «Обране» (❤️ з лічильником), «Кошик» (🛒 з лічильником товарів та сумою), «Вхід/Кабінет» (👤)
   • Навігаційне меню: горизонтальний список категорій з випадаючими підкатегоріями (mega-menu); на мобільних — гамбургер-меню
   • Міні-кошик (dropdown): при наведенні/кліку на іконку кошика — випадаючий блок з переліком товарів (фото, назва, кількість, ціна), загальна сума, кнопки «До кошика» та «Оформити замовлення»
   • Для гуртівників: додаткова позначка «Гуртовий клієнт» біля імені, ціни відображаються гуртові
   • Адаптивність: на мобільних — логотип + іконки (пошук, кошик, меню); пошук розкривається на повну ширину при кліку
   2.0.2. Футер сайту
   Футер відображається на всіх сторінках та містить:
   • Колонка 1 — Про компанію: логотип, коротка інформація (1-2 речення), посилання на соцмережі (Telegram, Viber, Facebook, Instagram)
   • Колонка 2 — Каталог: посилання на основні категорії товарів
   • Колонка 3 — Інформація: посилання на «Про нас», «Оплата та доставка», «Повернення», «Умови співпраці» (для гуртівників), «FAQ»
   • Колонка 4 — Контакти: адреса, телефон(и), email, графік роботи
   • Форма підписки на розсилку — поле email + кнопка «Підписатися» (детально в розділі 6.3)
   • Нижня смуга (bottom bar): копірайт ©, посилання на Політику конфіденційності та Публічну оферту, іконки платіжних систем
   • Адаптивність: на мобільних колонки складаються вертикально (accordion або повний список)
   2.1. Головна сторінка
   Головна сторінка є вітриною магазину. Порядок блоків налаштовується менеджером (drag-and-drop у кабінеті, розділ 3.4.2). За замовчуванням: 1. Хедер (описаний в розділі 2.0.1) 2. Банер-слайдер з актуальними акціями та спеціальними пропозиціями (управляється через кабінет менеджера, розділ 3.4.3); auto-play з паузою при hover; навігація: стрілки + dots-індикатори; swipe на мобільних 3. Блок переваг (USP) — 3–4 іконки з коротким текстом: «Оригінальна продукція», «Швидка відправка 1–2 дні», «Безкоштовна доставка від 2000₴», «Гуртові ціни від 1 упаковки»; тексти редагуються менеджером 4. Каталог категорій — візуальна сітка з іконками/зображеннями категорій товарів (6 основних); кнопка «Усі категорії» 5. Блок «Акційні товари» — горизонтальна карусель або сітка карток товарів з is_promo = true; заголовок «🔥 Акції» 6. Блок «Новинки» — карусель товарів, відсортованих за датою додавання (DESC); заголовок «✨ Новинки» 7. Блок «Топ продажів» — карусель товарів з найбільшою кількістю замовлень за останні 30 днів; заголовок «🏆 Найпопулярніші» 8. Блок «Нещодавно переглянуті» (розділ 2.2.5) — якщо клієнт вже переглядав товари; приховується для нових відвідувачів 9. Блок брендів/виробників — горизонтальний рядок логотипів (trust-блок); auto-scroll marquee або статична сітка 10. SEO-текстовий блок — HTML-текст внизу сторінки (перед футером); редагується менеджером через кабінет; за замовчуванням: опис магазину, ключові слова для індексації 11. Футер (описаний в розділі 2.0.2)
   2.2. Картка товару
   Кожен товар у каталозі відображається у вигляді картки, яка містить:
   • Фотографія товару (завантажується з окремої папки, де ім’я файлу = код товару з Excel)
   • Назва товару
   • Короткий опис (з таблиці Product_Content, якщо заповнений менеджером; інакше — не відображається)
   • Ціна (роздрібна для клієнтів / гуртова для гуртівників; неавторизовані користувачі бачать роздрібну ціну)
   • Наявність на складі (є/немає або кількість)
   • Позначка «Акція» (якщо товар акційний)
   • Кнопка «Додати до кошика»
   2.2.1. Детальна сторінка товару
   При натисканні на картку відкривається повна сторінка товару з двоколонковим layout:
   • Ліва колонка (50–60%) — галерея зображень: головне фото (масштабується при наведенні — zoom-ефект), рядок thumbnail-мініатюр знизу (клік → зміна головного фото), підтримка свайпу на мобільних, lightbox при кліку (повноекранний перегляд з стрілками навігації)
   • Права колонка (40–50%) — інформаційний блок: назва товару (h1), код товару (SKU), бейджі (акція, новинка, хіт), ціна (роздрібна або гуртова залежно від ролі), стара ціна з відсотком знижки, наявність на складі, кнопка «Додати до кошика» з лічильником кількості (+/–), кнопка «Додати до обраного» (♡), кнопка «Поділитися» (Copy URL, Telegram, Viber, Facebook, Instagram Stories)
   • Вкладки під основним блоком: «Опис» (повний текст з Product_Content, rich-text), «Характеристики» (таблиця: склад, об’єм, вага, призначення, виробник, країна), «Доставка та оплата» (стандартний текст зі статичної сторінки + калькулятор доставки)
   • Хлібні крихти (breadcrumbs): Головна > Категорія > Назва товару (розділ 2.2.6)
   • Блок «Рекомендовані товари» (розділ 2.2.4) — горизонтальна карусель 4–6 карток
   • Блок «Нещодавно переглянуті» (розділ 2.2.5) — горизонтальна карусель
   • Контент з Product_Content заповнюється менеджером поступово через кабінет. Якщо не заповнений — вкладки «Опис» та «Характеристики» приховуються, відображається лише базова інформація з Excel
   • SEO: унікальний title та meta description (автогенерація з шаблонів, розділ 11.4), Schema.org Product розмітка, OG-теги з фото товару
   2.2.2. Система бейджів товарів
   Кожен товар може мати один або декілька візуальних бейджів, що відображаються на картці та детальній сторінці. Бейджі налаштовуються менеджером через кабінет:
   • «Акція» — червоний/кораловий бейдж, автоматично з’являється для товарів з позначкою is_promo в Excel
   • «Новинка» — синій/бірюзовий бейдж, автоматично для товарів, доданих протягом останніх 14 днів (термін налаштовується)
   • «Хіт продажів» — золотий бейдж, автоматично за кількістю замовлень за останній місяць (поріг налаштовується)
   • «Еко» — зелений бейдж, позначається менеджером вручну для екологічних товарів
   • «Вигідна ціна» — помаранчевий бейдж, позначається менеджером
   • Кастомні бейджі — менеджер може створювати власні бейджі з довільним текстом та кольором
   Бейджі відображаються у верхньому лівому куті зображення товару. Максимум 2 бейджі одночасно на картці (пріоритет визначається порядком: Акція > Хіт > Новинка > Еко > інші).
   2.2.3. Відображення цін та історія знижок
   Система підтримує візуальне відображення знижок для стимулювання покупок:
   • Перекреслена стара ціна — для акційних товарів відображається початкова ціна (закреслена) поруч з акційною ціною (виділена кольором --color-cta)
   • Відсоток знижки — автоматично розраховується та відображається як бейдж «-15%» на зображенні товару
   • Історія цін — система зберігає кожну зміну ціни при імпорті Excel із датою зміни (таблиця Price_History)
   • Графік ціни (опціонально) — на детальній сторінці товару можна відобразити графік зміни ціни за останні 3-6 місяців (підвищує довіру покупця)
   • Для гуртівників: відображаються обидві ціни (роздрібна перекреслена + гуртова), що візуально підкреслює вигоду гуртового статусу
   2.2.4. Рекомендовані товари
   Блоки рекомендацій відображаються на детальній сторінці товару та в кошику для збільшення середнього чека:
   • «З цим товаром купують» — на сторінці товару, горизонтальна карусель з 4-8 товарами, що часто замовляються разом (на основі аналізу замовлень)
   • «Схожі товари» — на сторінці товару, товари з тієї ж категорії в аналогічному ціновому діапазоні
   • «Хіти продажів» — на головній сторінці та в кошику, топ-10 найпопулярніших товарів за останній місяць
   • «Доповніть замовлення» — у кошику перед оформленням, товари з категорій, які клієнт ще не додав
   На початковому етапі (при малій кількості замовлень) рекомендації формуються вручну менеджером через зв’язки товарів або автоматично за категорією. З накопиченням даних система переходить на автоматичні рекомендації на основі історії замовлень.
   2.2.5. Нещодавно переглянуті товари
   Блок «Ви нещодавно переглядали» відображається на сторінці товару, головній сторінці та сторінках категорій:
   • Горизонтальна карусель з останніми 10-15 переглянутими товарами (фото, назва, ціна)
   • Для неавторизованих користувачів — дані зберігаються в localStorage браузера
   • Для авторизованих — синхронізація з сервером (таблиця Recently_Viewed: user_id, product_id, viewed_at)
   • При авторизації — дані з localStorage мержаться з серверними (без дублікатів)
   • Автоматичне очищення: записи старше 30 днів видаляються (cron-задача)
   • Блок не відображається, якщо клієнт ще не переглядав жодного товару
   2.2.6. Хлібні крихти (Breadcrumbs)
   Навігаційний ланцюжок відображається на всіх сторінках каталогу для зручності навігації та SEO:
   • Формат: Головна → Категорія → Назва товару
   • На сторінці категорії: Головна → Назва категорії
   • На статичних сторінках: Головна → Назва сторінки
   • Кожен елемент — клікабельне посилання (крім поточної сторінки)
   • Schema.org розмітка BreadcrumbList для коректного відображення в результатах Google
   • Адаптивність: на мобільних показується скорочений варіант (... → Категорія → Товар) при глибокій вкладеності
   2.3. Категорії каталогу
   Товари групуються за категоріями, визначеними у полі «Категорія» Excel-файлу. Орієнтовний перелік категорій:
   • Пральні засоби (порошки, гелі, капсули)
   • Миючі засоби для посуду
   • Засоби для прибирання (підлога, скло, сантехніка)
   • Засоби для кухні
   • Дезінфікуючі засоби
   • Засоби особистої гігієни
   • Освіжувачі повітря
   • Господарські товари
   Категорії формуються автоматично з Excel та управляються менеджером через кабінет (порядок, видимість, іконки, описи, SEO — розділ 3.4.4).
   2.4. Пошук та фільтрація
   Система забезпечує гнучкий пошук та фільтрацію товарів з URL-стратегією для SEO та збереження стану:
   • Текстовий пошук — пошук за назвою та описом товару; fuzzy match (пошук за частиною слова); мінімум 2 символи для початку пошуку; автодоповнення (розділ 2.4.1)
   • Фільтр за категорією — checkbox-список категорій з кількістю товарів у кожній; множинний вибір (кілька категорій одночасно)
   • Фільтр за ціною — range-slider з двома бігунками (від–до) + поля для ручного введення; діапазон визначається автоматично (мінімальна і максимальна ціна в каталозі)
   • Фільтр «Тільки акційні товари» — toggle-перемикач; показує лише товари з is_promo = true
   • Фільтр «Тільки в наявності» — toggle-перемикач; приховує товари з quantity = 0
   • Сортування — dropdown: «За популярністю» (default), «Від дешевих до дорогих», «Від дорогих до дешевих», «За назвою (А–Я)», «Новинки» (за датою додавання)
   • URL-стратегія — усі фільтри відображаються в URL query params: /catalog?category=pralni-zasoby&price_min=100&price_max=500&promo=1&sort=price_asc&page=2; це дозволяє: зберегти посилання, кнопка «Назад» працює коректно, SEO-індексація відфільтрованих сторінок
   • Кнопка «Скинути фільтри» — повертає всі фільтри до початкового стану; відображається тільки коли хоча б один фільтр активний
   • Лічильник результатів — «Знайдено 42 товари» відображається над сіткою; оновлюється при зміні фільтрів
   • Порожній стан — якщо за фільтрами нічого не знайдено: ілюстрація + текст «На жаль, товарів за вашими критеріями не знайдено» + кнопка «Скинути фільтри»
   • Мобільна версія — фільтри згортаються в bottom-sheet або бокову панель з кнопкою «Фільтри (3)» де цифра = кількість активних
   2.4.1. Пошук з автодоповненням (Autocomplete)
   При введенні тексту в рядок пошуку (в хедері) з’являється dropdown з підказками:
   • Тригер: після введення 2+ символів, з затримкою 300мс (debounce) для зменшення навантаження
   • Результати розділені на блоки: «Товари» (до 5 шт) та «Категорії» (до 3 шт)
   • Кожен товар у підказці: мініатюра (thumbnail 50x50), назва (з підсвіченням збігу жирним), ціна, наявність
   • Кожна категорія: назва з кількістю товарів («Пральні засоби (42 товари)»)
   • Клік на товар — перехід на детальну сторінку товару
   • Клік на категорію — перехід до відфільтрованого каталогу
   • Кнопка «Показати всі результати» (або Enter) — перехід на повну сторінку результатів пошуку
   • Якщо нічого не знайдено: «Нічого не знайдено за запитом «{query}». Спробуйте інший запит»
   • Пошук нечутливий до регістру, розпізнає базові помилки (нечіткий пошук / fuzzy search, підготовка)
   • Історія пошуку (для авторизованих): під рядком пошуку при фокусі — останні 5 запитів з кнопкою «✕» для очищення
   • Адаптивність: на мобільних — рядок пошуку розкривається на повну ширину, dropdown займає весь екран
   2.4.2. Сторінка результатів пошуку
   • Заголовок: «Результати пошуку за «{query}» — знайдено {N} товарів»
   • Результати відображаються як стандартні картки товарів (фото, назва, ціна, кнопка «До кошика»)
   • Фільтри та сортування доступні на сторінці результатів (категорія, ціна, наявність, акційні)
   • Пагінація або infinite scroll (підтримка обох варіантів, конфігурується)
   • При 0 результатах: блок «Можливо, вас зацікавить» з популярними товарами

   3.2. Реєстрація та авторизація

   Авторизація відбувається через email та пароль. Передбачено функціонал відновлення пароля через email.
   3.2.1. Соціальний логін (Google Sign-In)
   Додатковий спосіб реєстрації та авторизації через акаунт Google (OAuth 2.0):
   • Кнопка «Увійти через Google» на сторінках реєстрації та авторизації (стандартний Google Identity Services button)
   • При першому вході — автоматичне створення акаунту: email, ім’я та аватар підтягуються з Google-профілю
   • При повторному вході — авторизація за Google ID без введення пароля
   • Прив’язка Google-акаунту до існуючого профілю — якщо email вже зареєстрований, пропонується зв’язати акаунти (з підтвердженням паролем)
   • Відв’язка Google-акаунту — можливість в налаштуваннях профілю (якщо встановлено пароль)
   • Зберігання: поле google_id в таблиці Users; один користувач може мати і пароль, і Google-логін одночасно
   • Телефон та додаткові дані — запитуються після першої авторизації через Google (доповнення профілю)

   • Блок «Мої замовлення» — 2-3 останні замовлення зі статусами та кнопкою «Деталі»
   • Блок «Обране» — мініатюри збережених товарів (до 6 штук) з кнопкою «Усі збережені»
   • Блок «Акції для вас» — 2-3 актуальні акції на товари з категорій, які клієнт купував раніше (персоналізація)
   • Промо-блок «Станьте гуртовим клієнтом!» — коротко переваги (нижчі ціни, персональний менеджер, пріоритетна обробка), кнопка «Дізнатися більше» → сторінка «Умови співпраці»
   • Лічильник непрочитаних сповіщень (badge 🔔)
   • Швидкі посилання: «Мої замовлення», «Обране», «Налаштування профілю»
   Головна для гуртівника:
   • Привітання з назвою компанії: «Вітаємо, {company_name}!»
   • Блок «Останні замовлення» — 3 останні замовлення з ключовою інформацією: номер, дата, сума, кількість позицій, статус
   • Кнопка «Повторити» на кожному замовленні — миттєве додавання всіх товарів до кошика без переходу в історію
   • Блок «Часто замовлені товари» — 5-10 товарів, які гуртівник замовляв найчастіше за останні 3 місяці, з можливістю швидко вказати кількість та додати до кошика
   • Блок «Ваш менеджер» — міні-картка персонального менеджера (фото, ім’я, кнопка Telegram)
   • Блок «Ваші умови» — поточна знижка, мінімальна сума замовлення, до наступного рівня
   • Лічильник непрочитаних сповіщень та кількість товарів у кошику
   • Швидкі посилання: «Швидке замовлення за кодами», «Мої списки», «Скачати прайс-лист», «Фінанси»
   3.3.1. Профіль та дані компанії
   Персональні дані (для всіх типів клієнтів: роздріб + опт):
   • ПІБ, email (підтверджений), телефон
   • Зміна пароля, налаштування сповіщень (email/Telegram)
   • Адреса доставки за замовчуванням (можливість зберігати кілька адрес)
   • Видалення акаунту та всіх даних (відповідно до Закону «Про захист персональних даних»)
   • Аватар — завантаження фото профілю або автоматична генерація з ініціалів
   Додаткові дані компанії (видимі лише для гуртівників):
   • Назва компанії та ЄДРПОУ — обов’язкові для отримання статусу гуртівника
   • Юридична адреса
   • Банківські реквізити (IBAN, назва банку, МФО) — для формування рахунків та накладних
   • Контактна особа для замовлень (якщо відрізняється від власника акаунту)
   • Форма власності (ФОП, ТОВ, ПП тощо)
   • Система оподаткування (з ПДВ / без ПДВ) — впливає на формування PDF-документів
   Реквізити автоматично підставляються в рахунки-фактури та накладні. Менеджер може верифікувати дані при затвердженні статусу гуртівника.
   3.3.2. Історія замовлень
   Доступна для всіх типів клієнтів. Повна історія всіх замовлень з розширеними можливостями:
   • Таблиця замовлень: номер, дата, кількість позицій, сума, статус (кольорове маркування: зелений — виконане, жовтий — в обробці, сірий — скасоване)
   • Фільтрація: за статусом, за періодом (від/до), за діапазоном суми
   • Пошук: за номером замовлення або назвою товару в замовленні
   • Деталі замовлення: перелік товарів з фото, кодами, кількістю, цінами; адреса доставки; коментар; хронологія зміни статусів (таймлайн)
   • Кнопка «Повторити замовлення» — додає всі товари з обраного замовлення до кошика з актуальними цінами та залишками; якщо товар більше недоступний — попередження
   • Кнопка «Друкувати замовлення» — відкриває версію для друку (оптимізований layout без зайвих елементів)
   • Скачування PDF (для гуртівників): рахунок-фактура та видаткова накладна для кожного замовлення (якщо статус ≥ «Підтверджене»). Для роздрібних клієнтів — спрощене підтвердження замовлення
   3.3.3. Фінансовий розділ
   Доступний лише для гуртівників. Зведена фінансова інформація:
   • Загальна сума закупівель: за весь час, за поточний місяць, за поточний квартал
   • Кількість замовлень: загальна та за поточний місяць
   • Середній чек
   • Графік закупівель по місяцях (стовпчаста діаграма) — допомагає гуртівнику аналізувати власні витрати
   • Архів PDF-документів — усі рахунки-фактури та накладні в одному місці з фільтрацією за датою та можливістю масового скачування (ZIP-архів за обраний період)
   • Баланс оплат (підготовка) — структура для майбутньої інтеграції з платіжною системою: скільки оплачено, скільки очікує оплати, заборгованість
   • Акти звірки (підготовка) — формування актів звірки взаєморозрахунків за період для бухгалтерії
   3.3.4. Персональний менеджер
   Доступний лише для гуртівників. Кожному гуртівнику призначається персональний менеджер для оперативного вирішення питань:
   • Блок «Ваш менеджер» у кабінеті: фото, ПІБ, телефон, email
   • Кнопка «Написати в Telegram» — пряме посилання на Telegram менеджера (t.me/username)
   • Кнопка «Зателефонувати» — клікабельне посилання tel: для мобільних
   • Графік роботи менеджера — години, в які можна зв’язатися
   • Менеджер призначається адміністратором при затвердженні статусу гуртівника (поле assigned_manager_id в таблиці Users)
   • Якщо менеджер не призначений — відображаються загальні контактні дані компанії
   3.3.5. Стрічка повідомлень та сповіщень
   Доступна для всіх типів клієнтів. Центр сповіщень у кабінеті — стрічка всіх важливих оновлень:
   • Замовлення: зміна статусу («Ваше замовлення #1234 відправлено»), підтвердження нового замовлення
   • Ціни: зміна ціни на товар зі списку бажань, завершення або початок акції на збережений товар
   • Наявність: повернення в наявність товару зі списку бажань («Товар X знову доступний!»)
   • Акції: нові акції та спеціальні пропозиції (з посиланнями на товари)
   • Системні: зміна статусу гуртівника, оновлення бізнес-правил, нові функції
   • Індикатор непрочитаних сповіщень (badge) у хедері сайту — цифра на іконці дзвіночка
   • Позначення як прочитане (одне або всі), видалення старих сповіщень
   • Налаштування: клієнт обирає які типи сповіщень отримувати та через які канали: email, в кабінеті, Telegram-бот (якщо акаунт прив’язаний, розділ 7.1.1), Viber-бот (якщо прив’язаний, розділ 7.2.1) — окремо для кожного типу (замовлення, ціни, акції, наявність). Зберігається в Users.notification_preferences (JSON)
   3.3.6. Статуси та ліміти гуртівника
   Для роздрібного клієнта:
   • Промо-блок «Станьте гуртовим клієнтом!» — переваги: знижені ціни (приклад: «економія до 30%»), персональний менеджер, пріоритетна обробка замовлень, PDF-документи для бухгалтерії
   • Кнопка «Подати заявку» → форма запиту на статус гуртівника
   • Якщо заявку вже подано — блок «Ваша заявка на розгляді» зі статусом та датою подання
   Для гуртівника:
   • Поточний статус: «Клієнт» / «Очікує підтвердження гуртівника» / «Гуртівник» / «VIP-гуртівник» (підготовка)
   • Форма запиту на статус гуртівника: ЄДРПОУ, назва компанії, вид діяльності, очікуваний обсяг закупівель на місяць, коментар; менеджер отримує сповіщення та затверджує або відхиляє
   • Блок «Ваші умови»: мінімальна сума замовлення, мінімальна кількість (якщо встановлена для окремих товарів), кратність упаковки — гуртівник завжди бачить актуальні правила
   • Прогрес-бар (опціонально): скільки до наступного рівня знижки (якщо шкала знижок активована)
   • Поточна знижка (якщо є) та умови для її збільшення
   • Дата отримання статусу гуртівника та загальний термін співпраці
   3.3.7. Списки бажань та обране
   Доступно для всіх типів клієнтів. Роздрібні клієнти мають одне «Обране», гуртівники — іменовані списки:
   • Кнопка «Додати в обране» (іконка серця/зірки) на кожній картці товару та детальній сторінці
   • Розділ «Обране» в особистому кабінеті — список збережених товарів із поточними цінами та наявністю
   • Можливість створювати іменовані списки (наприклад: «Щомісячне замовлення», «Для офісу», «Для виробництва») — особливо корисно для гуртівників
   • Кнопка «Додати весь список до кошика» — швидке формування типового замовлення
   • Сповіщення при зміні ціни або поверненні в наявність товару зі списку бажань (email, опціонально)
   Для гуртівників ця функція замінює «шаблони замовлень» — зберігши одного разу типовий набір товарів, можна повторювати замовлення в один клік.
   3.3.8. Приватні нотатки до товарів
   Доступно лише для гуртівників. Гуртівник може залишати приватні замітки на будь-якому товарі в каталозі. Нотатки видимі лише власнику:
   • Іконка нотатки (📝) на картці товару та детальній сторінці — при натисканні відкривається текстове поле
   • Приклади використання: «цей бере клієнт Петренко», «замовляти щомісяця по 24 шт», «замінити на BH-042 коли закінчиться», «не брати — повільно продається»
   • Нотатки відображаються у вигляді жовтого стікера під назвою товару (лише для авторизованого власника)
   • Нотатки видимі також у списках бажань та в кошику поруч із відповідним товаром
   • Максимальна довжина: 500 символів
   • Розділ «Мої нотатки» в кабінеті: список усіх товарів із нотатками з можливістю редагування, видалення та пошуку
   3.3.9. Персональні ціни та реферальна програма (підготовка)
   Структура підготовлена для майбутньої реалізації:
   • Персональні ціни — менеджер може встановити індивідуальну знижку або фіксовану ціну для конкретного гуртівника на конкретний товар або категорію (таблиця Personal_Prices: user_id, product_id/category_id, discount_percent або fixed_price)
   • Пріоритет цін: персональна ціна > шкала гуртових знижок > стандартна гуртова ціна > роздрібна ціна
   • Гуртівник бачить свою персональну ціну з позначкою «індивідуальна ціна» на картці товару
   • Реферальна програма — гуртівник отримує унікальне реферальне посилання; якщо новий клієнт реєструється та оформлює перше замовлення за цим посиланням, реферер отримує бонус (знижку на наступне замовлення або грошовий бонус)
   • Розділ «Мої реферали»: список запрошених клієнтів, статус (зареєстрований / зробив замовлення), нарахований бонус
   3.4. Кабінет менеджера
   Кабінет менеджера — адміністративна панель для управління всіма аспектами магазину. Включає dashboard, управління замовленнями, користувачами, товарами, контентом, дизайном, аналітику та звітність (детально в розділі 20), а також журнал дій.
   Навігація кабінету менеджера — бокове sidebar-меню (зліва), фіксоване на desktop, згортається в burger на мобільних:
   • 📊 Dashboard (зведена панель) — завжди перший пункт; розділ 3.4.1
   • 📦 Замовлення — badge з кількістю нових (статус «Нове»); розділ 3.4.6
   • 👥 Користувачі — badge з кількістю нових гуртових запитів; підменю: Усі користувачі, Гуртові запити; розділ 3.4.7
   • 🛒 Товари та контент — підменю: Каталог товарів, Імпорт Excel, Категорії; розділи 3.4.8, 4._, 3.4.4
   • 🌟 Акції та бейджі — розділ 3.4.9
   • 📢 Публікації — підменю: Telegram, Viber, Instagram, Історія публікацій; розділ 3.4.11
   • 📈 Аналітика — підменю: Продажі, Товари, Клієнти, Замовлення, Канали, Склад, Ціни, Воронка; розділ 20
   • 🎨 Дизайн — підменю: Теми, Банери, Налаштування головної; розділи 3.4.13, 3.4.3
   • ✉️ Контент — підменю: Статичні сторінки, FAQ, Email-шаблони, SEO; розділи 3.4.12, 3.4.5
   • ⚙️ Налаштування — підменю: Загальні, Доставка, Оплата, Бізнес-правила; розділи 3.4.2, 3.4.10
   • 📋 Журнал дій — розділ 3.4.15
   • Sidebar: згортання/розгортання з іконками (collapsed mode); підменю розгортається при кліку; активний пункт виділений кольором теми; sticky-позиціонування з внутнім скролом при великій кількості пунктів
   3.4.1. Dashboard (Зведена панель)
   Головна сторінка кабінету менеджера — інтерактивна зведена панель з ключовими метриками бізнесу в реальному часі:
   • Блок «Замовлення сьогодні» — кількість нових замовлень, загальна сума, порівняння з вчора (↑ зелена / ↓ червона стрілка)
   • Блок «Нові замовлення» — кількість необроблених замовлень зі статусом «Нове» з кнопкою швидкого переходу до списку
   • Блок «Оборот за період» — графік продажів за останні 7 / 30 / 90 днів (лінійний або стовпчастий), з перемикачем періоду
   • Блок «Топ-5 товарів» — найпопулярніші товари за кількістю замовлень за останній місяць
   • Блок «Топ-5 клієнтів» — клієнти з найбільшою сумою замовлень за період
   • Блок «Статистика користувачів» — загальна кількість: зареєстрованих, гуртівників, нових за тиждень
   • Блок «Контент» — прогрес-бар наповнення описів товарів (заповнено X з Y), кількість товарів без фото
   • Блок «Зворотний зв’язок» — кількість необроблених повідомлень та запитів на дзвінок
   • Блок «Залишки» — кількість товарів з нульовим залишком, товари з критично низьким залишком (поріг налаштовується)
   • Блок «Останні дії» — стрічка останніх 10 подій системи (нове замовлення, новий користувач, імпорт, публікація)
   Усі блоки оновлюються автоматично (polling кожні 60 секунд або WebSocket у майбутньому). Менеджер може налаштувати розташування та видимість блоків (drag-and-drop компонування).
   3.4.2. Загальні налаштування сайту
   Єдине місце для управління глобальними параметрами магазину:
   • Назва магазину — відображається в хедері, title сторінок, email-листах, PDF-документах
   • Логотип та favicon — завантаження з прев’ю
   • Контактна інформація: телефон(и), email, адреса (відображаються в хедері, футері, на сторінці «Контакти»)
   • Графік роботи — дні тижня та години (відображається на сайті, в Google Business Profile, у ботах)
   • Посилання на соціальні мережі: Telegram-канал, Viber-спільнота, Facebook, Instagram — іконки у футері
   • Реквізити компанії: юридична назва, ЄДРПОУ, ІПН, банківські реквізити, юридична адреса — використовуються в PDF-документах (рахунки, накладні)
   • Валюта та формат цін: символ (₴), кількість знаків після коми, роздільник тисяч
   • Глобальні SEO-налаштування: title за замовчуванням, meta description, OG-зображення за замовчуванням
   • Google Analytics ID, Facebook Pixel ID (підключення зовнішньої аналітики, опціонально)
   • Режим обслуговування (Maintenance Mode) — тимчасове закриття сайту з кастомною сторінкою «Технічні роботи»; доступ лише для менеджерів
   3.4.3. Управління банерами та слайдером
   Менеджер повністю контролює банер-слайдер на головній сторінці:
   • Додавання нового слайду: завантаження зображення (рекомендований розмір: 1920x600px десктоп + 768x400px мобільний), заголовок (опціонально), підзаголовок, кнопка з посиланням
   • Редагування існуючих слайдів: зміна зображення, текстів, посилань
   • Drag-and-drop сортування — зміна порядку відображення слайдів перетягуванням
   • Увімкнення/вимкнення окремих слайдів без видалення
   • Налаштування слайдера: швидкість автопрокрутки (секунди), увімкнення/вимкнення автопрокрутки, нескінченний цикл
   • Прев’ю: попередній перегляд слайдера з усіма змінами перед публікацією
   • Максимальна кількість активних слайдів: 5-7 (налаштовується)
   • Окреме зображення для мобільної версії — менеджер може завантажити адаптовану версію банера для смартфонів
   3.4.4. Управління категоріями
   Хоча категорії автоматично створюються з Excel-файлу при імпорті, менеджер має розширені можливості управління ними через кабінет:
   • Порядок відображення — drag-and-drop сортування категорій на головній сторінці та в меню навігації
   • Видимість — увімкнення/вимкнення категорії (прихована категорія не відображається в каталозі, але товари залишаються в системі)
   • Іконка категорії — завантаження іконки або вибір із вбудованого набору (SVG-іконки) для візуального відображення в сітці категорій на головній
   • Зображення категорії — фонове зображення для шапки сторінки категорії
   • Опис категорії — текстовий блок під заголовком сторінки категорії (WYSIWYG-редактор); корисно для SEO та для пояснення асортименту
   • SEO-поля категорії: seo_title, seo_description, URL-slug (автоматична транслітерація з можливістю ручного редагування)
   • Об’єднання категорій — якщо в Excel є схожі назви («Пральні засоби» і «Засоби для прання»), менеджер може об’єднати їх в одну
   • Підкатегорії (підготовка) — структура БД готова для вкладеності; на поточному етапі підтримується один рівень категорій
   3.4.5. Редагування email-шаблонів
   Менеджер може редагувати тексти всіх автоматичних email-листів без доступу до коду:
   • Список шаблонів: 15 шаблонів, згрупованих за типом — для клієнтів (registration_confirm, welcome, password_reset, order_confirmed_client, order_status_changed, order_shipped, order_cancelled, invoice_created, cart_abandoned), для менеджера (order_new_manager, wholesale_request, feedback_received, daily_digest), для гуртівників (wholesale_approved, wholesale_rejected). Повний опис кожного з переліком змінних — розділ 6.1
   • WYSIWYG-редактор для кожного шаблону з підтримкою змінних: {name}, {order_number}, {order_total}, {new_status}, {tracking_url}, {company_name}, {ttn_number}, {items_table}, {verify_url}, {reset_url} та інші (повний перелік — розділ 6.1)
   • Кнопка «Надіслати тестовий лист» — відправляє лист з тестовими даними на email менеджера для перевірки
   • Прев’ю — попередній перегляд листа з підставленими тестовими даними
   • Версіонування — система зберігає останні 5 версій кожного шаблону з можливістю відкату
   • Базовий HTML-шаблон обгортки: хедер з логотипом, футер з контактами та посиланнями на відписку — єдиний для всіх листів, редагується окремо
   3.4.6. Управління замовленнями
   • Quick status bar — зведена панель вгорі: кількість замовлень по статусах (Нове: 5, В обробці: 3, Відправлене: 12…) з кольоровими бейджами; клік по статусу — фільтрація списку
   • Список замовлень — таблиця з колонками: №, дата, клієнт, тип (роздріб/опт), сума, статус, оплата, доставка; сортування по кожній колонці; пагінація (20/50/100 на сторінку)
   • Пошук та фільтрація — пошук за номером замовлення, ПІБ клієнта, телефоном, кодом товару; фільтри: статус, тип клієнта, період, спосіб доставки, спосіб оплати, діапазон суми; збережені фільтри для менеджера
   • Детальний вигляд замовлення — повна інформація: дані клієнта (ПІБ, телефон, email, тип), перелік товарів (фото, код, назва, кількість, ціна, підсумок), адреса доставки, коментар клієнта, таймлайн зміни статусів (з таблиці Order_Status_History, розділ 23.51)
   • Зміна статусу замовлення — кнопки переходу між дозволеними статусами (матриця — розділ 5.2.2) з автоматичним сповіщенням клієнту; обов’язковий коментар при скасуванні
   • Коментарі менеджера — внутрішнє поле коментаря (не видиме клієнту); відображається в деталях замовлення; зберігається в Orders.manager_comment
   • Bulk-дії — множинний вибір замовлень (checkbox); масова зміна статусу (наприклад, «Відправлене» для всіх підтверджених); масовий друк накладних; масовий експорт
   • Генерація PDF-документів — рахунок-фактура, видаткова накладна, комерційна пропозиція; з реквізитами компанії (із Site_Settings) та клієнта; кнопка «Друкувати» та «Завантажити PDF»
   • Експорт замовлень — вивантаження у Excel (.xlsx) та CSV з урахуванням поточних фільтрів; формат: номер, дата, клієнт, товари, сума, статус, ТТН
   • Шаблони звітів — «Замовлення за місяць», «Топ-товари», «Активні клієнти»; збережені налаштування з кастомними метриками (розділ 20.6)
   3.4.7. Управління користувачами та гуртівниками
   • Список зареєстрованих користувачів — пошук, фільтрація за роллю, датою реєстрації, активністю
   • Перегляд профілю клієнта — контактні дані, дані компанії, історія замовлень, загальна сума закупівель
   • Зміна статусу — Клієнт → Гуртівник → VIP-гуртівник
   • Запити на статус гуртівника — перегляд нових запитів з деталями компанії (ЄДРПОУ, вид діяльності, обсяг), затвердження/відхилення з коментарем
   • Призначення персонального менеджера — вибір зі списку менеджерів при затвердженні гуртівника
   • Персональні ціни (підготовка) — встановлення індивідуальних знижок або фіксованих цін для конкретних гуртівників
   • Експорт списку клієнтів — email, ПІБ, компанія, ЄДРПОУ, тип, дата реєстрації, кількість замовлень, загальна сума
   3.4.8. Управління товарами та контентом
   • Імпорт/оновлення даних з Excel — завантаження файлу, валідація, фоновий імпорт, звіт про помилки (детально в розділі 4)
   • Завантаження зображень — архівом (ZIP) або по одному; автоматичне зіставлення з товарами за кодом у назві файлу
   • Перегляд каталогу — таблиця товарів з індикатором наповнення контентом (є опис / немає опису / немає фото)
   • Редагування контенту товару — WYSIWYG-редактор для опису, характеристик, інструкцій; додавання фото та відео
   • SEO-поля товарів — seo_title, seo_description, alt для зображень
   • Прогрес-бар наповнення — скільки товарів із загальної кількості мають повний контент
   3.4.9. Управління акціями, бейджами та рекомендаціями
   • Акції — позначення акційних товарів, встановлення акційних цін (стара ціна зберігається автоматично)
   • Бейджі — створення кастомних бейджів (текст + колір), призначення бейджів товарам, налаштування автоправил («Новинка»: N днів, «Хіт»: поріг замовлень)
   • Рекомендації — ручне створення зв’язків «З цим купують» між товарами, перегляд автоматичних рекомендацій (на основі історії замовлень)
   • PDF-каталог — генерація прайс-листа (роздрібний / гуртовий) та ілюстрованого каталогу за запитом
   3.4.10. Бізнес-правила гуртових замовлень
   • Мінімальна сума замовлення — глобальна для всіх гуртівників (наприклад, 2000 грн)
   • Мінімальна кількість одиниць — для конкретних товарів або глобально
   • Кратність упаковки — замовлення лише кратне пакуванню (наприклад, по 6 або 12 штук)
   • Шкала гуртових знижок (підготовка) — чим більше сума, тим вища знижка
   • Кредитний ліміт (підготовка) — максимальна сума неоплачених замовлень для довірених гуртівників
   • Усі правила відображаються гуртівнику в кошику та кабінеті (розділ 3.3.6)
   3.4.11. Публікації та месенджери
   • Публікація контенту — створення публікацій для Telegram-каналу, Viber-спільноти, Instagram та сайту; єдина форма з адаптацією під кожну платформу (детально в розділі 7.5)
   • Керування каналами — статистика підписників, історія публікацій з метриками, вітальні повідомлення, автовідповіді, модерація (детально в розділі 7.6)
   • Instagram — підключення акаунту, публікація фото/каруселей, хештеги, аналітика engagement (детально в розділі 7.3)
   3.4.12. Контент та SEO
   • Статичні сторінки — WYSIWYG-редактор для «Про нас», «Контакти», «Умови співпраці», «Оплата та доставка», «Повернення»
   • FAQ — додавання/редагування питань по категоріях (загальні, гуртові, доставка, оплата)
   • Юридичні сторінки — Політика конфіденційності, Угода користувача, Публічна оферта, Повернення
   • SEO — глобальні мета-теги, OG-зображення, SERP preview, robots.txt (детально в розділі 11)
   • SEO-шаблони — редактор шаблонів автогенерації Title, Description, Alt зі змінними ({product_name}, {category}, {price}); окремі шаблони для категорій; масова перегенерація; фільтр «проблемних SEO» (детально в розділі 11.4)
   3.4.13. Дизайн та оформлення
   • Вибір активної теми — три палітри: «Свіжість та Органіка», «Кристальна чистота», «Домашній затишок»
   • Налаштування кольорів (color-picker), шрифтів, whitespace
   • Попередній перегляд теми в iframe
   • Завантаження нових тем (ZIP-архів з CSS-змінними)
   • Детально в розділі 13
   3.4.14. Зворотний зв’язок та сповіщення
   • Перегляд повідомлень від клієнтів — контактна форма, запити на зворотний дзвінок; статуси: новий / оброблений / відхилений
   • Сповіщення — налаштування email та Telegram сповіщень менеджеру (нове замовлення, новий запит, критичні залишки)
   • Експорт даних — вивантаження замовлень, клієнтів, каталогу у формати Excel (.xlsx) та CSV
   3.4.15. Журнал дій (Аудит-лог)
   Система веде детальний журнал усіх значущих дій для прозорості, безпеки та можливості розслідування інцидентів:
   • Фіксуються дії: вхід/вихід менеджера, зміна ролі користувача, імпорт Excel (з результатами), зміна статусу замовлення, створення/редагування публікацій, зміна теми оформлення, редагування статичних сторінок, зміна бізнес-правил гуртівників, видалення даних
   • Для кожного запису зберігається: дата та час, хто виконав (id та ім’я менеджера), тип дії, об’єкт дії (замовлення #, користувач #, товар #), деталі змін (старе значення → нове значення), IP-адреса
   • Інтерфейс: таблиця з фільтрацією за типом дії, періодом, менеджером; пошук за ключовими словами
   • Журнал доступний лише менеджерам; записи не можуть бути видалені або відредаговані (immutable log)
   • Зберігання: останні 12 місяців в БД, архів у файлах (для відповідності вимогам аудиту)
   • Експорт журналу в Excel/CSV за обраний період

   4.2. Зображення товарів
   Фотографії товарів зберігаються в окремій директорії на сервері. Система підтримує кілька способів завантаження та автоматичну обробку:
   • Naming convention — файл зображення має назву, що відповідає коду товару: BH-001.jpg, BH-001.png або BH-001.webp; при імпорті система шукає файл за кодом у визначеній директорії
   • Підтримувані формати — JPG (JPEG), PNG, WebP; рекомендований розмір оригіналу: від 800×800 px; максимальний розмір файлу: 5 МБ; автоматична конвертація в WebP для веб-відображення
   • Способи завантаження — (1) автоматичне зіставлення за кодом при імпорті Excel; (2) ручне завантаження через кабінет менеджера (drag-and-drop або вибір файлу); (3) масове завантаження ZIP-архіву з файлами за кодами
   • Множинні зображення — для одного товару підтримується до 10 фотографій (таблиця Product_Images, розділ 23.46); перше зображення — головне (is_main = true); сортування drag-and-drop в кабінеті менеджера
   • Placeholder — якщо зображення відсутнє, відображається стандартний placeholder з іконкою категорії товару
   4.2.1. Pipeline обробки зображень
   При завантаженні зображення (через імпорт, кабінет менеджера або ZIP-архів) система автоматично створює оптимізовані варіанти:
   • Оригінал — зберігається без змін як резервна копія (для подальшого перегенерування)
   • Full size (800x800) — для детальної сторінки товару та zoom-перегляду
   • Medium (400x400) — для картки товару в каталозі та результатах пошуку
   • Thumbnail (150x150) — для міні-кошика, кабінету (історія замовлень), рекомендацій
   • Lazy placeholder (20x20, blur) — розмита мініатюра для миттєвого відображення під час lazy-loading
   4.2.2. Формати та стиснення
   • Конвертація у WebP — автоматично для всіх розмірів; оригінальний формат (JPG/PNG) зберігається як fallback для старих браузерів
   • Якість стиснення: WebP — 80%, JPEG — 85% (оптимальний баланс розмір/якість)
   • Максимальний розмір файлу: оригінал до 5 МБ; якщо більше — стискається автоматично
   • Назви файлів: {product_code}*{size}.{format} (наприклад: BH-001_400x400.webp)
   • HTML: елемент <picture> з srcset для автоматичного вибору оптимального розміру під пристрій клієнта
   4.2.3. Кешування та CDN (підготовка)
   • HTTP-заголовки: Cache-Control: public, max-age=31536000, immutable для зображень (1 рік)
   • Версіонування: при зміні зображення — нове ім’я файлу з хешем (cache busting)
   • Підготовка до CDN: зображення зберігаються в окремій директорії /uploads/products/ для легкого підключення Cloudflare або S3+CloudFront
   • Lazy-loading: атрибут loading="lazy" для всіх зображень нижче першого viewport; noscript-фолбек для пошукових ботів
   4.3. Процес імпорту
   Імпорт Excel оновлює лише комерційні дані товарів (ціни, залишки, акції). Описи та контент товарів зберігаються окремо в таблиці Product_Content і не зачіпаються при імпорті. 1. Менеджер завантажує Excel-файл через кабінет менеджера 2. Система валідує структуру файлу та вміст полів 3. Нові товари додаються, існуючі (за кодом) оновлюються (ціни, кількість, акція) 4. Контент товарів (описи, характеристики) НЕ перезаписується — зберігається ізольовано 5. Формується звіт про імпорт: додано X, оновлено Y, помилок Z 6. Менеджер завантажує зображення (архів або по одному) до відповідної директорії
5. Система замовлень
   Система замовлень охоплює повний цикл: від додавання товару до кошика до отримання клієнтом.
   5.0. Кошик
   Кошик — центральний елемент процесу покупки. Доступний для авторизованих та неавторизованих користувачів:
   5.0.1. Додавання товару до кошика
   • Кнопка «Додати до кошика» на картці товару та на детальній сторінці — додає 1 одиницю
   • Поле кількості (input + кнопки +/-) на детальній сторінці — клієнт може задати кількість перед додаванням
   • Для гуртівників: валідація кратності упаковки (наприклад, тільки по 6 або 12 шт) з повідомленням «Мінімальна кратність: 6 шт»
   • Анімація підтвердження: коротке повідомлення «✓ Товар додано до кошика» (toast notification, 2-3 секунди)
   • Оновлення лічильника кошика в хедері (кількість товарів + сума) без перезавантаження сторінки (AJAX)
   5.0.2. Сторінка кошика
   Повна сторінка кошика з переліком обраних товарів:
   • Таблиця товарів: фото (мініатюра), назва (посилання на товар), код товару, ціна за одиницю, кількість (редагований input з +/-), сума за позицію
   • Для гуртівників: відображення гуртової ціни, нотатка до товару (якщо є, розділ 3.3.8), позначка кратності
   • Видалення товару — іконка 🗑️ з підтвердженням («Видалити товар з кошика?»)
   • Кнопка «Очистити кошик» — видалення всіх товарів з підтвердженням
   • Перерахунок суми при зміні кількості — миттєво (debounce 300мс)
   • Відображення наявності: якщо товар закінчився після додавання — рядок виділяється червоним з повідомленням «Товар більше недоступний»
   • Якщо ціна змінилась після додавання — повідомлення «Ціна оновлена» з показом старої та нової ціни
   5.0.3. Підсумковий блок кошика
   • Загальна кількість товарів та позицій
   • Проміжна сума (товари без доставки)
   • Знижка (якщо застосовано промокод або гуртова знижка) — окремий рядок з сумою знижки
   • Вартість доставки (розраховується після вибору способу доставки, розділ 5.10)
   • Підсумкова сума (жирним шрифтом, великий розмір)
   • Для гуртівників: блок валідації бізнес-правил — «Мінімальна сума замовлення: 2000 грн. Ваша сума: 1500 грн. Додайте товарів на 500 грн» (кнопка «Оформити» заблокована до виконання мінімуму)
   • Кнопка «Оформити замовлення» — перехід до checkout (розділ 5.1)
   • Кнопка «Продовжити покупки» — повернення до каталогу
   5.0.4. Збереження кошика
   • Неавторизований користувач: кошик зберігається в localStorage браузера; при авторизації — мержиться з серверним кошиком
   • Авторизований користувач: кошик зберігається на сервері (таблиця Cart_Items: user_id, product_id, quantity, added_at)
   • Час життя кошика: серверний — 30 днів без активності; localStorage — без обмежень
   • При поверненні на сайт кошик відновлюється з актуальними цінами та наявністю
   5.0.5. Блок рекомендацій у кошику
   • Під таблицею товарів: блок «Доповніть замовлення» — 4-6 товарів з категорій, які клієнт ще не додав
   • Кнопка «Додати» на кожному рекомендованому товарі — додає без перезавантаження сторінки
   5.1. Процес оформлення замовлення
   Оформлення замовлення реалізовано як покроковий wizard з progress bar зверху. Авторизовані користувачі мають автозаповнення полів з профілю:
   5.1.1. Крок 1: Контактні дані
   • Для авторизованих — дані підтягуються автоматично з профілю, можна редагувати
   • Для неавторизованих — пропозиція увійти або продовжити без реєстрації; при оформленні без реєстрації автоматично створюється акаунт з паролем, надісланим на email
   • Поля: Прізвище* (валідація: мін. 2 символи, кирилиця/латиниця), Ім’я* (аналогічно), По батькові (опціонально), Телефон* (+380 маска, валідація формату), Email* (валідація формату, перевірка унікальності для нового акаунту)
   • Для гуртівників додатково: Назва компанії*, ЄДРПОУ* (8 цифр, валідація контрольної суми)
   5.1.2. Крок 2: Доставка
   • Вибір способу доставки: Нова Пошта (відділення або поштомат), Укрпошта (підготовка), Самовивіз (розділ 5.10)
   • Для авторизованих — вибір зі збережених адрес (User_Addresses) або додати нову
   • Нова Пошта: пошук міста (autocomplete), вибір відділення/поштомату (dropdown з пошуком), автоматичний розрахунок вартості
   • Самовивіз: адреса складу, графік роботи, карта
   5.1.3. Крок 3: Оплата
   • Вибір способу оплати залежить від типу клієнта та суми (розділ 5.9.3)
   • Для гуртівників: оплата на розрахунковий рахунок (генерація рахунку PDF)
   • Для роздрібних: оплата при отриманні (накладений платіж) або передоплата на рахунок
   • Поле «Коментар до замовлення» (textarea, необов’язкове, до 500 символів)
   5.1.4. Крок 4: Підтвердження
   • Summary блок: перелік товарів (фото, назва, кількість, ціна), вартість доставки, загальна сума
   • Контактні дані та адреса доставки — з можливістю повернення на попередній крок для редагування
   • Checkbox «Погоджуюсь з умовами публічної оферти»\* (посилання на оферту)
   • Кнопка «Підтвердити замовлення» — при натисканні: замовлення зберігається зі статусом «Нове», відображається сторінка «Дякуємо!» з номером замовлення, менеджер отримує email + Telegram
   • Sidebar (desktop): фіксований підсумковий блок з сумою, кількістю товарів, вартістю доставки, що супроводжує всі кроки
   5.2. Статуси замовлень
   Замовлення проходить через наступні статуси з чіткими правилами переходів:
   5.2.1. Перелік статусів
   • Нове (new) — колір: синій; щойно створене клієнтом; автоматично присвоюється при оформленні; дія: email + Telegram менеджеру
   • В обробці (processing) — колір: жовтий; менеджер прийняв замовлення в роботу; дія: email клієнту «Ваше замовлення обробляється»
   • Підтверджене (confirmed) — колір: зелений; менеджер узгодив замовлення з клієнтом (наявність, ціна); дія: email клієнту з деталями + рахунок (якщо оплата на рахунок)
   • Оплачене (paid) — колір: бірюзовий; оплата отримана (для замовлень з передоплатою); дія: автоматичний перехід після підтвердження оплати менеджером
   • Відправлене (shipped) — колір: фіолетовий; товар відвантажено; дія: email клієнту з номером ТТН та посиланням на трекінг
   • Виконане (completed) — колір: сірий; клієнт отримав товар; дія: email «Дякуємо за покупку!»
   • Скасоване (cancelled) — колір: червоний; замовлення скасовано будь-якою стороною; дія: email клієнту з причиною скасування
   • Повернення (returned) — колір: оранжевий; клієнт повернув товар; дія: менеджер обробляє вручну
   5.2.2. Матриця дозволених переходів
   Перехід між статусами контролюється на рівні API. Заборонені переходи повертають помилку 400:
   • Нове → В обробці (менеджер), Скасоване (менеджер або клієнт)
   • В обробці → Підтверджене (менеджер), Скасоване (менеджер або клієнт)
   • Підтверджене → Оплачене (менеджер), Відправлене (менеджер, якщо оплата при отриманні), Скасоване (менеджер)
   • Оплачене → Відправлене (менеджер), Скасоване (менеджер, з поверненням оплати)
   • Відправлене → Виконане (менеджер), Повернення (менеджер)
   • Виконане → Повернення (менеджер, протягом 14 днів)
   • Скасоване, Повернення → фінальні статуси, подальші переходи заборонені
   • Клієнт може скасувати замовлення тільки в статусах Нове та В обробці (через кабінет)
   5.2.3. Автоматичні дії при зміні статусу
   • Кожна зміна статусу записується в Order_Status_History (розділ 23.51): хто ініціював (менеджер/клієнт/система), коли, з якого на який статус, коментар; додатково дублюється в Audit_Log для загального аудиту
   • Email-сповіщення клієнту — автоматично при кожній зміні (шаблон order_status_changed)
   • Telegram-сповіщення менеджеру — при створенні нового замовлення та при скасуванні клієнтом
   • Запис в User_Notifications — для стрічки сповіщень у кабінеті клієнта
   • Автоматичне скасування: якщо замовлення в статусі Нове більше 72 годин без дії менеджера — автоматичне скасування з email-сповіщенням (cron-задача)
   5.3. Замовлення в кабінеті менеджера
   Для кожного замовлення менеджер бачить:
   • Номер замовлення та дата створення
   • Дані замовника (ПІБ, телефон, email, компанія)
   • Тип клієнта (роздріб/опт)
   • Перелік товарів із кодами, назвами, кількістю та цінами
   • Загальна сума замовлення
   • Адреса доставки та коментар
   • Кнопки зміни статусу замовлення
   5.4. Бізнес-правила для гуртових замовлень
   Для гуртових клієнтів діють окремі бізнес-правила, які налаштовуються менеджером через кабінет (розділ 3.4.10). Основні правила:
   • Мінімальна сума замовлення — гуртівник не може оформити замовлення нижче встановленої суми (наприклад, 2000 грн); при недосягненні мінімуму відображається попередження з поточною сумою та різницею
   • Мінімальна кількість одиниць товару — для окремих товарів може бути встановлено мінімальне замовлення (наприклад, не менше 6 одиниць); менеджер налаштовує це поле через картку товару
   • Кратність замовлення — деякі товари можна замовляти лише кратно упаковці (наприклад, по 12 штук)
   • Шкала гуртових знижок (опціонально) — підготовлена структура для майбутньої реалізації: чим більше сума замовлення, тим вища знижка
   • Кредитний ліміт (опціонально, підготовка) — максимальна сума неоплачених замовлень для довірених гуртівників
   Усі ліміти та правила відображаються гуртівнику при оформленні замовлення. При порушенні правил замовлення не блокується, але з’являється попередження та пропозиція скоригувати кошик.
   5.5. Документи замовлення (PDF)
   Для кожного замовлення система автоматично генерує документи у форматі PDF, доступні для скачування з кабінету клієнта та менеджера:
   • Рахунок-фактура — з реквізитами компанії, переліком товарів, цінами, загальною сумою, ПДВ (якщо застосовується)
   • Видаткова накладна — формується при зміні статусу замовлення на «Відправлене»
   • Комерційна пропозиція — менеджер може сформувати PDF з підбіркою товарів та гуртовими цінами для надсилання потенційному клієнту
   Шаблони PDF-документів містять логотип компанії, реквізити та відповідають стандартам українського діловодства. Менеджер може налаштувати реквізити компанії (назва, ЄДРПОУ, ІПН, банківські реквізити, юридична адреса) через кабінет.
   5.6. Експорт замовлень
   Менеджер може експортувати дані замовлень для аналізу та обліку:
   • Експорт у Excel (.xlsx) та CSV з фільтрацією: за періодом (від/до), за статусом, за типом клієнта (роздріб/опт), за конкретним клієнтом
   • Експорт списку клієнтів: email, ПІБ, компанія, ЄДРПОУ, тип, дата реєстрації, кількість замовлень, загальна сума
   • Експорт каталогу товарів: поточні залишки, ціни, статус наповнення контентом
   • Шаблони звітів: «Замовлення за місяць», «Топ-товари», «Активні клієнти»
   5.7. Швидке замовлення за кодами товарів (B2B)
   Спеціальний інструмент для гуртових клієнтів, які мають готові списки кодів товарів. Це критично важлива функція для B2B-сегменту, яка суттєво прискорює оформлення великих замовлень:
   • Розділ «Швидке замовлення» доступний з кабінету гуртівника та з головного меню (для авторизованих гуртівників)
   • Текстове поле для масового вводу: гуртівник вставляє список у форматі «код — кількість» (по одному товару на рядок, роздільник — табуляція, крапка з комою або пробіл)
   • Альтернативний ввід: завантаження файлу (.txt, .csv, .xlsx) зі списком кодів та кількостей
   • Система миттєво розпізнає коди, підтягує назви, фото, ціни (гуртові) та залишки
   • Валідація: невідомі коди підсвічуються червоним з пропозицією перевірити; товари з недостатнім залишком — жовтим із зазначенням доступної кількості
   • Можливість коригування кількості безпосередньо в результатах перед додаванням до кошика
   • Кнопка «Додати все до кошика» — усі розпізнані товари додаються одночасно
   Приклад вводу:
   BH-001 24
   BH-015 12
   BH-042 48
   5.8. PDF-каталог та прайс-лист
   Система автоматично генерує PDF-каталог та прайс-лист для скачування та друку:
   5.8.1. Прайс-лист
   • Табличний формат: код товару, назва, категорія, ціна (роздріб/опт), наявність
   • Автоматична генерація при кожному імпорті Excel (завжди актуальний)
   • Два варіанти: роздрібний прайс (для клієнтів) та гуртовий прайс (для гуртівників, доступний після авторизації)
   • Посилання для скачування: на сторінці «Умови співпраці», в кабінеті гуртівника, у футері сайту
   • Формат: PDF з логотипом, датою генерації, контактами та QR-кодом на сайт
   5.8.2. Ілюстрований каталог
   • PDF з фотографіями товарів, описами, кодами та цінами — по 4-6 товарів на сторінку
   • Групування за категоріями зі змістом на першій сторінці
   • Менеджер може обрати: повний каталог, окрема категорія, лише акційні товари
   • Генерація за запитом з кабінету менеджера (фонова задача через чергу, оповіщення по готовності)
   • QR-код на кожному товарі — посилання на сторінку товару на сайті для швидкого замовлення
   5.9. Оплата
   Система підтримує кілька способів оплати з різною логікою для роздрібних та гуртових клієнтів:
   5.9.1. Способи оплати
   • Накладений платіж (Нова Пошта) — основний спосіб для роздрібних клієнтів; клієнт оплачує при отриманні на відділенні або кур’єру; комісія Нової Пошти відображається як інформаційне повідомлення при виборі цього способу
   • Оплата на розрахунковий рахунок — основний спосіб для гуртівників; менеджер формує рахунок-фактуру (PDF) з реквізитами компанії та клієнта; гуртівник оплачує через банк; менеджер вручну підтверджує оплату в кабінеті
   • Онлайн-оплата карткою (підготовка) — інтеграція з LiqPay або Monobank Acquiring; структура БД та інтерфейс кошика підготовлені для підключення платіжного шлюзу; callback-ендпоінт для підтвердження оплати
   • Передоплата на картку (опціонально) — менеджер надає реквізити картки; клієнт оплачує та повідомляє менеджера; менеджер підтверджує вручну
   5.9.2. Статуси оплати
   • Очікує оплати — замовлення створено, оплата не надійшла
   • Оплачено — оплата підтверджена (автоматично через платіжний шлюз або вручну менеджером)
   • Часткова оплата (підготовка) — для великих гуртових замовлень з розбивкою на транші
   • Повернення коштів — замовлення скасовано після оплати, кошти повертаються
   Статус оплати відображається окремо від статусу замовлення: замовлення може бути «Відправлене» зі статусом оплати «Накладений платіж». Менеджер бачить обидва статуси в панелі замовлень.
   5.9.3. Логіка вибору способу оплати
   • Роздрібний клієнт бачить: накладений платіж, передоплата на картку, онлайн-оплата (коли підключено)
   • Гуртівник бачить: оплата на розрахунковий рахунок (основний), накладений платіж (опціонально)
   • Менеджер може обмежити доступні способи оплати через налаштування сайту (розділ 3.4.2)
   • При виборі «Оплата на рахунок» автоматично формується PDF-рахунок з реквізитами обох сторін
   5.10. Доставка
   Система підтримує інтеграцію з основними поштовими операторами України та самовивіз:
   5.10.1. Способи доставки
   • Нова Пошта (основний) — доставка на відділення або адресна доставка кур’єром
   • Укрпошта — доставка на відділення або за адресою; для клієнтів у невеликих населених пунктах
   • Самовивіз — зі складу/офісу компанії; адреса відображається на карті (Google Maps embed)
   • Палетна доставка (підготовка) — для великих гуртових замовлень; окремий тариф, узгоджується з менеджером
   5.10.2. Інтеграція з Новою Поштою (API)
   Інтеграція через офіційний API Нової Пошти (api.novaposhta.ua):
   • Пошук міста — автодоповнення назви міста при введенні (мінімум 2 символи), метод searchSettlements
   • Пошук відділення / поштомату — список відділень у вибраному місті з фільтрацією (відділення / поштомат / вантажне), метод getWarehouses
   • Розрахунок вартості доставки — відображається в кошику при виборі Нової Пошти, на основі ваги та габаритів замовлення, метод getDocumentPrice
   • Створення ТТН (для менеджера) — автоматичне формування товарно-транспортної накладної з кабінету менеджера; метод save (InternetDocument); дані відправника підтягуються з налаштувань сайту
   • Трекінг статусу доставки — автоматичне оновлення статусу замовлення при зміні статусу ТТН (метод getStatusDocuments); polling кожні 2 години для активних замовлень
   • Відображення трекінг-номеру клієнту — в деталях замовлення, з посиланням на сторінку відстеження на сайті Нової Пошти
   5.10.3. Інтеграція з Укрпоштою (підготовка)
   Базова підтримка Укрпошти без глибокої API-інтеграції на першому етапі:
   • Вибір способу доставки «Укрпошта» в кошику
   • Введення адреси вручну (місто, вулиця, індекс) або номер відділення
   • Вартість доставки: фіксована сума (налаштовується менеджером) або «за тарифами Укрпошти» (інформаційне повідомлення)
   • Підготовка: API StatusTracking для відстеження статусу (інтеграція в наступних фазах)
   5.10.4. Самовивіз
   • Вибір «Самовивіз» в кошику — вартість доставки = 0 грн
   • Відображення адреси складу/офісу з міні-картою (Google Maps embed або посилання)
   • Графік роботи пункту самовивозу (підтягується з налаштувань сайту)
   • Менеджер отримує сповіщення «Замовлення для самовивозу»
   5.10.5. Вибір доставки в кошику
   • Крок 1: Вибір способу доставки (Нова Пошта / Укрпошта / Самовивіз)
   • Крок 2: Для НП — пошук міста → вибір відділення/адресна; для Укрпошти — введення адреси; для самовивозу — підтвердження
   • Крок 3: Автоматичний розрахунок вартості (для НП) або відображення фіксованої вартості
   • Збережені адреси — авторизований клієнт може вибрати зі списку своїх адрес (розділ 3.3.1) замість повторного введення
   • Гуртівник бачить додатковий варіант «Палетна доставка (узгодити з менеджером)»
6. Система сповіщень
   6.1. Email-сповіщення
   Автоматичні email-листи надсилаються через чергу Notification_Queue. Тексти всіх шаблонів редагуються менеджером через кабінет (розділ 3.4.5). Кожен шаблон має унікальний код (template_code):

   • Telegram Bot API через webhook (POST /api/webhooks/telegram); верифікація secret_token
   • Бібліотека: node-telegram-bot-api (Node.js) або python-telegram-bot / aiogram (Python)
   • Обробка оновлень: message (текстові команди та пошук), callback_query (натискання inline-кнопок), inline_query (inline-режим пошуку)
   • Медіа: фото товарів надсилаються через sendPhoto з URL (CDN); для каруселей — sendMediaGroup (до 10 фото)
   • Антифлуд: обмеження 1 callback_query / секунду на користувача; answerCallbackQuery з cache_time=5
   • Помилки: при недоступності API — повідомлення потрапляє в Notification_Queue з retry; при помилці sendMessage — логування + алерт менеджеру
   • Зберігання стану: chat_id + поточний «стан» (browsing_catalog, searching, viewing_order) у Redis з TTL 1 година; для складних flow — Finite State Machine

   • Viber REST API через webhook (POST /api/webhooks/viber); верифікація через X-Viber-Content-Signature (HMAC-SHA256 з auth_token)
   • Бібліотека: viber-bot (Node.js) або viberbot (Python)
   • Типи повідомлень: text (текстові команди), picture (фото товарів), rich_media (каруселі та Rich Media), keyboard (persistent меню), contact, url

   • Обмеження: broadcast-повідомлення (масова розсилка) — не більше 500 отримувачів за запит; для більших аудиторій — пакетна відправка з паузою
   • Статуси доставки: delivered (доставлено на пристрій), seen (прочитано) — зберігаються для аналітики публікацій (розділ 7.6.2)
   • Зберігання стану: аналогічно Telegram — viber_user_id + стан у Redis; Viber не підтримує inline_query, тому пошук тільки через текстові повідомлення
   • Fallback при помилках: якщо Rich Media не підтримується (старі версії Viber) — відправка простого текстового повідомлення з посиланням
   7.2.8. Viber-спільнота (Community)
   • Окрема Viber-спільнота для публікацій та обговорень (аналог Telegram-каналу)
   • Публікації з кабінету менеджера (розділ 7.5) автоматично дублюються в спільноту
   • Модерація: бот може виконувати роль адміністратора — видалення спаму, автовідповіді (розділ 7.6.4)
   • Запрошення: QR-код та deep link для вступу в спільноту, розміщуються на сайті та в публікаціях
   7.3. Instagram (публікації через API)
   Інтеграція з Instagram через Graph API дозволяє менеджеру публікувати контент в Instagram безпосередньо з кабінету, без необхідності заходити в додаток:
   7.3.1. Підключення Instagram
   • Вимоги: Instagram Professional Account (Business або Creator) + підключена Facebook Page
   • Авторизація: OAuth 2.0 через Facebook Login; менеджер натискає «Підключити Instagram» у кабінеті → перенаправлення на Facebook → надання дозволів (instagram_basic, instagram_content_publish, pages_read_engagement)
   • Після підключення: в кабінеті відображається ім’я профілю, аватар, кількість підписників; токен зберігається у Site_Settings (зашифрований)
   • Long-lived token: автоматичне оновлення кожні 60 днів (cron-задача); алерт менеджеру за 7 днів до закінчення якщо автооновлення не вдалося
   7.3.2. Типи публікацій
   • Одне фото — зображення товару або банер акції; підпис (caption) до 2200 символів з хештегами; геолокація (опціонально)
   • Карусель (до 10 фото) — підбірка товарів або колекція: кожне фото = окремий товар з фото; один спільний caption з описом усіх товарів
   • Reels (відео, підготовка) — короткі відео-огляди товарів; завантаження MP4 (до 90 секунд, до 1GB); thumbnail з відео
   • Stories — НЕ підтримується через API (обмеження Instagram); менеджер публікує Stories вручну через додаток
   7.3.3. Форма публікації в Instagram
   • Зображення — завантаження або вибір з каталогу товарів; автоматичний crop до формату 1:1 (1080×1080) або 4:5 (1080×1350) з можливістю ручного кадрування; прев’ю результату
   • Caption (підпис) — текстове поле з лічильником символів (макс. 2200); підтримка emoji, line breaks, згадок (@username)
   • Хештеги — окреме поле або inline в caption; автопропозиція хештегів з раніше використаних; рекомендація: до 30 хештегів (ліміт Instagram)
   • Вибір товарів — прив’язка публікації до конкретних товарів (для аналітики конверсії); посилання на товар у caption: «Посилання в bio!» або прямий URL
   • Перший коментар — опція автоматично додати перший коментар з хештегами (поширена Instagram-практика для чистого caption)
   • Прев’ю — mock-рендер як виглядатиме пост у стрічці Instagram (фото + caption + хештеги)
   7.3.4. Шаблони Instagram-публікацій
   • Шаблон «Товар дня» — 1 фото товару + caption: назва, ціна, короткий опис, CTA («Посилання в bio!»), набір тематичних хештегів
   • Шаблон «Акція» — банер-зображення + caption: опис акції, стара/нова ціна, терміни, хештеги #акція #знижка
   • Шаблон «Підбірка» — карусель 3–10 товарів + caption з описом кожного + загальний CTA
   • Шаблон «Behind the scenes» — фото/відео процесу роботи, складу, команди; для формування довіри
   • Менеджер може зберігати власні шаблони з набором хештегів та структурою caption
   7.3.5. Планування та автопостинг
   • Негайна публікація — натискання «Опублікувати зараз»; API створює media container → публікує → повертає permalink
   • Запланована публікація — вибір дати та часу; збереження в чергу; cron-задача публікує у призначений час
   • Найкращий час для публікації (підготовка) — система рекомендує оптимальний час на основі статистики охоплення попередніх постів
   • Автопостинг акцій (підготовка) — при створенні акції на сайті автоматично генерується Instagram-пост зі зображенням товару та caption з акційною ціною
   7.3.6. Аналітика Instagram
   • Метрики профілю (Instagram Insights API): підписники (динаміка), охоплення, покази, відвідування профілю, кліки по посиланню в bio
   • Метрики кожної публікації: likes, comments, saves, shares, reach, impressions; зберігаються в Channel_Stats
   • Engagement Rate: (likes + comments + saves) / reach × 100% — для кожного поста та середній по профілю
   • Конверсія: скільки переходів на сайт з bio-посилання (UTM-мітка); скільки з них привели до замовлення
   • Порівняння каналів: Instagram vs Telegram vs Viber — зведена таблиця в дашборді аналітики (розділ 20.4)
   7.3.7. Технічна реалізація Instagram
   • Instagram Graph API v18+ (Content Publishing API): POST /{ig-user-id}/media → POST /{ig-user-id}/media_publish
   • Для каруселей: створення кожного item (POST /media з is_carousel_item=true) → створення carousel container → publish
   • Зображення: повинні бути доступні за публічним URL (CDN); підтримувані формати: JPEG, PNG; макс. 8MB; мін. 150×150, макс. 1920×1080
   • Rate Limits: 25 API-запитів на Instagram-акаунт за 24 години для публікацій (Content Publishing); 200 запитів на годину для читання
   • Помилки: при перевищенні rate limit — retry через 1 годину; при невалідному зображенні — повідомлення менеджеру з деталями; при expired token — алерт та інструкція переавторизації
   • Webhook (підготовка): Instagram може надсилати webhook при нових коментарях — для автовідповідей та модерації
   • Зберігання: ig_media_id, permalink, метрики — в таблиці Publications (додаткові поля: ig_media_id, ig_permalink, ig_insights_json)
   7.4. Синхронізація Telegram, Viber та Instagram
   Єдиний інтерфейс для роботи з усіма месенджерами та соцмережами з кабінету менеджера:
   • Усі публікації через кабінет надсилаються одночасно у вибрані канали (Telegram, Viber, Instagram, сайт — будь-яка комбінація)
   • Адаптація контенту: один текст автоматично форматується під специфіку кожної платформи — Markdown для Telegram, Rich Media для Viber, caption + хештеги для Instagram
   • Зображення: одне фото автоматично адаптується — оригінал для Telegram, resize до 400×400 для Viber каруселей, crop до 1:1 (1080×1080) для Instagram
   • Кнопки/посилання: InlineKeyboard для Telegram, Buttons у Rich Media для Viber, URL у caption («Посилання в bio!») для Instagram
   • Статистика: зведена аналітика по всіх каналах в єдиному дашборді (розділ 7.6.1)
   • Прив’язка акаунту: один користувач може мати прив’язку до Telegram та Viber одночасно; сповіщення надходять у вибраний канал (або обидва)
   • Календар публікацій — візуальний календар у кабінеті: показує заплановані та опубліковані пости по всіх каналах; drag-and-drop для перепланування
   7.5. Функціонал публікацій з кабінету менеджера
   Менеджер створює та керує публікаціями через спеціальний редактор в кабінеті. Одна публікація — одночасна відправка у вибрані канали:
   7.5.1. Форма створення публікації
   • Заголовок публікації (до 100 символів) — відображається жирним у повідомленні
   • Текст — rich-text редактор з підтримкою: жирний, курсив, посилання, списки, emoji; автоматична конвертація у Markdown (Telegram), plain text (Viber) та caption з хештегами (Instagram)
   • Зображення — завантаження нового або вибір з каталогу товарів (підтягує фото товару); автоматичний resize під вимоги месенджерів; прев’ю перед відправкою
   • Посилання на товар або категорію — dropdown з пошуком по каталогу; автоматично генерує URL з UTM-мітками
   • Кнопки (до 3) — менеджер додає inline-кнопки: текст + URL; приклади: «Переглянути акцію», «Купити зараз», «Весь каталог»
   • Вибір каналів: checkbox «Telegram-канал», «Viber-спільнота», «Instagram», «Сайт (Новини та акції)» — можна обрати будь-яку комбінацію; при виборі Instagram автоматично додається поле хештегів
   7.5.2. Шаблони публікацій
   • Шаблон «Новий товар» — автозаповнення: фото товару, назва, ціна, опис, кнопка «Переглянути»; менеджер обирає товар з каталогу
   • Шаблон «Акція» — автозаповнення: фото, стара/нова ціна, відсоток знижки, кнопка «Купити зі знижкою»
   • Шаблон «Новина» — вільний формат: заголовок + текст + зображення
   • Шаблон «Дайджест» — підбірка товарів (3–5): кілька фото + короткий опис кожного + загальна кнопка «До каталогу»
   • Менеджер може зберігати власні шаблони для повторного використання
   7.5.3. Планування та статуси
   • Негайна публікація — кнопка «Опублікувати зараз»; підтвердження: «Відправити в Telegram + Viber + Instagram?» (відображаються лише обрані канали)
   • Запланована публікація — вибір дати та часу; публікація потрапляє в чергу; cron-задача відправляє в призначений час
   • Чернетка — збереження без відправки для подальшого редагування
   • Статуси: Чернетка → Запланована → Опублікована; або Чернетка → Опублікована (негайна)
   • Помилка публікації — якщо API месенджера недоступне: статус «Помилка», автоматичний retry через 5 хвилин (до 3 спроб), сповіщення менеджеру
   • Прев’ю перед відправкою — менеджер бачить як виглядатиме повідомлення в Telegram, Viber та Instagram (mock-рендер для кожної платформи)
   7.6. Керування каналами (Telegram, Viber, Instagram)
   Кабінет менеджера надає централізований інтерфейс для управління присутністю магазину в месенджерах та соцмережах без необхідності заходити в Telegram, Viber або Instagram окремо.
   7.6.1. Статистика каналів
   Панель статистики для кожного підключеного каналу (Telegram-канал, Telegram-група, Viber-спільнота, Instagram-профіль):
   • Поточна кількість підписників/учасників з динамікою за обраний період (графік: день/тиждень/місяць)
   • Нові підписники та відписки за період — окремі лінії на графіку для розуміння чистого приросту
   • Джерела підписок (якщо доступно через API) — з сайту, з QR-коду, за запрошенням, органічно
   • Активність групи — кількість повідомлень від учасників за період (для груп з увімкненими коментарями)
   • Порівняння каналів — зведена таблиця: Telegram vs Viber vs Instagram, кількість підписників, активність, охоплення, конверсія в замовлення
   • Автоматичне оновлення статистики кожні 6 годин або за запитом менеджера
   7.6.2. Історія публікацій з метриками
   Менеджер бачить повну історію всіх публікацій з детальною аналітикою по кожній:
   • Таблиця публікацій: дата, заголовок, канали (Telegram/Viber/Instagram/сайт), статус (опублікована/запланована/чернетка/помилка)
   • Метрики Telegram (через Bot API / Channel Stats): кількість переглядів, кількість пересилань (forward), реакції
   • Метрики Viber: кількість доставлених повідомлень, кількість прочитаних (seen), кількість кліків по кнопках
   • Метрики сайту: кількість переглядів публікації на сторінці «Новини та акції»
   • UTM-аналітика: скільки переходів на сайт згенерувала публікація, скільки з них привели до замовлення (конверсія)
   • Сортування та фільтрація: за датою, каналом, типом (акція/новина/новий товар), ефективністю
   • Кнопка «Повторити публікацію» — створює копію для редагування та повторної відправки
   • Кнопка «Видалити з каналу» — видаляє опубліковане повідомлення з Telegram/Viber/Instagram (якщо API підтримує)
   7.6.3. Вітальне повідомлення для нових підписників
   Автоматичне привітання нових учасників каналу/групи:
   • Для Telegram-бота: при першому натисканні /start надсилається налаштовуване вітальне повідомлення
   • Для Telegram-групи (якщо бот є адміністратором): привітання нового учасника з інструкцією та посиланням на каталог
   • Для Viber-бота: вітальне повідомлення при підписці з кнопками швидкого доступу («Каталог», «Акції», «Зв’язатися з менеджером»)
   • Менеджер налаштовує через кабінет: текст привітання (з підтримкою емодзі та форматування), зображення (опціонально), набір inline-кнопок
   • Можливість A/B тестування: два варіанти привітання, система випадково обирає та збирає метрики (який варіант краще конвертує)
   • Автоматична відправка промо-коду або посилання на акцію новому підписнику (опціонально, вмикається менеджером)
   7.6.4. Автовідповіді та модерація
   Система автоматичних відповідей та модерації для ботів та груп:
   Автовідповіді для ботів:
   • Менеджер створює набір пар «ключове слово / фраза → відповідь» через кабінет
   • Приклади: «ціна» → «Перегляньте наш каталог: [посилання]», «доставка» → «Інформація про доставку: [посилання]»
   • Пріоритет відповідей: спочатку точне співпадіння, потім часткове, потім відповідь за замовчуванням
   • Відповідь за замовчуванням — якщо бот не розпізнав запит: «Дякуємо за повідомлення! Ось що я можу: [меню команд]. Або зв’яжіться з менеджером: [контакт]»
   • Меню команд бота: /catalog, /promo, /order, /help, /contact — тексти та дії кожної команди налаштовуються менеджером
   Модерація груп:
   • Антиспам: автоматичне видалення повідомлень з посиланнями від нових учасників (перші 24 години або N повідомлень)
   • Стоп-слова: список заборонених слів/фраз; повідомлення з ними автоматично видаляються з повідомленням автору
   • Обмеження частоти: не більше N повідомлень за M хвилин від одного учасника (захист від флуду)
   • Сповіщення менеджеру: при спрацюванні модерації менеджер отримує повідомлення з деталями (хто, що написав, яке правило спрацювало)
   • Журнал модерації: історія всіх дій модерації з можливістю перегляду видалених повідомлень та розблокування помилково заблокованих
   • Чорний список: учасники, заблоковані за порушення правил; менеджер може додавати/видаляти вручну
   Налаштування графіку роботи бота:
   • В робочий час (налаштовується): бот відповідає одразу та може переключити на менеджера
   • Поза робочим часом: бот повідомляє графік роботи та пропонує залишити контакт для зворотного зв’язку
8. Архітектура проєкту
   8.1. Принципи архітектури
   Проєкт будується за принципами модульної архітектури, що забезпечує легке масштабування, підтримку та розширення функціоналу без переписування існуючого коду. Кожен функціональний блок (каталог, замовлення, користувачі, сповіщення, месенджери) реалізується як незалежний модуль з чітко визначеним API.
   8.1.1. Модульна структура
   Backend поділяється на ізольовані модулі, кожен з яких відповідає за свою доменну область:
   • Модуль Auth — реєстрація, авторизація, ролі, сесії
   • Модуль Catalog — товари, категорії, пошук, фільтрація
   • Модуль Content — описи товарів, медіа-файли, SEO
   • Модуль Orders — кошик, замовлення, статуси
   • Модуль Import — парсинг Excel, валідація, синхронізація даних
   • Модуль Notifications — email, Telegram, Viber сповіщення
   • Модуль Messenger — боти Telegram та Viber, публікації в канали, керування каналами (статистика, автовідповіді, модерація, вітальні повідомлення)
   • Модуль Theme — система тем оформлення, шаблони дизайну
   • Модуль Admin — кабінет менеджера, управління платформою

   8.1.2. Підготовка до масштабування
   Архітектура з самого початку враховує майбутнє зростання:
   • REST API з версіонуванням (v1/v2) — дозволяє оновлювати API без порушення роботи існуючих клієнтів
   • Роздільні шари: презентація (frontend), бізнес-логіка (backend), дані (БД) — кожен шар можна масштабувати незалежно
   • Черга задач (Redis Queue / Bull) — для фонових операцій: імпорт Excel, розсилка email, публікації в месенджери
   • Кешування (Redis) — каталог, сесії, часті запити; інвалідація кешу при оновленні даних
   • Контейнеризація (Docker) — кожен сервіс в окремому контейнері, Docker Compose для оркестрації
   • Міграції БД — версіонування схеми бази даних для безпечних оновлень
   • Environment-конфігурація — усі налаштування (ключі API, підключення БД, SMTP) через змінні оточення, жодних хардкодів

   • Дати та валюта — форматування через бібліотеку Intl (JavaScript) або аналог; валюта на поточній фазі — тільки UAH (₴)
   • Адмін-панель — залишається тільки українською, мультимовність стосується лише публічної частини сайту

   8.5. Стандарти коментування коду
   Код повинен бути самодокументованим (зрозумілі назви змінних і функцій), але критичні місця обов’язково коментуються:
   8.5.1. Обов’язкове документування (JSDoc / docstrings)
   • Кожен модуль (файл) — блок-коментар на початку з описом призначення модуля, автора та дати створення
   • Кожен клас — JSDoc / docstring з описом призначення, залежностей та прикладів використання
   • Кожен публічний метод / функція — опис, параметри (@param з типами), значення що повертається (@returns), винятки (@throws)
   • Кожен API-ендпоінт — Swagger/OpenAPI анотації: URL, метод, параметри, тіло запиту, відповіді (200, 400, 401, 404, 500), приклади
   • Кожна міграція БД — коментар з описом змін, причиною та rollback-стратегією
   • Конфігураційні файли — коментарі до кожного параметра з поясненням та допустимими значеннями
   8.5.2. Формат JSDoc (JavaScript/TypeScript)
   Приклад документування функції:
   • /\** Розраховує фінальну ціну товару з урахуванням типу клієнта та поточних акцій. @param {Product} product - Товар для розрахунку. @param {UserType} userType - Тип клієнта ('retail' | 'wholesale'). @param {Promotion[]} activePromotions - Масив активних акцій. @returns {PriceResult} Об’єкт з фінальною ціною, розміром знижки та джерелом знижки. @throws {ProductNotAvailableError} Якщо товар не в наявності. @example calculateFinalPrice(product, 'wholesale', promotions) // { price: 250, discount: 15, source: 'wholesale' } \_/
   8.5.3. Формат docstrings (Python)
   Використовується Google-стиль docstrings:
   • Перший рядок — короткий опис (одне речення). Далі — розгорнутий опис (якщо потрібно). Блоки Args:, Returns:, Raises:, Example:
   8.5.4. Інлайн-коментарі (inline comments)
   • Складна бізнес-логіка — пояснити ЧОМУ, а не ЩО робить код (код показує що, коментар пояснює чому)
   • Формули та розрахунки — коментар з посиланням на бізнес-правило або ТЗ (// Розрахунок за правилом з розділу 5.4 ТЗ)
   • Workaround та хакі — обов’язковий коментар // HACK: або // WORKAROUND: з поясненням причини та посиланням на issue/задачу
   • TODO / FIXME — стандартний формат: // TODO(author): Опис задачі [#issue-number]. Заборонено залишати TODO без issue
   • Не коментувати очевидний код — // increment counter → i++ (антипатерн, заборонено)
   • Мова коментарів — англійська для коду, українська допускається для бізнес-правил

   8.7. Реєстр системних cron-задач
   Повний перелік фонових задач, що виконуються за розкладом. Усі задачі логуються в структуровані логи та мають алерти при збої:
   • Агрегація аналітики воронки — щоночі о 02:00; обробляє Client_Events за добу, записує в Daily_Funnel_Stats
   • Очищення покинутих кошиків — щоночі о 03:00; видаляє Cart_Items старше 30 днів для неавторизованих, 90 днів для авторизованих
   • Резервне копіювання БД — щоденно о 03:30 (повний pg_dump); кожні 6 годин (інкрементальний WAL-backup)
   • Очищення прострочених токенів — щоночі о 04:00; видаляє Refresh_Tokens з expires_at < NOW() - 7 днів
   • Обробка черги сповіщень — кожні 30 секунд; бере pending з Notification_Queue, відправляє, retry при помилці
   • Перевірка статусів НП — кожні 2 години (08:00–20:00); оновлює трекінг відправлень через API Нова Пошта
   • Генерація sitemap.xml — щоночі о 05:00; оновлює карту сайту з усіма товарами та категоріями
   • Очищення тимчасових файлів — щоночі о 05:30; видаляє файли з /tmp та невикористані upload-файли старше 24 годин
   • Перевірка здоров’я зовнішніх сервісів — кожні 5 хвилин; перевіряє SMTP, Telegram API, Viber API, Instagram Graph API, НП API; алерт при недоступності > 15 хвилин
   • Щотижневий email-дайджест (підготовка) — понеділок о 09:00; надсилає менеджеру зведення за тиждень
   • Автоскасування протермінованих замовлень — щогодини; скасовує замовлення в статусі «Нове» без дії менеджера понад 72 години; надсилає email клієнту з причиною «Автоматичне скасування через відсутність підтвердження» (розділ 5.2.3)
   • Відправка запланованих публікацій — щохвилини; перевіряє Publications зі статусом scheduled та scheduled_at ≤ NOW(); відправляє у відповідні канали (Telegram, Viber, Instagram); оновлює статус на published або failed (розділ 7.5.3)
   • Збір статистики каналів (Telegram/Viber) — кожні 6 годин; опитує Telegram Bot API (getChatMembersCount) та Viber API; записує в Channel_Stats (розділ 7.6.1)
   • Оновлення Instagram Long-Lived Token — кожні 50 днів (запас до закінчення 60-денного TTL); запит на /oauth/access_token?grant_type=ig_exchange_token; при помилці — алерт менеджеру за 7 днів до закінчення токена
   • Збір Instagram Insights — щоденно о 06:00; оновлює ig_insights_json для публікацій за останні 7 днів; записує метрики профілю в Channel_Stats

   8.9. Версіонування API
   Стратегія версіонування забезпечує зворотну сумісність при розвитку API:
   • Поточна версія: /api/v1/ — усі ендпоінти мають префікс версії
   • Підхід: URL-based versioning (/api/v1/, /api/v2/) як найбільш прозорий
   • Зворотна сумісність: при додаванні нових полів у відповідь — існуючі клієнти не ламаються (additive changes)
   • Breaking changes: нова мажорна версія API (/api/v2/) з паралельною підтримкою v1 мінімум 6 місяців
   • Deprecation policy: заголовок Sunset з датою відключення; попередження в Swagger; email-сповіщення споживачам API
   • Версія у відповіді: заголовок X-API-Version: v1 для прозорості
   8.10. Специфікація Webhook-ендпоінтів
   Webhook-ендпоінти для прийому зворотних викликів від зовнішніх сервісів:
   • POST /api/webhooks/telegram — прийом оновлень від Telegram Bot API; верифікація через secret_token у заголовку X-Telegram-Bot-Api-Secret-Token
   • POST /api/webhooks/viber — прийом подій від Viber Bot; верифікація через підпис у заголовку X-Viber-Content-Signature (HMAC-SHA256)
   • POST /api/webhooks/nova-poshta (підготовка) — оновлення статусу відправлень; верифікація по IP whitelist НП
   • POST /api/webhooks/instagram (підготовка) — прийом подій від Instagram (нові коментарі, згадки); верифікація через hub.verify_token при підписці + X-Hub-Signature-256 (HMAC-SHA256 з app_secret) для кожного запиту
   • POST /api/webhooks/payment (підготовка) — callback від платіжної системи; верифікація по підпису + IP whitelist
   • Загальні вимоги: всі webhook-ендпоінти відповідають 200 OK протягом 5 секунд; обробка — в черзі (Notification_Queue); логування всіх вхідних запитів; retry-захист через ідемпотентність (перевірка update_id / event_id)
9. Безпека та захист даних
   9.1. Захист авторизації та сесій
   • Хешування паролів алгоритмом bcrypt з salt (cost factor ≥ 12)
   • JWT-токени для авторизації з коротким терміном дії access-токена (15–30 хвилин) та довготривалим refresh-токеном
   • Зберігання refresh-токенів у httpOnly cookies з прапорцями Secure та SameSite=Strict
   • Обмеження спроб авторизації: максимум 5 невдалих спроб на 15 хвилин (rate limiting), далі — тимчасове блокування з CAPTCHA
   • Автоматичний logout при тривалій бездіяльності (30 хвилин для менеджерів)
   • Логування всіх дій авторизації (вхід, вихід, невдалі спроби) з IP-адресою
   9.2. Захист від атак
   • HTTPS (TLS 1.2+) для всіх з’єднань, автоматичний редирект з HTTP
   • Захист від XSS — санітизація всіх вхідних даних, Content-Security-Policy заголовки, екранування виводу
   • Захист від CSRF — CSRF-токени для всіх мутуючих запитів (POST, PUT, DELETE)
   • Захист від SQL-ін’єкцій — параметризовані запити через ORM (Prisma / Sequelize / SQLAlchemy), заборона raw-запитів з конкатенацією
   • Захист від брутфорсу — rate limiting на рівні API (100 запитів/хвилину для звичайних, 10/хвилину для авторизації)
   • Захист завантаження файлів — перевірка MIME-type, обмеження розміру (Excel до 10MB, зображення до 5MB), зберігання поза публічною директорією
   • HTTP Security Headers: X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, Strict-Transport-Security
   9.3. Захист даних
   • Розмежування доступу за ролями (RBAC) — перевірка прав на кожному ендпоінті API
   • Клієнт не має доступу до гуртових цін, кабінету менеджера та адмін-функцій
   • Гуртівник бачить лише свої замовлення та гуртові ціни
   • Менеджер має доступ до адмін-панелі, але не може змінювати системні налаштування (роль адміністратора)
   • Шифрування чутливих даних у БД (email, телефон — за потреби)
   • Регулярне автоматичне резервне копіювання БД (щоденно, зберігання 30 днів)
   • Логування критичних дій (зміна ролей, видалення даних, імпорт) з аудит-трейлом (розділ 3.4.15, таблиця Audit_Log)
   9.4. Безпека API та інтеграцій
   • API-ключі для Telegram, Viber ботів та Instagram Graph API зберігаються у змінних оточення, не в коді
   • CORS-політика — лише дозволені домени
   • Валідація всіх вхідних даних на стороні сервера (Joi / Zod / Pydantic) незалежно від клієнтської валідації
   • Обмеження розміру тіла запиту (body limit)
   • Захист від DDoS на рівні reverse-proxy (Nginx) та CDN (Cloudflare)
   9.4.1. Конфігурація Rate Limiting
   Детальна таблиця лімітів запитів по групах ендпоінтів:
   • Авторизація (POST /auth/login, /auth/register) — 10 запитів / 15 хвилин на IP; при перевищенні: 429 Too Many Requests + CAPTCHA
   • Відновлення пароля (POST /auth/reset-password) — 3 запити / 60 хвилин на email
   • Каталог (GET /products, /categories, /search) — 120 запитів / хвилину на IP (для ботів/парсерів)
   • Кошик та замовлення (POST /cart, /orders) — 60 запитів / хвилину на авторизованого користувача
   • Імпорт Excel (POST /admin/import) — 5 запитів / годину на менеджера (захист від подвійного імпорту)
   • Webhook-ендпоінти (/webhooks/\*) — без ліміту (верифікація за підписом)
   • Загальний ліміт API — 300 запитів / хвилину на IP для неавторизованих; 600 / хвилину для авторизованих
   • Реалізація: middleware на рівні додатку (express-rate-limit / slowapi) + Nginx limit_req_zone на рівні reverse-proxy
   • Заголовки відповіді: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset (UNIX timestamp)
   9.5. Резервне копіювання та відновлення (Backup & Disaster Recovery)
   Стратегія резервного копіювання забезпечує збереження даних та швидке відновлення системи у разі збою:
   9.5.1. Регулярні бекапи
   • База даних PostgreSQL: повний бекап щоденно о 03:00 (pg_dump), інкрементальні бекапи кожні 6 годин
   • Зображення товарів та медіа-файли: синхронізація з S3-сумісним сховищем щоденно
   • Конфігурація сервера та Docker-образи: бекап при кожній зміні (Infrastructure as Code)
   • Зберігання: останні 7 щоденних + 4 щотижневих + 3 щомісячних бекапів
   • Географічна розподіленість: бекапи зберігаються в іншому дата-центрі або регіоні
   9.5.2. Цільові показники відновлення
   • RPO (Recovery Point Objective) — максимальна допустима втрата даних: 6 годин (відстань між інкрементальними бекапами)
   • RTO (Recovery Time Objective) — максимальний час відновлення: 2 години для повного відновлення системи
   • Для критичного збою БД: відновлення з останнього бекапу за 30 хвилин
   • Для збою сервера: перезапуск контейнерів на резервному сервері за 15 хвилин (Docker Compose pull + up)
   9.5.3. Процедура відновлення
   • Документована покрокова інструкція відновлення для кожного типу збою: БД, сервер, мережа, storage
   • Автоматичний health-check кожні 5 хвилин; при 3 послідовних збоях — алерт менеджеру (email + Telegram)
   • Щомісячне тестування відновлення з бекапу на staging-середовищі (drill test)
   • Логування всіх бекапів та відновлень з верифікацією цілісності (checksums)
   9.6. Стратегія міграції даних (початкове наповнення)
   План перенесення існуючих даних при першому запуску системи у production:
   9.6.1. Джерела даних
   • Товари — імпорт з існуючого Excel-файлу через стандартний модуль імпорту (розділ 4)
   • Категорії — створення через seed-скрипт або ручне додавання в кабінеті менеджера
   • Клієнти (якщо є база) — міграційний скрипт з CSV/Excel: email, телефон, тип (роздріб/опт); пароль — тимчасовий, з обов’язковим скиданням при першому вході
   • Зображення — пакетне завантаження з директорії; автоматична обробка через image pipeline (розділ 4.2.1)
   • Статичні сторінки — ручне наповнення через кабінет менеджера
   9.6.2. Процедура міграції
   • Крок 1: Розгортання системи на staging з порожньою БД; запуск міграцій
   • Крок 2: Імпорт категорій (seed-скрипт); перевірка структури
   • Крок 3: Імпорт товарів через Excel; верифікація кількості, цін, зображень
   • Крок 4: Імпорт клієнтської бази (якщо є); відправка welcome-email з посиланням на скидання пароля
   • Крок 5: Наповнення контенту: статичні сторінки, FAQ, банери, налаштування
   • Крок 6: Smoke-тестування на staging: замовлення, оплата, сповіщення, пошук
   • Крок 7: Деплой на production; повторний імпорт з фінальним Excel; DNS-переключення
   • Rollback: повне видалення даних та повторний імпорт з нуля (система stateless щодо legacy)
10. Продуктивність та оптимізація
    10.1. Цільові показники швидкодії
    • Час завантаження головної сторінки (TTFB): до 200 мс
    • Largest Contentful Paint (LCP): до 2.5 секунд
    • Cumulative Layout Shift (CLS): до 0.1
    • First Input Delay (FID): до 100 мс
    • Час відповіді API для типових операцій: до 300 мс
    • Обробка імпорту Excel (до 5000 позицій): до 30 секунд (фоновий процес)
    • Одночасна робота до 200 користувачів без деградації продуктивності
    10.2. Стратегії оптимізації
    Серверна оптимізація: кешування каталогу та категорій у Redis (TTL 5–15 хвилин з інвалідацією при імпорті), індексація БД за полями code, category, is*promo, created_at для швидких вибірок, пагінація всіх списків (товари, замовлення, користувачі) з обмеженням 20–50 елементів на сторінку, фоновий імпорт Excel через чергу задач (не блокує основний потік), connection pooling для PostgreSQL.
    Клієнтська оптимізація: lazy-loading зображень товарів (Intersection Observer API), стиснення зображень з підтримкою WebP (автоматична конвертація при завантаженні), мініфікація та бандлінг CSS/JavaScript, серверний рендеринг (SSR) для сторінок каталогу (SEO + швидке відображення), CDN для статичних ресурсів (зображення, CSS, JS), Service Worker для кешування статики та офлайн-заглушки.
    10.2.1. Стратегія кешування Redis
    Детальна специфікація ключів кешу, TTL та стратегії інвалідації:
    • products:list:{page}:{sort}:{filters_hash} — кешована сторінка каталогу; TTL: 10 хвилин; інвалідація: при імпорті Excel, при зміні товару менеджером → FLUSH products:list:*
    • product:{id} — детальна інформація товару з Product*Content; TTL: 5 хвилин; інвалідація: при зміні товару → DEL product:{id}
    • categories:tree — дерево категорій з кількістю товарів; TTL: 30 хвилин; інвалідація: при зміні категорій або імпорті → DEL categories:tree
    • search:autocomplete:{query_prefix} — результати автодоповнення; TTL: 15 хвилин; інвалідація: при імпорті → FLUSH search:*
    • cart:{session_id} — кошик неавторизованого користувача; TTL: 30 днів; інвалідація: при оформленні замовлення або очищенні кошика
    • session:{token_hash} — дані сесії авторизованого користувача; TTL: 24 години; інвалідація: при logout
    • rate_limit:{ip}:{endpoint} — лічильник rate limiting; TTL: залежно від ендпоінту (1–60 хвилин); автоматичне видалення
    • Загальна стратегія: Cache-Aside pattern (перевірка кешу → якщо miss → запит до БД → запис у кеш); максимальний розмір Redis: 256MB (eviction policy: allkeys-lru)
    10.2.2. Стратегія пагінації
    Різні типи пагінації залежно від контексту використання:
    • Каталог товарів — offset-based: ?page=2&per_page=24; UI: номери сторінок з «...» (1 2 3 ... 12); SEO: rel='prev'/'next' заголовки для індексації; per_page за замовчуванням: 24 (desktop), 12 (mobile)
    • Замовлення, сповіщення (стрічки) — cursor-based: ?cursor={last_id}&limit=20; UI: кнопка «Завантажити ще»; більш продуктивний для великих списків з частими вставками
    • Аналітика (таблиці) — offset-based з вибором per_page (10/25/50/100); UI: таблиця з пагінатором знизу
    • Мобільні пристрої — infinite scroll (підготовка): автозавантаження при досягненні кінця списку; з fallback на кнопку «Завантажити ще»
    • API відповідь: { data: [...], meta: { page, per_page, total, total_pages, has_next } } для offset; { data: [...], meta: { cursor, has_next, limit } } для cursor
    10.3. Моніторинг та логування
    Комплексна система моніторингу забезпечує раннє виявлення проблем та швидке реагування:
    10.3.1. Application Performance Monitoring (APM)
    • Трекінг часу відповіді кожного API-ендпоінту з розбивкою: загальний час, час БД, час зовнішніх API
    • Автоматичне сповіщення при деградації: відповідь > 1с для критичних ендпоінтів (catalog, cart, checkout)
    • Dashboard з графіками: p50, p95, p99 latency, RPS (requests per second), error rate
    • Інструменти: Sentry Performance, New Relic APM, або Prometheus + Grafana (self-hosted)
    10.3.2. Структуроване логування
    • Формат: JSON-логи з полями timestamp, level, module, message, requestId, userId, duration
    • Рівні: error (збої, критичні помилки), warn (підозрілі дії, rate limit), info (бізнес-події: замовлення, імпорт), debug (деталі для розробки, тільки dev)
    • Request ID (correlation ID) — унікальний ідентифікатор запиту через усі сервіси для трейсингу
    • Інструменти: Winston / Pino (Node.js) або structlog (Python)
    • Ротація логів: зберігання 30 днів, автоматичне архівування
    10.3.3. Error Tracking
    • Автоматичний збір помилок з backend та frontend (Sentry або аналог)
    • Групування помилок за типом, модулем, частотою; автоматичне призначення пріоритету
    • Контекст помилки: stack trace, request body, userId, browser info, OS
    • Сповіщення: Telegram менеджеру при нових critical errors; email-дайджест щоденно
    • Source maps для frontend — коректні stack traces у production
    10.3.4. Health Checks та Uptime
    • GET /api/health — перевірка стану: app (up/down), database (connected/disconnected), redis (connected/disconnected), disk space
    • GET /api/health/detailed — розширена перевірка (тільки для менеджера): час відповіді БД, розмір черги, кількість активних з’єднань
    • Зовнішній uptime-моніторинг (UptimeRobot / Better Uptime) — перевірка кожні 60 секунд
    • Сповіщення при даунтаймі: Telegram + SMS (якщо налаштовано)
11. SEO-оптимізація
    11.1. Технічне SEO
    Сайт повинен бути повністю оптимізований для індексації пошуковими системами Google, Bing та іншими з моменту запуску:
    • Серверний рендеринг (SSR) або статична генерація (SSG) для всіх публічних сторінок — критично для індексації контенту пошуковими ботами
    • Чисті семантичні URL: /category/pralni-zasoby, /product/bh-001-poroshok-pralnyi замість /product?id=123
    • Автоматична генерація sitemap.xml з усіма сторінками каталогу, категоріями та товарами; оновлення при імпорті Excel
    • Файл robots.txt з правилами індексації: дозвіл публічних сторінок, заборона кабінетів, кошика, адмін-панелі
    • Канонічні URL (rel=canonical) для запобігання дублювання контенту при фільтрації та сортуванні
    • Коректна обробка 404 сторінок з пошуком та посиланнями на популярні категорії
    • 301-редиректи при зміні URL товару або видаленні товару з каталогу
    • Швидкість завантаження: Core Web Vitals в зеленій зоні (LCP < 2.5с, CLS < 0.1, FID < 100мс)
    • Мобільна оптимізація — сайт повинен проходити Google Mobile-Friendly Test
    11.2. On-Page SEO
    Для кожної сторінки забезпечуються наступні мета-елементи:
    • Title (<title>) — унікальний для кожної сторінки; генерується автоматично за шаблонами (розділ 11.4) або задається вручну
    • Meta Description — унікальний опис до 160 символів; генерується автоматично за шаблонами або задається вручну
    • Open Graph теги (og:title, og:description, og:image, og:url) — для коректного відображення при шерінгу в соцмережах та месенджерах
    • Twitter Card теги (twitter:card, twitter:title, twitter:image) — для красивих прев’ю у Twitter/X
    • Семантична HTML-розмітка: h1 (один на сторінку), h2-h6 для структури, <article>, <nav>, <main>, <aside>
    • Alt-атрибути для всіх зображень товарів — генеруються автоматично за шаблоном (розділ 11.4.1), або задаються вручну через seo_description
    • Хлібні крихти (breadcrumbs) з розміткою BreadcrumbList — Головна > Категорія > Товар
    11.3. Структуровані дані (Schema.org)
    JSON-LD розмітка для покращення відображення у пошуковій видачі (Rich Snippets):
    • Product — для кожного товару: name, description, image, sku (код товару), offers (ціна, валюта, наявність)
    • Organization — інформація про компанію: назва, логотип, контакти, соціальні профілі
    • BreadcrumbList — хлібні крихти для навігації
    • WebSite з SearchAction — для відображення пошукового рядка в Google
    • ItemList — для сторінок категорій (список товарів)
    • Offer / AggregateOffer — ціни та спеціальні пропозиції для акційних товарів
    11.4. Автогенерація SEO та шаблони
    SEO-поля генеруються автоматично за шаблонами. Менеджер може перевизначити будь-яке поле вручну — ручне значення завжди має пріоритет. Принцип: «автоматично для тисяч товарів, вручну — для ключових».
    11.4.1. Шаблони автогенерації (налаштовуються менеджером)
    В кабінеті менеджера (розділ 3.4.12) доступний редактор SEO-шаблонів з підтримкою змінних:
    • Змінні для товарів: {product_name}, {category}, {price}, {brand}, {shop_name}, {sku}
    • Змінні для категорій: {category_name}, {parent_category}, {product_count}, {shop_name}
    • Змінні для статичних сторінок: {page_title}, {shop_name}
    Шаблони за замовчуванням (вбудовані, менеджер може змінити):
    • Title товару: «{product_name} — купити {category} | {shop_name}»
    • Meta Description товару: «{product_name} за ціною від {price} грн. {category} з доставкою по Україні. Гуртом та в роздріб. {shop_name}»
    • Alt зображення: «{product_name} — фото {N}» (де N — порядковий номер фото)
    • URL slug товару: транслітерація {product_name} за стандартом КМУ 2010 (наприклад, «Гель для прання Persil» → gel-dlya-prannya-persil)
    • Title категорії: «{category_name} — купити гуртом та в роздріб | {shop_name}»
    • Meta Description категорії: «{category_name} ({product_count} товарів) — ціни від виробника, доставка по Україні. {shop_name}»
    11.4.2. Пріоритет значень
    • Пріоритет 1 (найвищий): ручне значення — менеджер заповнив поле seo_title або seo_description в редакторі товару
    • Пріоритет 2: шаблон категорії — якщо для категорії задано окремий шаблон (наприклад, для «Пральні засоби» інший формат Title)
    • Пріоритет 3 (найнижчий): глобальний шаблон за замовчуванням — застосовується до всіх товарів/категорій, де немає ручного або категорійного значення
    Система позначає в каталозі менеджера джерело SEO: іконка «✍️ ручний» або «🤖 авто» біля кожного поля.
    11.4.3. Масове управління SEO
    • Масова автогенерація — кнопка «Перегенерувати SEO для всіх товарів» (застосовує поточні шаблони до товарів без ручних значень)
    • Перегенерація при імпорті — при імпорті нових товарів з Excel SEO-поля генеруються автоматично
    • Масовий SERP-прев’ю — таблиця всіх товарів із відображенням Title + Description + URL як у видачі Google
    • Фільтр «Проблемні SEO» — товари з Title > 60 символів, Description > 160 символів, відсутнім alt, дублями Title
    • Експорт/імпорт SEO-полів через Excel — колонки: код товару, seo_title, seo_description, slug; менеджер може масово редагувати в Excel та завантажити назад
    11.4.4. SEO-поля в кабінеті менеджера (ручне перевизначення)
    Менеджер може вручну задати SEO для конкретного товару, категорії або сторінки:
    • Для кожного товару (через Product_Content): seo_title, seo_description, slug — якщо заповнені, мають пріоритет над шаблоном
    • Для категорій: назва, опис, мета-теги, власний шаблон SEO для товарів цієї категорії
    • Глобальні налаштування: назва сайту, опис за замовчуванням, OG-зображення за замовчуванням
    • Налаштування robots.txt через інтерфейс (без доступу до файлів сервера)
    • Перегляд прев’ю: як сторінка виглядатиме в Google (SERP preview) та при шерінгу в месенджерах
    11.5. Технічні SEO-вимоги до реалізації
    • Транслітерація URL: назви категорій та товарів автоматично конвертуються в латиницю (українська транслітерація за стандартом КМУ 2010)
    • Lazy-loading зображень з noscript-фолбеком для пошукових ботів
    • Пагінація каталогу з rel=prev/next та параметром page в URL
    • hreflang-теги (підготовка до мультимовності)
    • Автоматичне стиснення HTML (мініфікація пробілів)
    • Відсутність дублювання контенту: фільтри додають параметри до URL, але канонічний URL вказує на базову сторінку категорії
    11.6. Google Business Profile
    Інтеграція з Google Business Profile для покращення локальної видимості в пошуку:
    • Створення та верифікація профілю компанії в Google Business з повною інформацією: назва, адреса, телефон, графік роботи, фото
    • На сторінці «Контакти» — вбудована карта Google Maps з маркером компанії
    • Schema.org розмітка LocalBusiness / Store з координатами, графіком роботи та контактами — для відображення в Knowledge Panel Google
    • Посилання на Google Maps з кнопкою «Прокласти маршрут»
    • Підготовка до збору відгуків: кнопка «Залишити відгук у Google» на сторінці подяки після замовлення
    11.7. QR-коди
    Система генерує QR-коди для з’єднання офлайн- та онлайн-каналів продажу:
    • QR-код на кожному товарі в PDF-каталозі та прайс-листі — сканування переводить на сторінку товару на сайті
    • QR-код на друкованих матеріалах (візитки, флаєри) — на головну сторінку або на акцію
    • QR-код з посиланням на Telegram-бот та Viber-спільноту — для друку на пакуванні або рекламних матеріалах
    • Автоматична генерація через бібліотеку qrcode (Node.js) або python-qrcode; формат: SVG або PNG
    • UTM-мітки в URL QR-кодів для відстеження ефективності офлайн-каналів

    • Лічильник переходів з кожного каналу (Telegram, Viber, Instagram, прямий трафік) — базова статистика в кабінеті менеджера

    • Канали поширення: Копіювати URL (з toast «Посилання скопійовано»), Telegram (share URL через t.me/share/url?url=), Viber (viber://forward?text=), Facebook (sharer.php), Instagram Stories (deep link instagram://story-camera або fallback на копіювання URL з підказкою «Вставте посилання в Stories», підготовка)

    • Мобільні пристрої: використання Web Share API (navigator.share) якщо підтримується браузером — нативне меню поширення ОС; fallback на кнопки каналів
    • Лічильник поширень (підготовка) — збирається через Channel_Visits для аналітики популярності товарів
13. Система тем та дизайну
    Візуальний дизайн сайту реалізований через систему тем, яка дозволяє змінювати зовнішній вигляд магазину без втручання в код.
    13.1. Архітектура системи тем
    Весь візуальний дизайн сайту винесений в окрему систему тем, повністю відділену від бізнес-логіки. Це дозволяє змінювати зовнішній вигляд магазину без втручання в код, а також створювати та перемикати різні варіанти дизайну.
    13.1.1. Структура файлів теми
    Кожна тема зберігається в окремій папці з чіткою структурою:
    • /themes/{theme-name}/theme.json — маніфест теми: назва, автор, версія, опис, скріншот прев’ю
    • /themes/{theme-name}/styles/ — CSS/SCSS файли: змінні кольорів, типографіка, компоненти, адаптивність
    • /themes/{theme-name}/templates/ — шаблони сторінок (якщо тема перевизначає структуру): header, footer, картка товару, сторінка каталогу
    • /themes/{theme-name}/assets/ — статичні ресурси теми: іконки, фонові зображення, шрифти
    • /themes/{theme-name}/preview.jpg — скріншот дизайну для вибору в кабінеті менеджера
    13.1.2. CSS-змінні та Design Tokens
    Кожна тема визначає набір CSS Custom Properties (Design Tokens), які контролюють увесь візуальний стиль сайту:
    • Основні кольори: --color-primary, --color-secondary, --color-accent, --color-cta (кнопки дії)
    • Фони: --color-bg-main, --color-bg-card, --color-bg-header, --color-bg-footer, --color-bg-promo
    • Текст: --color-text-primary, --color-text-secondary, --color-text-on-primary
    • Типографіка: --font-family-heading, --font-family-body, --font-size-base, --line-height-base
    • Відступи та розміри: --spacing-unit, --border-radius, --container-max-width, --card-shadow
    • Компоненти: --button-bg, --button-hover, --input-border, --badge-promo-bg, --badge-promo-text
    Завдяки цьому підходу зміна однієї змінної автоматично оновлює всі елементи інтерфейсу, що її використовують.
    13.2. Управління темами через кабінет менеджера
    В кабінеті менеджера розділ «Дизайн та оформлення» надає наступний функціонал:
    13.2.1. Вибір активної теми
    • Галерея доступних тем з візуальними прев’ю (скріншотами)
    • Кнопка «Попередній перегляд» — показує сайт з обраною темою в iframe без її активації
    • Кнопка «Активувати» — застосовує тему для всіх відвідувачів
    • Можливість повернутися до попередньої теми в один клік
    13.2.2. Налаштування поточної теми
    • Візуальний редактор кольорів — color-picker для зміни основних кольорів (primary, cta, bg, text) без редагування CSS
    • Вибір палітри — швидке перемикання між трьома попередньо налаштованими палітрами: «Свіжість та Органіка», «Кристальна чистота», «Домашній затишок»
    • Завантаження власного логотипу та favicon
    • Зміна фонового зображення банера
    • Вибір шрифтів з рекомендованого переліку (Montserrat, Poppins, Nunito, Inter, Open Sans) або повного каталогу Google Fonts
    • Налаштування відображення каталогу: кількість карток в ряду (2-4), стиль карток (компактні/розширені), розмір whitespace
    • Перемикач «Темна тема» / «Світла тема» (якщо тема підтримує обидва режими)
    13.2.3. Завантаження нових тем
    Менеджер може завантажити нову тему у вигляді ZIP-архіву. Система валідує наявність обов’язкових файлів (theme.json, базові стилі), перевіряє безпеку (заборона скриптів у шаблонах), та додає тему до галереї після успішної валідації.
    13.3. Попередньо встановлені теми
    Система поставляється з трьома ретельно підібраними палітрами, оптимізованими для маркетингу побутової хімії та максимального сприйняття користувачами:
    13.3.1. Тема «Свіжість та Органіка» (за замовчуванням, найтрендовіша)
    Гама підкреслює натуральність, безпеку та екологічність. Викликає довіру та відчуття затишку. Створює образ магазину, де продають «розумну» та безпечну хімію.
    • --color-primary: М’ятний / Світло-бірюзовий (#78C4AA) — основний колір, асоціюється зі свіжістю та чистотою
    • --color-bg-main: Теплий білий / Milk White (#FAF9F6) — фон сайту, не «стерильний» і не холодний, на відміну від чистого білого
    • --color-text-primary: Глибокий хвойний (#2D4A3E) або темно-сірий (#3A3A3A) — для тексту, читабельно та солідно
    • --color-cta: М’який жовтий (#F5D76E) або кораловий (#F7786B) — для кнопок «Купити» та акцій, додає енергії без подразнення
    • --color-secondary: Пастельний зелений (#A8D8B9) — для маркування еко-товарів та фонових блоків
    13.3.2. Тема «Кристальна чистота» (класична)
    Акцент на ефективність засобів та ідеальну чистоту в домі. Блакитні відтінки підсвідомо сприймаються як «стерильні» та надійні. Синій + помаранчевий — класична маркетингова пара, яка змушує звернути увагу.
    • --color-primary: Небесно-блакитний (#87CEEB) — головний акцент, колір води та неба
    • --color-secondary: Яскраво-синій / Ultramarine (#3F51B5) — для логотипу та важливих елементів, символізує професіоналізм
    • --color-bg-main: Білий (#FFFFFF) з елементами світло-сірого (#F5F5F5) — для другорядних блоків, додає сучасності
    • --color-cta: Помаранчевий (#FF8C42) — для CTA-кнопок («Додати в кошик», «Замовити»), контрастна увага
    • --color-text-primary: Темно-синій (#1A237E) — для заголовків, глибокий графітовий (#424242) для основного тексту
    13.3.3. Тема «Домашній затишок» (найбільш «дружня»)
    Підходить для позиціонування товарів для догляду за домом, аромадифузорів та товарів для всієї родини. Ця гама найменше схожа на «хімію» і найбільше — на «турботу».
    • --color-primary: Ніжна лаванда / бузковий (#B39DDB) — основний колір, асоціюється з приємним ароматом та спокоєм
    • --color-bg-main: Бежевий / пісочний (#FFF8F0) — для фону, створює відчуття «дому»
    • --color-secondary: Зелений мох (#7CB342) — для маркування еко-товарів та натуральних засобів
    • --color-cta: Золотистий (#EDAB5C) або теплий рожевий (#E8919A) — для маркетингових плашок та кнопок дії
    • --color-text-primary: Темний шоколадний (#4E342E) — теплий, «домашній» колір тексту
    13.4. Принципи маркетингового дизайну
    Незалежно від обраної теми, дизайн сайту повинен дотримуватись наступних маркетингових принципів, оптимізованих для сегменту побутової хімії:
    13.4.1. Білий простір (Whitespace)
    У дизайні побутової хімії має бути багато «повітря» — вільного простору навколо елементів. Чим більше whitespace, тим чистішим здається товар і тим легше сприймається каталог. Конкретні вимоги:
    • Мінімальний відступ між картками товарів: 24px (мобільні) / 32px (десктоп)
    • Padding всередині карток: не менше 16px з кожного боку
    • Відстань між секціями на головній сторінці: не менше 48px
    • Зображення товару займає не більше 60% площі картки, решта — «повітря»
    • Заборонено: щільні сітки без розділювачів, перевантажені банери, текст впритул до країв
    13.4.2. Типографіка
    Рекомендовані шрифти — округлі, без засічок (sans-serif), які виглядають сучасно та «добріше» до покупця:
    • Для заголовків: Montserrat (Bold/SemiBold) — чистий, геометричний, впізнаваний
    • Для основного тексту: Poppins (Regular/Medium) — округлий, легко читається на будь-якому екрані
    • Альтернативи: Nunito, Inter, Open Sans — всі доступні через Google Fonts
    • Мінімальний розмір тексту: 16px для основного тексту на десктопі, 14px на мобільних
    • Міжрядковий інтервал: 1.5-1.6 для основного тексту, 1.2-1.3 для заголовків
    • Заборонено: декоративні шрифти, шрифти із засічками (serif) для основного контенту, розмір менше 12px
    13.4.3. Візуальна ієрархія
    Чітка візуальна ієрархія спрямовує погляд відвідувача до цільової дії (додати в кошик / замовити):
    • Головний акцент: ціна та кнопка «Додати в кошик» — найконтрастніший елемент на картці (колір --color-cta)
    • Другорядний акцент: бейджі «Акція», «Новинка», «Топ продажів» — помітні, але не відволікають від CTA
    • Фонові елементи: категорії, навігація, хлібні крихти — нейтральні кольори, не конкурують із товарами
    • Принцип F-паттерну: ключова інформація розташовується зліва та зверху (ціна, назва, фото)
    13.4.4. Фотографії товарів
    Вимоги до зображень для максимального маркетингового ефекту:
    • Чистий білий або прозорий фон — товар «літає», не відволікає від форми та кольору упаковки
    • Єдиний стиль освітлення для всіх товарів — каталог виглядає професійно і цілісно
    • Квадратний формат зображень (1:1 або 4:3) — стабільна сітка без «стрибків» layout
    • Можливість hover-zoom на десктопі: при наведенні показує збільшений фрагмент
    Кожна тема адаптивна та підтримує всі breakpoints, визначені в розділі адаптивного дизайну.
14. Адаптивний дизайн та кросплатформність
    Дизайн сайту повинен бути повністю адаптивним (responsive) та забезпечувати однаково зручний досвід використання на всіх типах пристроїв, розмірах екранів та операційних системах. Це є критичною вимогою до проєкту.
    14.1. Підтримувані розширення екранів (Breakpoints)
    • Мобільні пристрої (портрет): 320px – 480px
    • Мобільні пристрої (ландшафт): 481px – 767px
    • Планшети (портрет): 768px – 1023px
    • Планшети (ландшафт) / невеликі ноутбуки: 1024px – 1279px
    • Десктопи та ноутбуки: 1280px – 1919px
    • Великі монітори / Full HD+: 1920px і вище
    • Ultra-wide та 4K монітори: контент центрується з максимальною шириною контейнера 1440px, фон заповнює весь екран
    14.2. Підхід до верстки
    Застосовується стратегія Mobile-First з поступовим розширенням для більших екранів:
    • Mobile-First — базові стилі пишуться для мобільних пристроїв (320px); медіа-запити @media (min-width) розширюють для планшетів (768px) та десктопів (1200px+)
    • CSS Grid + Flexbox — Grid для макетів сторінок (каталог, dashboard); Flexbox для вирівнювання всередині компонентів (картка товару, хедер, кнопки)
    • Відносні одиниці — rem для типографіки та spacing, % та vw для ширини контейнерів, vh для hero-секцій; фіксовані px лише для border та мінімальних розмірів
    • Адаптивні зображення — елемент <picture> з srcset для завантаження оптимального розміру (thumbnail/medium/full) під конкретний пристрій та DPI; WebP з JPEG fallback
    • Container queries (підготовка) — структура CSS підготовлена для переходу на container queries для компонентної адаптації
    14.3. Адаптація компонентів інтерфейсу
    Кожен компонент UI адаптується під три viewport-зони: мобільний (<768px), планшет (768–1199px), десктоп (1200px+):
    • Навігаційне меню — десктоп: горизонтальне mega-menu з випадаючими категоріями; планшет/мобільний: гамбургер-іконка (☰) з висувною бічною панеллю (slide-in, z-index overlay)
    • Каталог товарів — десктоп: сітка 3–4 картки в ряд; планшет: 2–3 картки; мобільний: 1–2 картки або вертикальний список; перемикач вигляду (grid/list) зберігається в localStorage
    • Кошик та форма замовлення — мобільний: повна ширина, великі touch-зони (кнопки ≥ 44×44px, поля введення height ≥ 48px); степпер кількості збільшений
    • Таблиці (замовлення, списки) — десктоп: стандартна таблиця; мобільний: картковий вигляд (кожний рядок = окрема картка) або горизонтальний скрол з фіксованою першою колонкою
    • Модальні вікна — десктоп: центроване вікно з overlay; мобільний: повноекранний slide-up (bottom sheet) з кнопкою закриття у верхньому правому куті
    • Sidebar фільтрів — десктоп: ліва колонка, завжди видима; мобільний: прихована за кнопкою «Фільтри», slide-in з overlay, кнопка «Застосувати»
    14.4. Сумісність з операційними системами та браузерами
    Сайт повинен коректно працювати на наступних платформах та браузерах:
    • Windows 10/11: Chrome (остання версія), Firefox, Edge, Opera
    • macOS (Monterey+): Safari, Chrome, Firefox
    • iOS (16+): Safari, Chrome — з урахуванням WebKit-обмежень
    • Android (11+): Chrome, Samsung Internet, Firefox
    • Linux: Chrome, Firefox
    Кросбраузерне тестування є обов’язковим перед кожним релізом. Сайт повинен зберігати повну функціональність в усіх зазначених середовищах.
    14.5. Touch-оптимізація
    Для мобільних та планшетних пристроїв з сенсорними екранами:
    • Мінімальний розмір інтерактивних елементів (кнопки, посилання, чекбокси) — 44×44 px згідно з рекомендаціями WCAG; для primary CTA («Додати до кошика», «Оформити замовлення») — не менше 48×48 px
    • Відстань між сусідніми інтерактивними елементами — не менше 8px для запобігання випадковим натисканням (tap target spacing)
    • Swipe-жести — горизонтальний свайп для каруселей товарів, галерей зображень та банерів; вертикальний свайп для pull-to-refresh у каталозі
    • Відсутність hover-залежних функцій — вся функціональність доступна через tap/click; tooltip-и замінюються на bottom sheet або inline-текст
    • Haptic feedback (підготовка) — структура для вібро-відгуку при додаванні до кошика на пристроях з підтримкою Vibration API
    14.6. Оптимізація завантаження
    Для забезпечення швидкого завантаження на мобільних мережах:
    • Lazy-loading зображень — loading="lazy" для всіх зображень нижче першого viewport; placeholder: blur-up з 20×20px мініатюри (розділ 4.2.1)
    • Формати зображень — WebP з JPEG fallback через <picture>; стиснення: quality 80% для full, 70% для medium, 60% для thumbnail
    • Code splitting — JS-бандли розділені по роутах; lazy-import для модальних вікон, кабінету, адмінки; критичний CSS inline в <head>
    • Мініфікація та стиснення — CSS/JS мініфікація (Terser, cssnano); Brotli/Gzip компресія на сервері; HTTP/2 для паралельного завантаження
    • SSR / SSG — серверний рендеринг для каталогу та сторінок товарів (SEO + швидкий перший рендер); статична генерація для FAQ, юридичних сторінок
    • Core Web Vitals цілі — LCP (Largest Contentful Paint): ≤ 2.5с; CLS (Cumulative Layout Shift): ≤ 0.1; INP (Interaction to Next Paint): ≤ 200ms
    14.7. Доступність (Accessibility, WCAG 2.1 AA)
    Сайт повинен відповідати базовим вимогам WCAG 2.1 рівня AA для забезпечення доступності людям з обмеженими можливостями:
    14.7.1. Зорові порушення
    • Контраст тексту: мінімум 4.5:1 для основного тексту, 3:1 для великого тексту (>18px) та елементів інтерфейсу
    • Масштабування тексту до 200% без втрати функціональності та горизонтальної прокрутки
    • Alt-атрибути для всіх зображень товарів — автогенерація за шаблоном або вручну через seo_description
    • Не передавати інформацію тільки через колір — бейджі мають текстові мітки, помилки форм мають текстові повідомлення
    • Підтримка режиму високого контрасту (prefers-contrast: more) через CSS media query
    14.7.2. Навігація клавіатурою
    • Повна навігація Tab / Shift+Tab по всіх інтерактивних елементах у логічному порядку
    • Видимий focus-індикатор для всіх інтерактивних елементів (кнопки, посилання, поля форм)
    • Enter — активація кнопок та посилань; Escape — закриття модалок, dropdown, autocomplete
    • Skip-to-content посилання для швидкого переходу до основного контенту
    • Модальні вікна: фокус-trap (фокус не виходить за межі модалки), повернення фокусу після закриття
    14.7.3. Семантика та скрінрідери
    • Семантична HTML-розмітка: <header>, <nav>, <main>, <aside>, <footer>, <article>, <section>
    • ARIA-атрибути: aria-label для іконок-кнопок, aria-expanded для dropdown, aria-live для динамічного контенту (кошик, сповіщення)
    • Структуровані заголовки h1–h6 без пропусків рівнів
    • Форми: <label> пов’язаний з <input> через for/id; групи полів у <fieldset> з <legend>
    • role='alert' для повідомлень про помилки; role='status' для підтверджень (товар додано)

    • Альтернатива: інтеграція з Telegram-ботом як основним каналом чату з менеджером

    • Політика повернення та обміну — відповідно до Закону України «Про захист прав споживачів»
    Усі документи редагуються менеджером через WYSIWYG-редактор у кабінеті. При реєстрації користувач повинен підтвердити ознайомлення з Політикою конфіденційності та Умовами використання (чекбокс).
    16.2. Cookie-банер та згода на обробку даних
    При першому відвідуванні сайту відображається Cookie-банер з наступним функціоналом:
    • Інформування про використання cookies з посиланням на Політику конфіденційності
    • Кнопки: «Прийняти всі», «Лише необхідні», «Налаштувати»

    • Повторна зміна налаштувань доступна через посилання в футері «Налаштування cookies»

    • Збір лише необхідних даних: email, ПІБ, телефон, адреса доставки — мінімально необхідний набір
    • Право на видалення — користувач може запросити видалення свого акаунту та всіх пов’язаних даних через кабінет
    • Право на експорт — користувач може завантажити свої персональні дані у форматі JSON/CSV
    • Шифрування чутливих даних у БД та при передачі (TLS)
    • Логування доступу до персональних даних (хто, коли, до яких даних звертався)
17. Обробка помилок та граничних випадків (Edge Cases)
    17.1. Помилки при оформленні замовлення
    • Товар закінчився під час оформлення — при переході до підтвердження система перевіряє актуальну наявність; якщо товар закінчився, відображається повідомлення з пропозицією оновити кошик або видалити товар
    • Зміна ціни під час оформлення — якщо ціна змінилась (наприклад, через імпорт Excel), клієнт отримує повідомлення з новою ціною та підтверджує замовлення повторно
    • Дублювання замовлення — захист від подвійного натискання кнопки «Підтвердити» (блокування кнопки + перевірка на сервері за idempotency key)
    • Таймаут сесії під час оформлення — кошик зберігається в localStorage та відновлюється після повторного входу
    17.2. Помилки імпорту Excel
    • Невалідна структура файлу — система перевіряє наявність обов’язкових стовпців та відображає детальне повідомлення про помилку
    • Дублікати кодів товарів — попередження з переліком дублікатів, менеджер обирає: оновити або пропустити
    • Некоректні дані у клітинках — від’ємна ціна, текст замість числа, порожні обов’язкові поля — звіт помилок з номерами рядків
    • Великий файл (>5000 рядків) — імпорт виконується у фоновому режимі через чергу задач із прогрес-баром та сповіщенням по завершенню
    • Переривання імпорту — транзакційність: або всі зміни застосовуються, або жодні (rollback при критичній помилці)
    17.3. Помилки месенджер-ботів та Instagram API
    • Недоступність Telegram/Viber API — повідомлення додаються в чергу повторних спроб (retry queue) з експоненціальною затримкою; менеджер отримує email про збій
    • Таймаут бота — відповідь користувачу: «Вибачте, сталася тимчасова помилка. Спробуйте пізніше або відвідайте наш сайт»
    • Невалідний callback від бота — логування помилки, graceful degradation (повернення до головного меню)
    • Instagram Graph API помилки — rate limit (25 posts/day): публікація відкладається з повідомленням менеджеру «Ліміт вичерпано, публікація буде опублікована завтра»; expired token: алерт менеджеру + інструкція переавторизації; невалідне зображення: повідомлення з деталями (розмір, формат) та кнопка «Замінити зображення»
    17.4. Загальна стратегія обробки помилок
    • Централізований обробник помилок на backend (Error Middleware) — єдиний формат відповіді з кодом помилки, повідомленням та деталями
    • Розмежування помилок: клієнтські (400, 401, 403, 404, 422) та серверні (500, 502, 503) — серверні логуються для розслідування
    • User-friendly повідомлення про помилки на frontend — без технічних деталей, з конкретними рекомендаціями дій
    • Кастомні сторінки помилок: 404 (з пошуком та популярними категоріями), 500 (з формою зв’язку), 503 (технічні роботи з орієнтовним часом)
    • Global error boundary на frontend (React/Vue) — запобігає «білому екрану» при неочікуваних помилках
    • Sentry або аналог для автоматичного збору та агрегації помилок на production
    17.5. Валідація форм
    Усі форми на сайті проходять подвійну валідацію: клієнтську (JavaScript, миттєвий зворотний зв’язок) та серверну (захист від обходу). Формати валідації:
    • Email — формат RFC 5322, перевірка на існування домену (DNS MX-запис, опціонально)
    • Телефон — формат +380XXXXXXXXX (9 цифр після коду), автоформатування при введенні, маска вводу
    • ЄДРПОУ — рівно 8 цифр, контрольна сума (алгоритм перевірки ЄДРПОУ)
    • IBAN — формат UA + 27 цифр (29 символів), контрольна сума за стандартом ISO 13616
    • Пароль — мінімум 8 символів, хоча б одна літера та одна цифра; індикатор надійності (слабкий/середній/сильний)
    • ПІБ — мінімум 2 символи, тільки літери (укр/лат), пробіли, апострофи та дефіси
    • Кількість товару — ціле число > 0, не більше наявного на складі, кратність упаковки (для гурту)
    • URL slug — тільки латиниця, цифри та дефіси; автоматична транслітерація
    Поведінка валідації: помилки відображаються під полем (червоний текст), поле обводиться червоною рамкою; валідне поле — зелена рамка з галочкою. Валідація спрацьовує при втраті фокусу (onBlur) та при спробі відправити форму. Кнопка «Відправити» заблокована якщо є помилки.
    17.6. Кастомні сторінки помилок
    Кожна сторінка помилки має бути оформлена в стилі активної теми з корисними елементами для користувача:
    • 404 (Сторінку не знайдено) — ілюстрація або іконка, текст «На жаль, цієї сторінки не існує», пошуковий рядок для пошуку товару, блок «Популярні категорії» (6-8 іконок), кнопка «На головну»
    • 500 (Помилка сервера) — текст «Щось пішло не так, ми вже працюємо над цим», контакти підтримки (телефон, email, Telegram), кнопка «Спробувати ще раз» (window.location.reload)
    • 503 (Технічні роботи) — сторінка Maintenance Mode, орієнтовний час завершення робіт (встановлюється менеджером), форма «Сповістити мене коли сайт запрацює» (email)
    • Офлайн-сторінка (PWA) — кешована Service Worker’ом, текст «Відсутнє з’єднання з інтернетом», інструкція перевірити підключення, кнопка «Повторити»
    • Всі сторінки помилок — адаптивні, з хедером та футером сайту, мультимовні (UA)
    17.7. Стани завантаження та порожні стани (Loading & Empty States)
    Кожна асинхронна операція повинна мати чітке візуальне відображення стану:
    17.7.1. Skeleton screens (скелетони)
    • Каталог товарів — сітка карток-заглушок з анімованими сірими блоками (замість фото, назви, ціни); відображається під час завантаження списку
    • Сторінка товару — skeleton для галереї (великий блок), назви, ціни, опису
    • Кабінет менеджера — skeleton для таблиць (рядки-заглушки), графіків (прямокутники), лічильників
    • Стрічки (замовлення, сповіщення) — skeleton для кожного елемента списку
    • Анімація: pulse (opacity 0.3 → 0.7 → 0.3) або shimmer (градієнт зліва направо)
    17.7.2. Спіннери та індикатори прогресу
    • Кнопки (додати в кошик, оформити замовлення) — при натисканні: текст замінюється на спіннер, кнопка блокується (disabled); після завершення: ✓ + повернення тексту через 1.5с
    • Імпорт Excel — progress bar з відсотком: «Обробка... 45% (2250 з 5000 рядків)»
    • Пошук — спіннер у полі пошуку під час завантаження результатів autocomplete
    • Inline-loading — для підвантаження «Завантажити ще» у стрічках
    17.7.3. Порожні стани (Empty States)
    • Кошик порожній — ілюстрація (порожній кошик), текст «Ваш кошик порожній», кнопка «Перейти до каталогу»
    • Немає замовлень — ілюстрація, текст «У вас ще немає замовлень», кнопка «Обрати товари»
    • Обране порожнє — ілюстрація (порожнє серце), текст «Ви ще нічого не зберегли»
    • Пошук без результатів — ілюстрація, текст «На жаль, нічого не знайдено», поради («перевірте написання», «спробуйте інші ключові слова»), кнопка «Скинути фільтри»
    • Сповіщення відсутні — ілюстрація (дзвіночок), текст «Нових сповіщень немає»
    17.7.4. Toast-сповіщення (Toasts)
    • Позиція: bottom-right (desktop), bottom-center (mobile); автозникнення через 3–5 секунд
    • Типи: success (зелений — «Товар додано до кошика», «Замовлення оформлено»), info (синій — «Посилання скопійовано»), warning (жовтий — «Товар закінчується на складі»), error (червоний — «Помилка, спробуйте ще раз»)
    • Кнопка закриття (×); для action-toasts: кнопка дії («Переглянути кошик», «Скасувати»)
    • Максимум 3 toast одночасно; нові зсувають старі вгору
    17.8. Модальні вікна та діалоги
    Специфікація модальних вікон, що використовуються в системі:
    • Quick View товару — відкривається з каталогу без переходу на сторінку; містить: фото, назва, ціна, кнопка «Додати до кошика», посилання «Детальніше»; закриття: Escape, клік на overlay, кнопка ×
    • Підтвердження видалення — для деструктивних дій (видалити товар з кошика, скасувати замовлення, видалити адресу): текст «Ви впевнені?», дві кнопки: «Скасувати» (secondary) та «Видалити» (danger, червона)
    • Lightbox для зображень — повноекранний перегляд фото товару; стрілки для навігації між фото; закриття: Escape, клік на overlay; swipe на мобільних
    • Cookie Consent — банер внизу сторінки (не модалка, але overlay-елемент); кнопки «Прийняти» та «Налаштувати» (розділ 16.2)
    • Підтвердження виходу — «Ви дійсно хочете вийти з акаунту?»; кнопки «Скасувати» / «Вийти»
    • Загальні вимоги: фокус-trap (Tab не виходить за межі модалки); aria-modal=true; блокування скролу body; анімація появи (fade + scale); повернення фокусу на елемент, що відкрив модалку
    17.9. Друк (Print Styles)
    CSS @media print стилі для коректного друку сторінок:
    • Друк замовлення (зі сторінки деталей замовлення в кабінеті) — чистий layout без навігації, хедера, футера; лише: логотип, номер замовлення, дані клієнта, таблиця товарів, сума, статус
    • Друк рахунку — формат для бухгалтерії: реквізити компанії, дані покупця, таблиця товарів з ПДВ (якщо є), підпис, печатка (placeholder)
    • Друк прайс-листа — таблиця товарів з цінами, згрупована по категоріях; кнопка «Друк» у кабінеті менеджера
    • Загальне: приховати навігацію, sidebar, кнопки, footer; чорно-білий режим; розриви сторінок (page-break) між логічними блоками; URL посилань у дужках після тексту
18. Progressive Web App (PWA)
    18.1. Базові можливості PWA
    Сайт реалізується як Progressive Web App, що дозволяє користувачам «встановити» його на пристрій як повноцінний додаток:
    • Web App Manifest (manifest.json) — назва, іконки (192x192, 512x512), тема, орієнтація, display: standalone
    • Кнопка «Встановити додаток» — з’являється при відвідуванні сайту на мобільному пристрої (beforeinstallprompt event)
    • Іконка на домашньому екрані — запуск без адресного рядка браузера, повноекранний режим
    • Splash Screen — екран завантаження при запуску з іконки
    18.2. Service Worker та кешування
    • Кешування статичних ресурсів (CSS, JS, шрифти, іконки) — стратегія Cache First
    • Кешування зображень товарів — стратегія Stale While Revalidate (показує з кешу, оновлює у фоні)
    • Кешування API-відповідей каталогу — стратегія Network First з фолбеком на кеш
    • Офлайн-заглушка — при відсутності інтернету відображається сторінка з повідомленням «Немає з’єднання» та кнопкою «Повторити», а також останній переглянутий каталог з кешу
    18.3. Push-сповіщення (підготовка)
    Закладається інфраструктура для web push-сповіщень, які будуть активовані в наступних фазах:
    • Запит дозволу на сповіщення (ненав’язливий, після другого відвідування)
    • Підготовлені типи сповіщень: нова акція, зміна статусу замовлення, повернення в наявність товару зі списку бажань
    • Push API + FCM (Firebase Cloud Messaging) або Web Push Protocol
19. Тестування та CI/CD
    19.1. Стратегія тестування
    Багаторівнева стратегія тестування забезпечує стабільність та надійність платформи:
    19.1.1. Unit-тести
    • Покриття бізнес-логіки: розрахунок цін (роздріб/опт), валідація замовлень, перевірка мінімальних сум, бізнес-правила гуртівників
    • Тестування сервісів: імпорт Excel, генерація PDF, відправка сповіщень
    • Мінімальне покриття: 70% для критичних модулів (Auth, Orders, Import)
    • Інструменти: Jest (Node.js) або pytest (Python)
    19.1.2. Інтеграційні тести
    • Тестування API-ендпоінтів: авторизація, CRUD товарів, оформлення замовлень, зміна статусів
    • Тестування взаємодії з БД: міграції, транзакційність імпорту, цілісність даних
    • Тестування інтеграцій: Telegram Bot API, Viber API, SMTP
    • Інструменти: Supertest (Node.js) або httpx (Python) + тестова БД
    19.1.3. E2E-тести
    • Критичні сценарії: реєстрація → вхід → каталог → кошик → замовлення
    • Сценарії менеджера: імпорт Excel → перевірка каталогу → обробка замовлення
    • Кросбраузерне тестування: Chrome, Firefox, Safari (мінімум)
    • Інструменти: Playwright або Cypress
    19.1.4. Навантажувальне тестування
    • Симуляція 200 одночасних користувачів
    • Тестування імпорту великих Excel-файлів (10000+ рядків)
    • Стрес-тестування API (rate limiting, черги)
    • Інструменти: k6 або Artillery
    19.2. Середовища
    • Development (dev) — локальне середовище розробника; Docker Compose з усіма сервісами (PostgreSQL, Redis, Node.js, Python worker); тестова БД із seed-даними (50 товарів, 10 замовлень, 3 користувачі); hot-reload для frontend та backend; URL: http://localhost:3000
    • Staging (stage) — точна копія production за конфігурацією; автоматичний деплой з develop-гілки при merge; тестування перед релізом; доступ лише для команди (Basic Auth або VPN); URL: https://stage.example.com; окрема БД з анонімізованими production-даними
    • Production (prod) — бойовий сервер; деплой лише з main-гілки після проходження всіх тестів та code review; автоматичне резервне копіювання (розділ 9.5); zero-downtime deployment (rolling update); URL: https://example.com
    • Змінні середовища — кожне середовище має власний .env файл (розділ 8.8); чутливі дані (API keys, DB credentials) зберігаються в secrets manager (Docker secrets або Vault); dev-середовище використовує тестові токени для Telegram/Viber/Instagram
    • Дані між середовищами — production → staging: щотижневий sync з анонімізацією персональних даних; staging → dev: seed-скрипт генерує мінімальний тестовий набір; міграції БД застосовуються послідовно на всіх середовищах
    19.3. CI/CD Pipeline
    Автоматизований процес збірки, тестування та деплою (GitHub Actions або GitLab CI): 1. Push у гілку → запуск лінтерів (ESLint/Prettier або flake8/black) та перевірка типів 2. Запуск unit-тестів та інтеграційних тестів у Docker-контейнері 3. Збірка Docker-образів, якщо тести пройшли 4. Merge в develop → автоматичний деплой на Staging 5. Merge в main → деплой на Production після ручного підтвердження (manual approval) 6. Автоматичні E2E-тести на Staging перед production-деплоєм 7. Rollback — можливість швидкого відкату до попередньої версії за 1 команду
    19.4. Стандарти коду та правила CI-перевірок
    Правила якості коду, що автоматично перевіряються на CI/CD (стандарти коментування та документації описані в розділах 8.5–8.6):
    • Єдиний code style через лінтери та форматери (автоматична перевірка на CI)
    • Обов’язковий Code Review перед merge у develop/main — мінімум 1 апрувер
    • Conventional Commits (feat:, fix:, docs:, refactor:, test:, chore:) для автоматичної генерації changelog
    • Git Flow: main (production), develop (staging), feature/_ (нові функції), hotfix/_ (термінові виправлення), release/\* (підготовка релізу)
    • Документація API через Swagger/OpenAPI (автогенерація з анотацій у коді)
    • Максимальна довжина функції — 50 рядків; якщо більше — декомпозиція на підфункції
    • Максимальна кількість параметрів функції — 4; якщо більше — об’єднати в об’єкт (DTO / options)
    • Заборонено: magic numbers (замість 30 — CART*EXPIRY_DAYS = 30), вкладеність > 3 рівнів, функції з побічними ефектами без явного позначення
    • Типізація: TypeScript — strict mode, no any (або мінімум); Python — type hints для всіх публічних функцій
    • Імпорти: абсолютні шляхи через аліаси (@modules/, @common/, @config/), заборонено ../../.. глибше 2 рівнів
    • Принцип DRY: дублювання коду > 3 рядків у 2+ місцях — виносити в спільну функцію / хелпер
    • Error handling: всі async-функції обгортаються в try-catch; кастомні класи помилок (OrderNotFoundError, ImportValidationError) замість generic Error
    19.5. Повна матриця тестів по модулях
    Кожен модуль системи повинен мати повний набір тестів, що покривають усі бізнес-сценарії, граничні випадки та помилки:
    19.5.1. Модуль Auth (авторизація)
    • Unit: хешування паролів, генерація/валідація JWT, перевірка refresh-токенів, валідація email-формату
    • Integration: реєстрація (створення юзера в БД, відправка email), логін (видача токенів), Google OAuth flow
    • E2E: реєстрація → підтвердження email → логін → доступ до кабінету; відновлення пароля
    • Security: rate limiting (5 невдалих спроб → блок), SQL injection у полях логіну, XSS у полі email
    • Edge cases: реєстрація з вже існуючим email, логін з неправильним паролем, просрочений refresh-токен
    19.5.2. Модуль Catalog (каталог товарів)
    • Unit: фільтрація товарів, сортування, розрахунок знижки, генерація slug, пошук (fuzzy match)
    • Integration: CRUD товарів через API, пагінація, фільтри, пошук з autocomplete
    • E2E: перегляд каталогу → фільтрація за категорією → сортування за ціною → перехід до товару
    • Edge cases: товар без зображення (placeholder), порожня категорія, пошук без результатів, товар з ціною 0
    19.5.3. Модуль Cart (кошик)
    • Unit: додавання товару, зміна кількості, видалення, розрахунок суми, валідація кратності (опт), перевірка мінімуму (опт)
    • Integration: збереження кошика в БД, відновлення кошика при логіні, мерж localStorage + серверний кошик
    • E2E: додати товар → змінити кількість → видалити → оформити замовлення; сценарій гуртівника з мінімальною сумою
    • Edge cases: товар став недоступним після додавання, ціна змінилась, кошик перевищує залишок на складі
    19.5.4. Модуль Orders (замовлення)
    • Unit: створення замовлення, зміна статусу (валідація переходів), розрахунок суми з доставкою, генерація номера замовлення
    • Integration: оформлення замовлення через API, зміна статусу менеджером, відправка сповіщень при зміні статусу
    • E2E: повний цикл: каталог → кошик → checkout → оплата → підтвердження → статус «Виконано»
    • Edge cases: замовлення з 0 товарів, замовлення з недоступним товаром, паралельне замовлення одного товару (конкуренція за залишок)
    19.5.5. Модуль Import (імпорт Excel)
    • Unit: парсинг Excel, валідація полів (ціна > 0, SKU унікальний), маппінг колонок, конвертація типів
    • Integration: імпорт файлу через API, створення/оновлення товарів у БД, генерація звіту, обробка зображень
    • E2E: менеджер завантажує Excel → прев’ю даних → підтвердження → звіт «додано X, оновлено Y, помилок Z»
    • Edge cases: порожній файл, файл з 10000+ рядків, невалідний формат, дублікати SKU, від’ємна ціна, пропущені обов’язкові поля
    19.5.6. Модуль Notifications (сповіщення)
    • Unit: рендеринг email-шаблонів, підстановка змінних, формування Telegram-повідомлень
    • Integration: відправка email через SMTP (mock), відправка Telegram (mock Bot API), черга сповіщень
    • E2E: створення замовлення → клієнт отримує email → менеджер отримує Telegram
    • Edge cases: невалідний email, Telegram бот заблокований користувачем, SMTP-сервер недоступний (retry)
    19.5.7. Модуль Payments (оплата)
    • Unit: розрахунок суми до оплати, валідація способу оплати, формування рахунку
    • Integration: створення рахунку через API, зміна статусу оплати, генерація PDF-рахунку
    • E2E: вибір «Оплата на рахунок» → генерація рахунку → підтвердження менеджером → статус «Оплачено»
    • Edge cases: подвійна оплата, оплата після скасування замовлення, часткова оплата
    19.5.8. Модуль Delivery (доставка)
    • Unit: розрахунок вартості доставки, валідація адреси, парсинг відповіді Нова Пошта API
    • Integration: пошук міст/відділень НП, створення ТТН (підготовка), трекінг статусу
    • E2E: вибір «Нова Пошта» → пошук міста → вибір відділення → розрахунок вартості
    • Edge cases: API НП недоступне (fallback), місто без відділень, доставка за кордон (заборона)
    19.5.9. Модуль Users (користувачі)
    • Unit: валідація ЄДРПОУ, IBAN, телефону, генерація гуртового запиту, зміна ролі
    • Integration: CRUD профілю, список адрес доставки, запит на гуртовий статус, зміна пароля
    • E2E: редагування профілю → збереження → перевірка оновлених даних; запит гуртового статусу → підтвердження менеджером
    • Edge cases: зміна email на вже зайнятий, видалення останньої адреси, ЄДРПОУ з невалідною контрольною сумою
    19.5.10. Модуль SEO
    • Unit: генерація title/description з шаблонів, транслітерація slug, валідація довжини (title ≤ 60, description ≤ 160)
    • Integration: автогенерація SEO при створенні товару, sitemap.xml генерація, robots.txt
    • E2E: створення товару → перевірка meta-тегів у HTML → перевірка Schema.org розмітки
    • Edge cases: товар без назви, дублікати slug (авто-суфікс), шаблон з невідомою змінною
    19.5.11. Модуль Analytics (аналітика)
    • Unit: агрегація подій воронки, розрахунок conversion rate, формування ABC-аналізу
    • Integration: запис client events через API, щоночна агрегація (cron), генерація звітів
    • E2E: менеджер відкриває дашборд → обирає період → графіки завантажуються → експорт у Excel
    • Edge cases: період без даних (порожні графіки), великий обсяг подій (пагінація), фільтр без результатів
    19.5.12. Модуль Messengers та Instagram (Telegram/Viber боти, Instagram API)
    • Unit: парсинг webhook-повідомлень (Telegram, Viber), формування відповідей, команди бота, Instagram caption генерація, хештег-валідація
    • Integration: обробка /start, /catalog, /order*{id} команд, автовідповіді, публікація з кабінету в Telegram + Viber + Instagram одночасно
    • Integration Instagram: OAuth підключення (mock), створення media container, publish, отримання insights
    • E2E: менеджер створює публікацію в кабінеті → пост з’являється в Telegram-каналі + Viber-спільноті + Instagram
    • E2E Instagram: менеджер обирає товар → шаблон «Товар дня» → crop фото → preview → публікація → метрики через 24 години
    • Edge cases: некоректний webhook, бот заблокований, відправка зображення > 10MB, Instagram token expired, Instagram rate limit (25 posts/day), карусель з > 10 фото
    19.5.13. Покриття та звітність тестів
    • Мінімальне покриття: 70% загальне, 85% для критичних модулів (Auth, Orders, Cart, Payments)
    • Інструменти покриття: Istanbul/NYC (JS/TS) або coverage.py (Python)
    • Звіт покриття генерується на кожен PR/MR, блокує merge при падінні нижче порогу
    • Mutation testing (підготовка): Stryker (JS) або mutmut (Python) для перевірки якості тестів
    • Тестові дані: окремий seed-файл з реалістичними даними для всіх модулів (50+ товарів, 10+ користувачів, 20+ замовлень)
    • Fixture: фабрики тестових даних (createTestProduct(), createTestOrder()) для швидкого створення моків
20. Аналітика та звітність
    Повноцінний модуль аналітики в кабінеті менеджера, що забезпечує прийняття рішень на основі даних. Усі звіти доступні для перегляду онлайн та експорту в Excel/PDF.
    20.1. Аналітика продажів
    20.1.1. Звіт «Продажі за період»
    • Графік продажів (лінійний / стовпчастий) з вибором періоду: сьогодні, тиждень, місяць, квартал, рік, довільний діапазон
    • Показники: кількість замовлень, загальна сума, середній чек, кількість проданих одиниць
    • Порівняння з попереднім аналогічним періодом (місяць до місяця, тиждень до тижня) з відсотком зростання/падіння
    • Деталізація: по днях, тижнях або місяцях залежно від обраного періоду
    • Фільтри: за типом клієнта (роздріб/опт), за категорією товарів, за конкретним товаром
    20.1.2. Звіт «Товари»
    • Топ-N товарів за кількістю продажів та за сумою виручки (N налаштовується: 10, 20, 50)
    • Аутсайдери — товари з нульовими або мінімальними продажами за період (кандидати на видалення або акцію)
    • Динаміка продажів конкретного товару — графік продажів за обраний період з позначками змін ціни та акцій
    • ABC-аналіз товарів: категорія A (80% виручки), B (15%), C (5%) — для оптимізації асортименту
    • Маржинальність (підготовка) — якщо додати собівартість у Excel, система розрахує маржу по кожному товару
    20.1.3. Звіт «Категорії»
    • Продажі по категоріях — кругова діаграма або стовпчаста: частка кожної категорії у загальній виручці
    • Динаміка категорій — як змінювалися продажі кожної категорії з часом
    • Середній чек по категоріях
    • Порівняння категорій між періодами
    20.2. Аналітика клієнтів
    20.2.1. Звіт «Клієнти»
    • Топ-N клієнтів за загальною сумою замовлень та за кількістю замовлень
    • Нові клієнти за період — графік реєстрацій по днях/тижнях
    • Конверсія: зареєстровані → зробили перше замовлення → зробили повторне замовлення (воронка)
    • Сегментація клієнтів: активні (замовлення за останні 30 днів), сплячі (60-90 днів без замовлень), втрачені (90+ днів)
    • LTV (Lifetime Value) — середня загальна сума замовлень клієнта за весь час
    • Розподіл клієнтів за типом (роздріб/опт) та динаміка переходів з роздрібу в опт
    20.2.2. Звіт «Гуртівники»
    • Окремий звіт по гуртових клієнтах: обсяги, частота, середнє замовлення
    • Порівняльна таблиця гуртівників: хто скільки замовляє, динаміка зростання/падіння
    • Регулярність замовлень — інтервали між замовленнями кожного гуртівника (допомагає прогнозувати)
    • Асортимент гуртівника — які категорії та товари замовляє конкретний клієнт
    20.3. Аналітика замовлень
    • Статуси замовлень — кругова діаграма: скільки замовлень в якому статусі зараз
    • Воронка замовлень — скільки замовлень проходить кожен етап: Нове → В обробці → Підтверджене → Відправлене → Виконане; де найбільше «відвалюється»
    • Середній час обробки — скільки часу від створення замовлення до зміни статусу на «Відправлене» (KPI менеджера)
    • Скасовані замовлення — кількість, причини (якщо зазначені), частка від загальної кількості
    • Географія замовлень — розподіл за регіонами/містами (на основі адреси доставки), теплова карта
    • Розподіл за часом — в які дні тижня та години найбільше замовлень (для планування роботи)
    20.4. Аналітика каналів (SMM)
    • Переходи по каналах — графік: скільки користувачів прийшли з Telegram, Viber, Instagram, прямий трафік, пошукові системи, QR-коди
    • Конверсія по каналах — який відсоток відвідувачів з кожного каналу зробив замовлення
    • Ефективність публікацій — кількість переходів з кожної публікації в Telegram/Viber/Instagram (на основі UTM-міток); для Instagram додатково: likes, saves, reach
    • ROI каналів (підготовка) — структура для розрахунку повернення інвестицій при додаванні витрат на рекламу
    • Підписники — динаміка кількості підписників Telegram-каналу, Viber-спільноти та Instagram-профілю (дані з таблиці Channel_Stats, розділ 7.6.1)
    20.5. Аналітика складу
    • Залишки по категоріях — кругова діаграма розподілу залишків у грошовому еквіваленті
    • Критичні залишки — список товарів з кількістю нижче встановленого порогу, з прогнозом дати закінчення на основі середніх продажів
    • Оборотність товарів — скільки разів товар «прокрутився» за період (продажі / середній залишок)
    • Мертвий stock — товари, які не продавалися більше 60 днів при ненульовому залишку
    • Історія імпортів — журнал завантажень Excel: дата, хто завантажив, кількість змін, помилки
    20.6. Аналітика цін
    • Історія змін цін — таблиця з фільтрацією: який товар, коли, стара ціна → нова ціна, відсоток зміни
    • Вплив знижок — як змінилися продажі товару після зниження ціни або запуску акції
    • Порівняння роздрібних та гуртових цін — середня маржа між роздрібом та гуртом по категоріях
    • Товари зі зниженням ціни — перелік товарів, де ціна знизилась при останньому імпорті
    20.7. Зведені звіти та автоматизація
    • Щоденний дайджест на email менеджера (о 09:00): нові замовлення за вчора, сума, нові реєстрації, критичні залишки
    • Щотижневий звіт (понеділок): продажі за тиждень, порівняння з попереднім, топ-товари, нові клієнти
    • Щомісячний звіт: повна зведена аналітика за місяць у PDF з графіками та таблицями
    • Кастомні звіти — менеджер обирає метрики, період, фільтри та зберігає як шаблон для повторного використання
    • Планові сповіщення — алерти при досягненні порогів: залишок < N, продажі за день < суми, новий великий клієнт
    • Усі звіти доступні для експорту: Excel (.xlsx), PDF (з графіками), CSV (для зовнішніх систем)
    20.8. Трекінг дій клієнта (аналітика воронки)
    Система фіксує ключові дії клієнтів на сайті для побудови воронки конверсії та поведінкової аналітики. Дані зберігаються в таблиці Client_Events:
    20.8.1. Відстежувані події
    • page_view — перегляд сторінки (URL, referrer, тип пристрою)
    • product_view — перегляд детальної сторінки товару (product_id, час на сторінці)
    • add_to_cart — додавання товару до кошика (product_id, кількість, джерело: картка/деталі/рекомендації)
    • remove_from_cart — видалення товару з кошика (product_id)
    • cart_view — перегляд сторінки кошика
    • checkout_start — початок оформлення замовлення (сума кошика, кількість товарів)
    • checkout_delivery — вибір способу доставки
    • checkout_payment — вибір способу оплати
    • order_complete — замовлення оформлено (order_id, сума)
    • search — пошуковий запит (query, кількість результатів)
    • add_to_wishlist — додавання до обраного (product_id)
    • registration — реєстрація нового акаунту (метод: email/google)
    • login — авторизація (метод: email/google)
    20.8.2. Воронка конверсії
    На основі подій автоматично будується воронка в кабінеті менеджера:
    • Етапи: перегляд каталогу → перегляд товару → додавання в кошик → перегляд кошика → початок checkout → завершення замовлення
    • Відсоток переходу між кожним етапом (conversion rate)
    • Фільтри: за періодом, типом клієнта (роздріб/опт), пристроєм (десктоп/мобільний), джерелом трафіку (UTM)
    • Графік: горизонтальна воронка з кількістю та відсотком на кожному етапі
    • Порівняння воронки за два періоди (цей тиждень vs минулий)
    20.8.3. Технічна реалізація
    • Збір подій: JavaScript-трекер на frontend відправляє події через API endpoint POST /api/events (асинхронно, без впливу на швидкість сторінки)
    • Ідентифікація: авторизований — user_id; гість — анонімний session_id (cookie); при авторизації гостьові події прив’язуються до акаунту
    • Зберігання: таблиця Client_Events (event_type, user_id, session_id, product_id, metadata JSON, created_at); партиціонування за місяцями
    • Агрегація: щоночі cron-задача агрегує дані в зведені таблиці для швидких звітів (Daily_Funnel_Stats)
    • Ретенція: сирі події зберігаються 90 днів, агреговані — безстроково
    • Інтеграція з Google Analytics 4 (підготовка): dataLayer events для майбутнього підключення GA4
21. Критерії приймання модулів (Acceptance Criteria)
    Для кожного модуля системи визначено чіткий чеклист критеріїв, за якими визначається готовність модуля до релізу. Модуль вважається прийнятим, коли всі обов’язкові критерії виконані.
    21.1. Модуль Каталог
    • ☐ Головна сторінка відображає акційні товари та категорії
    • ☐ Картка товару показує фото, назву, ціну, бейджі, наявність
    • ☐ Детальна сторінка товару відображає повний опис (з Product_Content) та рекомендовані товари
    • ☐ Пошук за назвою повертає релевантні результати за < 500мс
    • ☐ Фільтрація за категорією, сортування за ціною/назвою працюють коректно
    • ☐ Для гуртівників відображаються гуртові ціни, для інших — роздрібні
    • ☐ Перекреслена ціна та відсоток знижки відображаються для акційних товарів
    • ☐ Сторінки каталогу мають SSR та коректну Schema.org розмітку
    21.2. Модуль Користувачі
    • ☐ Реєстрація через email з підтвердженням працює
    • ☐ Авторизація, відновлення пароля працюють
    • ☐ Менеджер може змінити роль користувача (Клієнт → Гуртівник)
    • ☐ Розмежування доступу: клієнт не бачить гуртових цін, гуртівник не бачить адмін-панель
    • ☐ Списки бажань: створення, додавання товарів, масове додавання до кошика
    • ☐ Rate limiting на авторизацію: блокування після 5 невдалих спроб
    21.3. Кабінет клієнта/гуртівника
    • ☐ Профіль компанії: ЄДРПОУ, реквізити, форма власності зберігаються та підставляються в PDF
    • ☐ Кілька адрес доставки: додавання, редагування, вибір за замовчуванням
    • ☐ Історія замовлень: фільтрація, пошук, деталі з таймлайном статусів
    • ☐ Повторне замовлення: додає товари до кошика з перевіркою актуальних цін та залишків
    • ☐ Фінансовий розділ: загальна сума, графік закупівель, архів PDF-документів
    • ☐ Персональний менеджер: блок з контактами відображається, кнопка Telegram працює
    • ☐ Стрічка сповіщень: badge непрочитаних, типи сповіщень відповідають налаштуванням
    • ☐ Запит на статус гуртівника: форма відправляється, менеджер отримує сповіщення
    • ☐ Блок «Ваші умови»: мінімальна сума, кратність, знижки відображаються актуально
    21.4. Модуль Замовлення
    • ☐ Повний цикл: кошик → оформлення → підтвердження → зміна статусу → виконання
    • ☐ Швидке замовлення за кодами: ввід списку, валідація, додавання до кошика
    • ☐ Бізнес-правила гуртівників: мінімальна сума, мінімальна кількість, кратність
    • ☐ PDF-документи генеруються коректно: рахунок-фактура, видаткова накладна, комерційна пропозиція
    • ☐ Edge-cases: товар закінчився, зміна ціни, дублювання замовлення — обробляються коректно
    • ☐ Експорт замовлень у Excel/CSV з фільтрацією за датою, статусом, типом клієнта
    • ☐ Таймлайн статусів: клієнт бачить хронологію зміни статусів у кабінеті (Order_Status_History)
    • ☐ Автоскасування: замовлення у статусі «Нове» без дії менеджера > 72 годин автоматично скасовується з email-сповіщенням
    • ☐ Коментарі менеджера: внутрішнє поле коментаря зберігається і не видиме клієнту
    • ☐ Bulk-дії: масова зміна статусу, масовий друк накладних працюють для ≥ 50 замовлень
    • ☐ Інтеграція трекінгу НП: ТТН зберігається, клієнт отримує посилання на відстеження, статус оновлюється автоматично (cron)
    • ☐ Підтвердження оплати: менеджер може вручну підтвердити оплату; при підтвердженні — автозміна статусу замовлення на «Оплачене»

    • ☐ Транзакційність: rollback при критичній помилці
    • ☐ Історія цін зберігається при кожній зміні ціни

Структура бази даних (основні сутності)
Основні таблиці системи:
23.1. Users (Користувачі)
Поля: id, email, password_hash, full_name, phone, role (client/wholesaler/manager/admin), company_name, edrpou (ЄДРПОУ), legal_address, bank_iban, bank_name, bank_mfo, ownership_type (ФОП/ТОВ/ПП/інше), tax_system (with_vat/without_vat), contact_person_name, contact_person_phone, wholesale_status (none/pending/approved/rejected), wholesale_request_date, wholesale_approved_date, wholesale_monthly_volume (очікуваний обсяг закупівель), assigned_manager_id (зовнішній ключ на Users для персонального менеджера), referral_code (унікальний код для реферальної програми), notification_preferences (JSON: email/telegram/viber/cabinet для кожного типу сповіщень), google_id (унікальний ідентифікатор Google-акаунту для соціального логіну, nullable), telegram_chat_id (BIGINT, nullable — ID чату Telegram для прив’язки акаунту до бота, розділ 7.1.1), viber_user_id (VARCHAR, nullable — ID користувача Viber для прив’язки до бота, розділ 7.2.1), avatar_url, is_verified, created_at, updated_at.
23.2. Products (Товари)
Поля: id, code (VARCHAR, UNIQUE — унікальний код товару з Excel), name (VARCHAR 255), slug (VARCHAR 255, UNIQUE — URL-ідентифікатор, транслітерація назви), category_id (FK → Categories, nullable), price_retail (DECIMAL 10,2), price_wholesale (DECIMAL 10,2), price_retail_old (DECIMAL 10,2, nullable — попередня ціна для відображення перекресленої ціни, заповнюється автоматично при зміні), price_wholesale_old (DECIMAL 10,2, nullable), quantity (INT, default 0), is_promo (BOOLEAN, default false), promo_start_date (DATE, nullable), promo_end_date (DATE, nullable), is_active (BOOLEAN, default true — чи відображається на сайті), image_path (VARCHAR, nullable — шлях до головного зображення), sort_order (INT, default 0 — для ручного сортування), views_count (INT, default 0 — кількість переглядів для «Популярне»), orders_count (INT, default 0 — кількість замовлень для бейджа «Хіт»), created_at, updated_at. Таблиця містить лише комерційні дані, що імпортуються з Excel-файлу; опис товару зберігається окремо в Product_Content (розділ 23.3). Індекси: code (UNIQUE), slug (UNIQUE), category_id, is_active + is_promo, price_retail, created_at DESC.
23.3. Product_Content (Контент товарів)
Окрема таблиця для зберігання описів та розширеного контенту товарів, що заповнюється менеджером вручну через кабінет. Ізольована від Excel-імпорту — при завантаженні нового файлу дані контенту не перезаписуються та не видаляються.
Поля: id, product_code (зовнішній ключ, зв’язок з Products.code), short_description (короткий опис для картки товару в каталозі, до 200 символів), full_description (повний опис для сторінки товару, підтримка базового HTML-форматування), specifications (текстові характеристики: склад, об’єм, вага, призначення тощо), usage_instructions (інструкція із застосування), additional_images (масив шляхів до додаткових фотографій товару), video_url (посилання на відео-огляд, опціонально), seo_title (заголовок для SEO, опціонально), seo_description (мета-опис для SEO, опціонально), is_filled (прапорець: чи заповнений опис менеджером), filled_by (id менеджера, який заповнив), created_at, updated_at.
При відображенні картки товару система перевіряє наявність запису в Product_Content за product_code. Якщо опис ще не заповнений — відображається лише назва, ціна та фото. Менеджер в кабінеті бачить перелік товарів із позначкою, для яких опис ще не створено, що дозволяє поступово наповнювати каталог контентом.

Поля: id, user_id (FK → Users, nullable для гостьових замовлень), order_number (VARCHAR, унікальний, формат: YYYYMMDD-NNNN), status (new/processing/confirmed/paid/shipped/completed/cancelled/returned — повний перелік та правила переходу в розділі 5.2), client_type (retail/wholesale — фіксується при створенні), total_amount (DECIMAL 10,2 — підсумкова сума), discount_amount (DECIMAL 10,2, default 0), delivery_cost (DECIMAL 10,2, default 0), items_count (INT — кількість позицій), contact_name (VARCHAR — ПІБ замовника), contact_phone (VARCHAR), contact_email (VARCHAR), delivery_method (nova_poshta/ukrposhta/pickup/pallet — розділ 5.10), delivery_city (VARCHAR, nullable), delivery_warehouse_ref (VARCHAR, nullable — код відділення НП), delivery_address (TEXT, nullable — адреса для кур’єра/Укрпошти), tracking_number (VARCHAR, nullable — номер ТТН), payment_method (cod/bank_transfer/online/card_prepay — розділ 5.9), payment_status (pending/paid/partial/refunded), comment (TEXT, nullable — коментар клієнта), manager_comment (TEXT, nullable — внутрішній коментар менеджера), cancelled_reason (TEXT, nullable), cancelled_by (ENUM: client/manager, nullable), source (web/telegram_bot/viber_bot — звідки створено замовлення), utm_source (VARCHAR, nullable), utm_medium (VARCHAR, nullable), utm_campaign (VARCHAR, nullable), ip_address (VARCHAR, nullable), user_agent (VARCHAR, nullable), created_at, updated_at. Індекси: order_number (UNIQUE), user_id + created_at DESC, status, status + created_at, payment_status.
23.5. Order_Items (Позиції замовлення)
Поля: id, order_id (FK → Orders, ON DELETE CASCADE), product_id (FK → Products, ON DELETE SET NULL — товар може бути видалений, але позиція замовлення залишається), product_code (VARCHAR — snapshot коду товару на момент замовлення), product_name (VARCHAR — snapshot назви), price_at_order (DECIMAL 10,2 — зафіксована ціна на момент замовлення), quantity (INT), subtotal (DECIMAL 10,2 — price_at_order × quantity, обчислювальне поле), is_promo (BOOLEAN — чи був товар акційним при замовленні), created_at. Індекси: order_id, product_id.

Індекси: status + scheduled_at, created_by, published_at DESC.

Визначення ключових термінів, що використовуються в документації:
• Гуртівник (wholesaler) — зареєстрований користувач з підтвердженим статусом гуртового покупця; отримує доступ до гуртових цін та розширеного функціоналу (розділ 3.2)
• Бейдж — візуальна позначка на картці товару: «Новинка» (товар створений менше 14 днів тому), «Акція» (is_promo = true), «Хіт» (orders_count > порогового значення), «Топ» (views_count > порогового значення)
• ТТН (товарно-транспортна накладна) — номер відправлення в системі Нова Пошта або Укрпошта; зберігається в Orders.tracking_number та Deliveries.tracking_number
• ЄДРПОУ — код Єдиного державного реєстру підприємств та організацій України; використовується для ідентифікації юридичних осіб гуртівників при реєстрації
• SKU (Stock Keeping Unit) — унікальний код товару в системі; відповідає полю Products.code та імпортується з Excel-файлу (розділ 4.1)
• Webhook — HTTP-callback від зовнішнього сервісу; використовується для прийому оновлень від Telegram, Viber, Instagram, Нова Пошта та платіжних систем (розділ 8.10)
• Pipeline (обробки зображень) — автоматичний ланцюг перетворення зображення при завантаженні: оригінал → full (800×800) → medium (400×400) → thumbnail (150×150) → blur placeholder (20×20) (розділ 4.2.1)
• i18n-ready — архітектурна підготовка до мультимовності: всі рядки інтерфейсу винесені у файли локалізації, дати та валюти форматуються через Intl API (розділ 8.3)
• Seed-дані — тестовий набір даних для development-середовища: товари, категорії, замовлення, користувачі; генерується скриптом (розділ 19.2)
• Double opt-in — двоетапне підтвердження підписки на розсилку: реєстрація → лист з посиланням → підтвердження кліком (розділ 6.3, таблиця 23.43)
• Graceful degradation — стратегія обробки помилок: при збої компонента система продовжує працювати з обмеженою функціональністю замість повної відмови (розділ 17.4)
• Rate limiting — обмеження кількості запитів від одного клієнта за одиницю часу для захисту від DDoS та brute-force атак (розділ 9.4.1)
• CLS (Cumulative Layout Shift) — метрика Core Web Vitals; вимірює візуальну стабільність сторінки; ціль: ≤ 0.1 (розділ 14.6)
• LCP (Largest Contentful Paint) — метрика Core Web Vitals; час відображення найбільшого видимого елемента; ціль: ≤ 2.5с (розділ 14.6)
