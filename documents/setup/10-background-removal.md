# Автоматичне видалення фону з фото товару

Коли увімкнено, при завантаженні фото товару в адмінці система автоматично
видалить фон (від постачальника, з телефона, будь-якого кольору) і композитно
накладе товар на нейтральний фон сайту (`#f5f5f5`). Адмін не мусить нічого
переробляти у фотошопі.

Без налаштування — фіча просто прихована (UI-чекбокс не показується), решта
обробки фото працює як раніше: padding до квадрата + watermark.

## Як це працює

1. Адмін заходить на `/admin/products/<id>` → секція "Зображення".
2. Бачить чекбокс **☑ Автоматично видалити фон при завантаженні** (default ON).
3. Завантажує фото будь-якого формату (JPG/PNG/WebP, до 5 МБ, мін. 300×300 px).
4. Сервер відправляє оригінал у [remove.bg API](https://www.remove.bg/api).
5. remove.bg повертає PNG з прозорим фоном (вирізаний товар).
6. Sharp накладає cutout на `#f5f5f5` і генерує 4 варіанти (full/medium/thumbnail/blur).
7. Фото зберігається в R2/локально як завжди — на сайті виглядає на нейтральному фоні.

## Налаштування

### Крок 1. Створити акаунт remove.bg

1. Відкрити [https://www.remove.bg/users/sign_up](https://www.remove.bg/users/sign_up).
2. Зареєструватись (email або Google/GitHub).
3. Перейти на [https://www.remove.bg/api](https://www.remove.bg/api).
4. Натиснути "Get API Key" → скопіювати ключ.

### Крок 2. Додати в `.env`

```env
REMOVEBG_API_KEY=YOUR-API-KEY-HERE
```

### Крок 3. Перезапустити сервер

```bash
pm2 restart clean   # або інша команда твого деплою
```

Перевірка: відкрити `/admin/products/<будь-який-id>` → під списком фото має
з'явитись чекбокс "Автоматично видалити фон при завантаженні". Якщо немає —
ключ не підхопився, перевір `.env` і логи на старті.

## Bulk завантаження

**Так, працює**, з нюансами.

### Як це поводиться

- В адмінці input `<file multiple>` дозволяє вибрати кілька фото за раз.
- Сервер обробляє файли **послідовно**, по одному, не паралельно. Чому:
  - remove.bg має rate-limit (~5 запитів за секунду на пробному акаунті,
    більше для платних). Burst → 429 errors → втрата фото.
  - Sharp — CPU-важка операція. Паралельні pipelines на $5 VPS можуть
    з'їсти всю пам'ять.
- Якщо одне фото впало (поганий формат, занадто мале, BG removal не вийшло) —
  інші продовжують оброблятися. Результат повертається як `{ ok: [...], failed: [...] }`.

### Скільки часу на завантаження

| К-ть фото | Без BG removal | З BG removal                   |
| --------- | -------------- | ------------------------------ |
| 1         | ~1-2 с         | ~3-5 с                         |
| 5         | ~5-7 с         | ~15-25 с                       |
| 10        | ~10-15 с       | ~30-50 с                       |
| 20        | ~20-30 с       | ~60-100 с (близько до timeout) |

### Ліміти

- **Next.js API route timeout**: 60 секунд (App Router default). За цим
  лімітом запит обірветься. Тому **не завантажуй більше 15-20 фото за раз
  з BG removal** — або вимкни чекбокс для bulk.
- **remove.bg квота**: free tier — 50/міс. 10 товарів × 5 фото = 50 фото = вся
  квота за раз. Слідкуй за використанням.
- **Розмір файлу**: 5 МБ на файл. Якщо клієнт шле з iPhone Pro (10-12 МБ) —
  стиснути перед завантаженням (наприклад, через [tinypng.com](https://tinypng.com)).

### Рекомендована стратегія для bulk

Якщо треба залити 100+ старих фото:

1. **Не вмикай BG removal** для першого проходу — просто завантажуй усе
   через існуючу логіку padding'я (швидко, без витрат).
2. Потім **вибірково ставити BG removal** для конкретних товарів через
   видалення старого фото + повторне завантаження з чекбоксом.
3. Або купи Subscription M ($25 / 800 фото) і прожени батч за раз з BG removal.

## Обробка помилок

### remove.bg API не відповідає

Якщо API повернув 5xx, 429 (rate limit) або інші помилки — service логує warning
і повертає `null`. `processProductImage` падає на звичайну логіку: padding до
квадрата + watermark, без BG removal. **Завантаження не обірветься** — фото
просто збережеться зі старою (фоновою) поведінкою.

Як перевірити що було пропущено — глянути в логах:

```
warn  remove.bg API failed  status=429 body=...
```

### Перевищення квоти

Коли квота вичерпана, remove.bg повертає 402 (Payment Required) з повідомленням
у JSON. Service логує це як warning і повертає `null` — поведінка така ж, як
при тимчасовому збої. Звичайне padding-завантаження зберігається.

Перевірити стан квоти: [https://www.remove.bg/dashboard](https://www.remove.bg/dashboard).

## Альтернативи

Якщо remove.bg не підходить (дорого / проблеми з API):

1. **Photoroom API** — $99 / 5000 photos, схожий API: [photoroom.com](https://www.photoroom.com/api).
2. **Cloudinary AI** — частина пакета (від $89 / міс), але великий ecosystem.
3. **Self-hosted ONNX** — `@imgly/background-removal-node`, безкоштовно,
   але +50MB пакета і 3-5s інференсу на CPU. Окрема задача — заміна бекенду
   за тим самим інтерфейсом `removeBackground(buffer): Buffer | null`.

Для переключення на іншого провайдера — переписати тіло функції
`removeBackground` в `src/services/background-removal.ts`. Контракт той самий.

## Що ще варто знати

- **Watermark** накладається після BG removal, на готовий cutout. Текст
  `pulito.trade` з'явиться у нижньому правому куті — на фоні товару, а не
  на padding-зоні.
- **Якщо фон уже прозорий** (PNG з alpha) — система все одно проганяє через
  remove.bg, бо API очищає і "несправжню прозорість" від нечіткого matting.
  Якщо хочеш зекономити кредити — пропусти remove.bg для PNG, додай перевірку
  на наявність alpha-каналу в `processProductImage`.
- **Темна тема сайту**: cutout зливається з `#f5f5f5`, тому на dark mode
  фото залишається світлим квадратом на темно-синьому контейнері. Це поточна
  поведінка сайту — нормально, користувачі звикли. Для повноцінної підтримки
  treба зберігати окремий PNG-варіант з прозорістю — окрема задача.
