# 12 — Безпека

## 12.1 CSRF захист

### TC-SE-1.1 Запит без X-Requested-With
**Передумова:** Авторизований як admin, відкрити DevTools → Network
**Кроки:**
1. Відкрити DevTools → Console
2. Виконати: `fetch('/api/v1/auth/login', { method: 'POST', headers: {'Content-Type':'application/json'}, body: '{}' }).then(r=>r.json()).then(console.log)`
3. Перевірити відповідь
**Очікуваний результат:** `CSRF check failed: missing X-Requested-With header`
**Статус:** ⬜

## 12.2 Rate Limiting

### TC-SE-2.1 Глобальний rate limit (API)
**Передумова:** Відкрити `/auth/login`, підготувати 6+ запитів
**Кроки:**
1. Швидко оновити сторінку 120+ разів за хвилину (або через curl loop)
**Очікуваний результат:** 429 Too Many Requests з Retry-After header
**Статус:** ⬜

### TC-SE-2.2 Rate limit на Server Actions
**Передумова:** Відкрити `/auth/login`, підготувати 6+ запитів
**Кроки:**
1. Спробувати оформити 6+ замовлень за хвилину
**Очікуваний результат:** "Забагато запитів. Спробуйте пізніше." після 5-го
**Статус:** ⬜

### TC-SE-2.3 Rate limit на логін
**Передумова:** Відкрити `/auth/login`, підготувати 6+ запитів
**Кроки:**
1. 5 невдалих спроб входу підряд
**Очікуваний результат:** Блокування на 15 хвилин
**Статус:** ⬜

## 12.4 Security Headers

### TC-SE-4.1 Перевірка headers
**Передумова:** Авторизований як admin, відкрити DevTools → Network
**Кроки:**
1. Відкрити DevTools → Network → будь-який запит
2. Перевірити наявність:
   - `X-Frame-Options: DENY`
   - `X-Content-Type-Options: nosniff`
   - `Strict-Transport-Security`
   - `Content-Security-Policy`
   - `Referrer-Policy`
   - `Permissions-Policy`
**Очікуваний результат:** Всі заголовки присутні
**Статус:** ⬜

## 12.6 XSS

### TC-SE-6.1 XSS в пошуку
**Передумова:** Авторизований, відкрити сторінку з user-generated content
**Кроки:**
1. Ввести в пошук: `<script>alert('xss')</script>`
**Очікуваний результат:** Текст відображається як текст, alert не спрацьовує
**Статус:** ⬜

### TC-SE-6.2 XSS в відгуку
**Передумова:** Авторизований, відкрити сторінку з user-generated content
**Кроки:**
1. Залишити відгук з текстом: `<img src=x onerror=alert(1)>`
2. Якщо опублікований — перевірити на сторінці товару
**Очікуваний результат:** HTML екрановано, скрипт не виконується
**Статус:** ⬜

## 12.8 Негативні та граничні сценарії безпеки

### TC-SE-8.1 SQL injection у пошуку
**Передумова:** Відкрити `/catalog`, користувач не авторизований
**Тестові дані:**
- Пошуковий запит: `'; DROP TABLE products; --`
**Кроки:**
1. Перейти на `/catalog`
2. В поле пошуку ввести `'; DROP TABLE products; --`
3. Натиснути Enter або кнопку пошуку
4. Перевірити відповідь сторінки
5. Перевірити що таблиця products існує (через адмінку — товари відображаються)
**Очікуваний результат:** Пошук виконується без помилок. Результат: "Нічого не знайдено" або порожній список. Запит санітизовано, SQL не виконано. Таблиця products не пошкоджена.
**Статус:** ⬜

### TC-SE-8.2 XSS у тексті відгуку — script tag
**Передумова:** Увійти як `client@test.com` (роль: client), `/product/test-product`
**Тестові дані:**
- Текст відгуку: `<script>alert(1)</script>`
- Рейтинг: 5
**Кроки:**
1. Перейти на сторінку товару `/product/test-product`
2. Натиснути "Залишити відгук"
3. Ввести рейтинг: 5
4. В текстове поле ввести `<script>alert(1)</script>`
5. Натиснути "Опублікувати"
6. Оновити сторінку та знайти відгук (або підтвердити через адмінку)
**Очікуваний результат:** Відгук відображається як звичайний текст: `<script>alert(1)</script>`. HTML екрановано у виводі (`&lt;script&gt;`). JavaScript alert НЕ спрацьовує. В DOM немає елемента `<script>`.
**Статус:** ⬜

### TC-SE-8.3 Path traversal у назві файлу при upload
**Передумова:** Увійти як `admin@test.com` (роль: admin), `/admin/products/new`
**Тестові дані:**
- Файл: валідне JPEG зображення
- Назва файлу змінена на: `../../etc/passwd`
**Кроки:**
1. Перейти на `/admin/products/new`
2. Натиснути "Завантажити зображення"
3. Обрати JPEG файл з назвою `../../etc/passwd` (змінити через interceptor або curl)
4. Відправити запит
**Очікуваний результат:** HTTP 400 Bad Request. Повідомлення: "Некоректне ім'я файлу". Файл не збережено. Шлях серверної файлової системи не скомпрометовано.
**Статус:** ⬜

### TC-SE-8.4 JWT expired token
**Передумова:** Мати валідний JWT токен, дочекатися його закінчення (або модифікувати exp claim)
**Тестові дані:**
- JWT токен з `exp` в минулому (наприклад, `exp: 1700000000`)
**Кроки:**
1. Відкрити DevTools → Application → Cookies
2. Замінити значення auth token на JWT з простроченим `exp`
3. Виконати будь-який API запит (наприклад, оновити сторінку `/account`)
**Очікуваний результат:** HTTP 401 Unauthorized. Відповідь: `{ "error": "Token expired" }`. Redirect на `/login`. Cookie видалено.
**Статус:** ⬜

### TC-SE-8.5 JWT tampered signature
**Передумова:** Мати валідний JWT токен
**Тестові дані:**
- Оригінальний JWT: `eyJhbGciOi...`
- Модифікований: змінити останній символ signature частини
**Кроки:**
1. Відкрити DevTools → Application → Cookies
2. Скопіювати auth token
3. Змінити останній символ в signature частині JWT (третя секція після `.`)
4. Вставити модифікований JWT назад в cookie
5. Виконати запит на `/api/v1/account/profile`
**Очікуваний результат:** HTTP 401 Unauthorized. Відповідь: `{ "error": "Invalid token" }`. Redirect на `/login`.
**Статус:** ⬜

### TC-SE-8.6 Rate limit на login — перевищення ліміту
**Передумова:** Сторінка `/login`, жоден користувач не авторизований
**Тестові дані:**
- Email: `client@test.com`
- Пароль: `wrongpassword123` (невірний)
- Кількість спроб: 6
**Кроки:**
1. Перейти на `/login`
2. Ввести `client@test.com` / `wrongpassword123`
3. Натиснути "Увійти" — помилка "Невірний пароль"
4. Повторити кроки 2-3 ще 5 разів (всього 6 спроб за 15 хвилин)
**Очікуваний результат:** Після 5-ї спроби: HTTP 429. Повідомлення: "Забагато спроб. Спробуйте через 15 хвилин." Форма login стає disabled. Навіть з правильним паролем вхід заблоковано до закінчення таймауту.
**Статус:** ⬜

### TC-SE-8.7 CSRF — POST запит без X-Requested-With header
**Передумова:** Увійти як `client@test.com` (роль: client)
**Тестові дані:**
- Endpoint: `/api/v1/cart/add`
- Body: `{ "productId": "123", "quantity": 1 }`
**Кроки:**
1. Відкрити DevTools → Console
2. Виконати: `fetch('/api/v1/cart/add', { method: 'POST', headers: {'Content-Type':'application/json'}, body: JSON.stringify({productId:'123',quantity:1}) }).then(r => console.log(r.status, r.statusText))`
3. Перевірити відповідь (без header `X-Requested-With`)
**Очікуваний результат:** HTTP 403. Відповідь: "CSRF check failed: missing X-Requested-With header". Товар НЕ додано до кошика.
**Статус:** ⬜

### TC-SE-8.8 Upload файлу більше 10MB
**Передумова:** Увійти як `admin@test.com` (роль: admin), `/admin/products/new`
**Тестові дані:**
- Файл: JPEG зображення розміром 12MB
**Кроки:**
1. Перейти на `/admin/products/new`
2. Натиснути "Завантажити зображення"
3. Обрати JPEG файл розміром 12MB
4. Дочекатися відповіді
**Очікуваний результат:** Помилка: "Файл занадто великий. Максимальний розмір: 10MB." Файл не завантажено. Progress bar зникає.
**Статус:** ⬜

### TC-SE-8.9 Upload non-image файлу як аватар
**Передумова:** Увійти як `client@test.com` (роль: client), `/account/profile`
**Тестові дані:**
- Файл: `malware.exe` перейменований в `photo.jpg` (MIME type не відповідає)
**Кроки:**
1. Перейти на `/account/profile`
2. Натиснути "Змінити аватар"
3. Обрати файл `malware.exe` (перейменований в `photo.jpg`)
4. Підтвердити завантаження
**Очікуваний результат:** Помилка: "Невідомий формат. Дозволені: JPEG, PNG, WebP." Файл відхилено на основі magic bytes (не розширення). Аватар не змінено.
**Статус:** ⬜

### TC-SE-8.10 Доступ до admin route як client
**Передумова:** Увійти як `client@test.com` (роль: client)
**Тестові дані:** немає
**Кроки:**
1. Увійти як `client@test.com`
2. Вручну перейти на `/admin/dashboard`
3. Спробувати `/admin/orders`
4. Спробувати `/admin/products`
**Очікуваний результат:** Кожен запит повертає HTTP 403 Forbidden. Відображається сторінка помилки: "У вас немає доступу до цієї сторінки." Жодні адмін-дані не завантажуються.
**Статус:** ⬜

### TC-SE-8.11 API key з невірним scope
**Передумова:** Створити API ключ з scope `read:products` через `/admin/api-keys`
**Тестові дані:**
- API key scope: `read:products`
- Запит: `POST /api/v1/products` (потребує `write:products`)
**Кроки:**
1. В адмінці створити API ключ з scope `read:products`
2. Виконати curl: `curl -X POST /api/v1/products -H "X-API-Key: <key>" -H "Content-Type: application/json" -d '{"name":"Test"}'`
3. Перевірити відповідь
**Очікуваний результат:** HTTP 403. Відповідь: `{ "error": "Insufficient permissions. Required scope: write:products" }`. Продукт не створено.
**Статус:** ⬜

### TC-SE-8.12 Brute force OTP — перевищення спроб
**Передумова:** Увійти як `client@test.com`, увімкнена 2FA, сторінка вводу OTP
**Тестові дані:**
- Невірний OTP: `000000`
- Кількість спроб: 6
**Кроки:**
1. Ініціювати логін з 2FA
2. На сторінці вводу OTP ввести `000000`
3. Натиснути "Підтвердити" — помилка "Невірний код"
4. Повторити кроки 2-3 ще 5 разів (всього 6 спроб)
**Очікуваний результат:** Після 5-ї невдалої спроби: "Забагато спроб. Почніть заново." OTP сесія інвалідовано. Redirect на `/login`. Потрібно заново ввести логін/пароль та отримати новий OTP.
**Статус:** ⬜
