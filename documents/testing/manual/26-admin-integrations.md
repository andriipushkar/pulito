## 26.1 Інтеграція 1С — сторінка та API-ключі

### TC-INT-1.1 Відображення сторінки інтеграцій
**Передумова:** Адміністратор авторизований, перейти на `/admin/integrations`. Сторінка завантажена.
**Кроки:**
1. Відкрити сторінку `/admin/integrations`
2. Дочекатися завантаження (зникає Spinner по центру)
3. Перевірити наявність заголовка та секцій
**Очікуваний результат:**
- Заголовок сторінки — «1C / BAS Integration» (h1, text-2xl font-bold)
- Сторінка поділена на 2 секції: «API Keys» та «Sync History»
- Якщо ключів немає — у таблиці «API Keys» відображається «No API keys yet» (сірий текст, вирівняний по центру)
- Якщо синхронізацій немає — у таблиці «Sync History» відображається «No sync history yet»
**Статус:** ⬜

### TC-INT-1.2 Створення нового API-ключа
**Передумова:** Адміністратор авторизований, сторінка `/admin/integrations`.
**Тестові дані:**
- Key name: «1C Production»
**Кроки:**
1. Знайти поле введення з placeholder «Key name (e.g. 1C Production)» та кнопку «Create API Key»
2. Перевірити що кнопка «Create API Key» неактивна (disabled) коли поле порожнє
3. Ввести у поле значення «1C Production»
4. Перевірити що кнопка «Create API Key» стала активною
5. Натиснути кнопку «Create API Key»
6. Дочекатися завершення (під час створення текст кнопки змінюється на «Creating...»)
**Очікуваний результат:**
- Зʼявляється зелений блок (border-green-300, bg-green-50) з текстом: «New API key created. Copy it now — it will not be shown again:»
- Під текстом відображається згенерований API-ключ моноширинним шрифтом (`code`, break-all)
- Поруч із ключем посилання «Copy & dismiss» (зелене підкреслене)
- Поле введення очищується
- У таблиці «API Keys» зʼявляється новий рядок із назвою «1C Production»
- Новий ключ має permissions: products, orders, stock, prices (всі true)
**Статус:** ⬜

### TC-INT-1.3 Копіювання та приховання створеного ключа
**Передумова:** Адміністратор щойно створив API-ключ, зелений блок із ключем відображається.
**Кроки:**
1. Натиснути посилання «Copy & dismiss»
**Очікуваний результат:**
- API-ключ копіюється у буфер обміну (navigator.clipboard.writeText)
- Зелений блок із ключем зникає
- Ключ більше ніде не відображається на сторінці (тільки prefix)
**Статус:** ⬜

### TC-INT-1.4 Таблиця API-ключів — колонки та дані
**Передумова:** Адміністратор авторизований, сторінка `/admin/integrations`. В базі є щонайменше 2 API-ключі (один активний, один деактивований).
**Тестові дані:** API-ключ «1C Production», prefix «clsk_abc1...», статус Active, Last Used — «15.03.2024, 14:30», Created — «01.01.2024».
**Кроки:**
1. Перевірити колонки таблиці «API Keys»
2. Перевірити відображення активного ключа
3. Перевірити відображення деактивованого ключа
**Очікуваний результат:**
- Таблиця має 6 колонок: «Name», «Prefix», «Status», «Last Used», «Created», «Actions»
- Активний ключ: статус — бейдж «Active» (bg-green-100 text-green-800), у колонці Actions — посилання «Deactivate» (червоне)
- Деактивований ключ: статус — бейдж «Inactive» (bg-gray-100 text-gray-600), у колонці Actions — порожньо (немає кнопки)
- Prefix відображається як `code` з трикрапкою: «clsk_abc1...»
- Last Used — дата/час або «Never» якщо ключ ніколи не використовувався
- Created — дата без часу
**Статус:** ⬜

### TC-INT-1.5 Деактивація API-ключа
**Передумова:** Адміністратор авторизований, сторінка `/admin/integrations`. Є активний API-ключ.
**Кроки:**
1. Знайти активний API-ключ у таблиці
2. Натиснути посилання «Deactivate» (text-red-600) у колонці «Actions»
3. Дочекатися оновлення таблиці
**Очікуваний результат:**
- Статус ключа змінюється на «Inactive» (bg-gray-100 text-gray-600)
- Посилання «Deactivate» зникає з колонки «Actions»
- Таблиця оновлюється автоматично (fetchData)
**Статус:** ⬜

### TC-INT-1.6 Валідація — створення ключа з порожньою назвою
**Передумова:** Адміністратор авторизований, сторінка `/admin/integrations`.
**Кроки:**
1. Залишити поле «Key name» порожнім
2. Перевірити стан кнопки «Create API Key»
3. Ввести тільки пробіли «   »
4. Перевірити стан кнопки
**Очікуваний результат:** Кнопка «Create API Key» залишається disabled коли поле порожнє або містить лише пробіли (`!newKeyName.trim()`). Запит на сервер не відправляється.
**Статус:** ⬜

## 26.2 Інтеграція 1С — Sync History

### TC-INT-2.1 Таблиця історії синхронізацій
**Передумова:** Адміністратор авторизований, сторінка `/admin/integrations`. В базі є записи синхронізацій з різними статусами.
**Тестові дані:** Синхронізація ID=1, type «one_c» → «1C», direction «inbound_sync» → «Inbound», entityType «products» → «Products», status «completed», progress 150/150, 0 failed.
**Кроки:**
1. Прокрутити до секції «Sync History»
2. Перевірити колонки таблиці
3. Перевірити відображення рядків
**Очікуваний результат:**
- Заголовок секції — «Sync History» (h2, text-lg font-semibold)
- Таблиця має 8 колонок: «ID», «Type», «Direction», «Entity», «Status», «Progress», «Started», «Completed»
- Type відображається UPPERCASE з заміною «_» на «/» (наприклад, «one_c» → «ONE/C»)
- Direction відображається Capitalize з видаленням «_sync» (наприклад, «inbound_sync» → «Inbound»)
- Entity відображається Capitalize (наприклад, «products» → «Products»)
- Status — кольоровий бейдж: pending (bg-yellow-100 text-yellow-800), running (bg-blue-100 text-blue-800), completed (bg-green-100 text-green-800), failed (bg-red-100 text-red-800)
- Progress — формат «processed/total», якщо є failed — додається червоний текст «(N failed)»
- Дати Started/Completed — toLocaleString() або «—» якщо null
**Статус:** ⬜

### TC-INT-2.2 Синхронізація з помилками — відображення failed
**Передумова:** Адміністратор авторизований, сторінка `/admin/integrations`. Є синхронізація з itemsFailed > 0.
**Тестові дані:** Синхронізація: itemsTotal=100, itemsProcessed=95, itemsFailed=5, status «completed».
**Кроки:**
1. Знайти рядок синхронізації з помилками у таблиці
2. Перевірити колонку «Progress»
**Очікуваний результат:** У колонці Progress відображається «95/100» та поруч червоним кольором (text-red-600) текст «(5 failed)».
**Статус:** ⬜

## 26.3 Фічефлаги

### TC-INT-3.1 Відображення списку фічефлагів
**Передумова:** Адміністратор авторизований, перейти на `/admin/feature-flags`. В базі є щонайменше 2 фічефлаги (один увімкнений, один вимкнений).
**Тестові дані:** Фічефлаг key=«enable_new_checkout», description=«Новий чекаут v2», isEnabled=true, rolloutPercent=75, targetRoles=[«admin», «wholesaler»].
**Кроки:**
1. Відкрити сторінку `/admin/feature-flags`
2. Дочекатися завантаження (зникає скелетон з 5 рядків і 4 колонок)
3. Перевірити заголовок та кнопку
4. Перевірити список фічефлагів
**Очікуваний результат:**
- Заголовок сторінки — «Фічефлаги» (text-xl font-bold)
- Кнопка «+ Додати фічефлаг» у верхній частині сторінки
- Кожен фічефлаг відображається як окремий блок (картка з border та padding)
- В кожному блоці зліва направо: кнопка-бейдж статусу «ON» (bg-green-100 text-green-700) або «OFF» (bg-gray-100 text-gray-500), ключ моноширинним шрифтом (code, font-mono), опис сірим текстом або «—», відсоток розкатки (наприклад, «75%»), ролі (наприклад, «Ролі: admin, wholesaler»), кнопка «Редагувати», посилання «Видалити» (червоне)
**Статус:** ⬜

### TC-INT-3.2 Порожній стан фічефлагів
**Передумова:** Адміністратор авторизований, сторінка `/admin/feature-flags`. В базі немає фічефлагів.
**Кроки:**

**Очікуваний результат:** Відображається текст «Фічефлагів немає» по центру, сірим кольором.
**Статус:** ⬜

### TC-INT-3.3 Створення нового фічефлагу
**Передумова:** Адміністратор авторизований, сторінка `/admin/feature-flags`.
**Тестові дані:**
- Ключ: «enable_new_checkout»
- Опис: «Новий чекаут v2»
- Розкатка: 50%
- Ролі: «admin, wholesaler»
- ID користувачів: «1, 42, 100»
- Увімкнено: так (чекбокс)
**Кроки:**
1. Натиснути кнопку «+ Додати фічефлаг»
2. Перевірити що зʼявилася форма, текст кнопки змінився на «Скасувати»
3. Ввести у поле «Ключ» (placeholder «new-checkout-flow») значення «enable_new_checkout»
4. Ввести у поле «Опис» (placeholder «Новий флоу оформлення») значення «Новий чекаут v2»
5. Перемістити повзунок «Розкатка» на 50% (перевірити що лейбл змінюється: «Розкатка: 50%»)
6. Ввести у поле «Ролі (через кому)» (placeholder «admin, wholesaler») значення «admin, wholesaler»
7. Ввести у поле «ID користувачів (через кому)» (placeholder «1, 42, 100») значення «1, 42, 100»
8. Поставити чекбокс «Увімкнено»
9. Натиснути кнопку «Створити»
**Очікуваний результат:**
- Зʼявляється toast-повідомлення «Фічефлаг створено» (success)
- Форма ховається
- У списку зʼявляється новий блок: бейдж «ON» (зелений), ключ «enable_new_checkout» моноширинним шрифтом, опис «Новий чекаут v2», «50%», «Ролі: admin, wholesaler»
- Поля форми скидаються (ключ порожній, опис порожній, розкатка 100%, ролі порожні, userIds порожні, чекбокс знятий)
**Статус:** ⬜

### TC-INT-3.4 Валідація — ключ обовʼязковий

## 26.4 Підписки

### TC-INT-4.1 Відображення таблиці підписок
**Передумова:** Адміністратор авторизований, перейти на `/admin/subscriptions`. В базі є підписки з різними статусами.
**Тестові дані:** Підписка: userName=«Олена Сидоренко», userEmail=«olena@example.com», frequency=«monthly», status=«active», nextDelivery=«2024-04-15», itemsCount=5, createdAt=«2024-01-10».
**Кроки:**
1. Відкрити сторінку `/admin/subscriptions`
2. Дочекатися завантаження (зникає скелетон з 6 рядків і 7 колонок)
3. Перевірити заголовок та таблицю
**Очікуваний результат:**
- Заголовок: «Підписки (N)» де N — кількість підписок у дужках, стиль N — text-base font-normal text-secondary
- Таблиця має 6 колонок: «Користувач», «Частота», «Статус», «Наступна доставка», «Товарів», «Створено»
- Колонка «Користувач»: імʼя жирним шрифтом, email — дрібний сірий текст з ml-2
- Колонка «Частота»: «Щотижня» / «Раз на 2 тижні» / «Щомісяця» (сірий текст)
- Колонка «Статус»: бейдж «Активна» (bg-green-100 text-green-700), «Призупинена» (bg-yellow-100 text-yellow-700), «Скасована» (bg-gray-100 text-gray-500)
- Колонка «Наступна доставка»: дата у форматі uk-UA (наприклад, «15.04.2024») або «—»
- Колонка «Товарів»: числове значення, вирівняне праворуч
- Колонка «Створено»: дата у форматі uk-UA
**Статус:** ⬜

### TC-INT-4.2 Фільтрація підписок за статусом — «Активні»
**Передумова:** Адміністратор авторизований, сторінка `/admin/subscriptions`. В базі є підписки з різними статусами.

3. Обрати «Активні» (value=«active»)
4. Дочекатися перезавантаження даних
**Очікуваний результат:** У таблиці відображаються тільки підписки зі статусом «Активна» (зелений бейдж). Кількість у заголовку оновлюється.
**Статус:** ⬜

### TC-INT-4.3 Фільтрація підписок за статусом — «Призупинені»

### TC-INT-4.4 Фільтрація підписок за статусом — «Скасовані»
