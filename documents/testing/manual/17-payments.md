# 17 — Платіжні системи: LiqPay, Monobank, WayForPay

> **Тестові ключі:** налаштовуються в `/admin/payment-settings`
> **Передумова:** для кожного тесту — створити замовлення з онлайн-оплатою

---

## 17.1 LiqPay

### TC-PM-1.1 Створення платежу LiqPay
**Передумова:** Замовлення з оплатою онлайн, провайдер LiqPay
**Кроки:**
1. Оформити замовлення → обрати "Оплата онлайн" → LiqPay
2. Підтвердити замовлення
3. Перевірити redirect на checkout.liqpay.ua
4. Перевірити що на сторінці LiqPay: сума в UAH, опис "Замовлення #XXX"
**Очікуваний результат:** Redirect на LiqPay з правильною сумою
**Статус:** ⬜

### TC-PM-1.2 Успішна оплата LiqPay (sandbox)
**Передумова:** Кошик з товарами, крок оплати, обрано LiqPay
**Тестові дані:**
- Картка: `4242424242424242`
- Термін: будь-який у майбутньому
- CVV: `123`
**Кроки:**
1. Ввести тестову картку
2. Підтвердити оплату
3. Перевірити redirect на сайт
4. Перевірити статус замовлення: "Оплачено"
5. Перевірити що webhook доставлено (в `/admin/webhook-logs`)
**Очікуваний результат:** Оплата пройшла, статус "Оплачено", конверсія відправлена в аналітику
**Статус:** ⬜

### TC-PM-1.3 Невдала оплата LiqPay
**Передумова:** Кошик з товарами, крок оплати, обрано LiqPay
**Кроки:**
1. На сторінці LiqPay натиснути "Скасувати"
2. Перевірити redirect на сайт
3. Перевірити статус замовлення: "Очікує оплати"
**Очікуваний результат:** Замовлення залишається неоплаченим
**Статус:** ⬜

### TC-PM-1.4 Повторна оплата LiqPay
**Передумова:** Замовлення зі статусом "Очікує оплати"
**Кроки:**
1. В кабінеті перейти до замовлення → "Оплатити"
2. Пройти оплату
**Очікуваний результат:** Оплата успішна, статус оновлено
**Статус:** ⬜

## 17.2 Monobank

**Передумова:** Замовлення з онлайн-оплатою, провайдер Monobank
**Кроки:**
1. Оформити замовлення → Monobank
2. Перевірити redirect на pay.mbnk.biz або подібне
3. Перевірити суму (в копійках: 100.50 грн = 10050)
**Очікуваний результат:** Redirect на Monobank з правильною сумою
**Статус:** ⬜

### TC-PM-2.2 Успішна оплата Monobank
**Передумова:** Кошик з товарами, крок оплати, обрано Monobank
**Кроки:**
1. Оплатити через Monobank (тестовий режим)
2. Перевірити redirect назад
3. Перевірити статус замовлення
4. Перевірити webhook з EC підписом в `/admin/webhook-logs`
**Очікуваний результат:** Оплата підтверджена, підпис верифіковано
**Статус:** ⬜

### TC-PM-2.3 Webhook replay protection
**Передумова:** Кошик з товарами, крок оплати, обрано Monobank
**Кроки:**
1. В webhook-logs знайти successful webhook
2. Перевірити що повторний webhook з тим самим transactionId ігнорується (лог "Duplicate webhook ignored")
**Очікуваний результат:** Дублікат ігнорується, замовлення не змінюється
**Статус:** ⬜

3. Перевірити суму та опис

**Статус:** ⬜

**Передумова:** Кошик з товарами, крок оплати, обрано WayForPay
**Кроки:**

2. Перевірити redirect

**Статус:** ⬜

2. Перевірити формат orderReference: `order_<id>_<timestamp>`
**Очікуваний результат:** Підпис валідний, orderId коректно витягнутий
**Статус:** ⬜

## 17.4 Загальні перевірки оплати

### TC-PM-4.1 Server-side tracking при оплаті
**Передумова:** Успішна оплата будь-яким провайдером
**Кроки:**
1. Перевірити логи сервера — `trackPurchase` викликано
2. Якщо FB CAPI налаштовано: перевірити Events Manager
3. Якщо GA4 налаштовано: перевірити Real-time Reports
**Очікуваний результат:** Конверсія відправлена в Facebook CAPI та GA4 Measurement Protocol
**Статус:** ⬜

### TC-PM-4.2 Перевірка суми (amount validation)
**Передумова:** Кошик з товарами, крок оплати, обрано LiqPay/Monobank/WFP
**Кроки:**
1. Створити замовлення на 500 грн
2. Симулювати webhook з amount=300 (невідповідність)
**Очікуваний результат:** Лог "Payment amount mismatch", статус залишається pending
**Статус:** ⬜

### TC-PM-4.3 Вже оплачене замовлення
**Передумова:** Кошик з товарами, крок оплати, обрано LiqPay/Monobank/WFP
**Кроки:**
1. Повторити webhook для вже оплаченого замовлення
**Очікуваний результат:** Лог "Order already paid, ignoring callback"
**Статус:** ⬜

### TC-PM-4.4 Налаштування в адмінці
**Передумова:** `/admin/payment-settings`
**Кроки:**
1. Перевірити конфігурацію кожного провайдера (ключі замасковані)
2. Натиснути "Тест підключення" для кожного
**Очікуваний результат:** Статус "Підключено" або помилка конкретного провайдера
**Статус:** ⬜

## 17.5 Негативні та граничні сценарії оплати

### TC-PM-5.1 LiqPay: скасування на сторінці оплати
**Передумова:** Замовлення #1001 з онлайн-оплатою, провайдер LiqPay, redirect на checkout.liqpay.ua виконано
**Тестові дані:**
- Замовлення: #1001, сума 750 грн
**Кроки:**
1. На сторінці LiqPay натиснути "Скасувати" (або закрити вкладку)
2. Перевірити redirect назад на сайт
3. Перейти в `/account/orders/1001`
4. Перевірити статус замовлення
**Очікуваний результат:** Redirect на сторінку замовлення. Статус: "payment_pending" (Очікує оплати). Відображається кнопка "Оплатити повторно". Замовлення НЕ скасовано автоматично.
**Статус:** ⬜

### TC-PM-5.2 Monobank: webhook timeout — callback не прийшов протягом 5 хвилин
**Передумова:** Замовлення #1002 з онлайн-оплатою, провайдер Monobank. Оплата ініційована, але webhook не доставлено.
**Тестові дані:**
- Замовлення: #1002, сума 500 грн
- Час очікування: 5 хвилин
**Кроки:**
1. Оформити замовлення з Monobank оплатою
2. Дочекатися redirect назад на сайт (або повернутися вручну)
3. Зачекати 5 хвилин без webhook callback
4. Перевірити статус замовлення в `/account/orders/1002`
5. Перевірити `/admin/webhook-logs` — відсутність записів для замовлення
**Очікуваний результат:** Замовлення залишається у статусі "payment_pending". В адмінці відображається попередження: "Webhook не отримано". Доступна кнопка "Перевірити статус вручну" для адміна.
**Статус:** ⬜

### TC-PM-5.3 WayForPay: невалідний номер картки
**Передумова:** Замовлення #1003 з онлайн-оплатою, провайдер WayForPay, redirect на secure.wayforpay.com
**Тестові дані:**
- Номер картки: `1234567890123456` (невалідний за алгоритмом Луна)
- Термін: 12/27
- CVV: 123
**Кроки:**
1. На сторінці WayForPay ввести невалідний номер картки `1234567890123456`
2. Ввести термін `12/27` та CVV `123`
3. Натиснути "Оплатити"
**Очікуваний результат:** Помилка на стороні WayForPay: "Невірний номер картки" або "Card number is invalid". Оплата не проходить. При поверненні на сайт статус замовлення: "payment_pending".
**Статус:** ⬜

### TC-PM-5.4 Подвійний клік "Оплатити" — idempotency
**Передумова:** Увійти як `client@test.com` (роль: client), `/checkout`, замовлення готове до оплати
**Тестові дані:**
- Замовлення на суму 600 грн
**Кроки:**
1. На сторінці checkout натиснути "Оплатити" двічі швидко (double click)
2. Перевірити кількість створених платіжних сесій в `/admin/webhook-logs`
3. Перевірити що списання відбулося один раз
**Очікуваний результат:** Кнопка "Оплатити" стає disabled після першого кліку. Створено лише одну платіжну сесію (idempotency key запобігає дублікату). Один redirect на платіжну сторінку. Подвійне списання неможливе.
**Статус:** ⬜

### TC-PM-5.5 Оплата зі зміненою ціною товару під час checkout
**Передумова:** Увійти як `client@test.com` (роль: client). Товар "Засіб X" в кошику за 200 грн. Адмін змінює ціну на 250 грн.
**Тестові дані:**
- Товар: "Засіб X"
- Ціна при додаванні в кошик: 200 грн
- Нова ціна (змінена адміном): 250 грн
**Кроки:**
1. Додати "Засіб X" (200 грн) в кошик
2. Перейти на `/checkout`
3. В іншому вікні (адмінка) змінити ціну "Засіб X" на 250 грн
4. Натиснути "Оплатити" в checkout
**Очікуваний результат:** Попередження: "Ціна товару 'Засіб X' змінилася: 200 грн → 250 грн. Підтвердіть нову суму замовлення." Кошик перераховано. Користувач повинен підтвердити нову суму перед оплатою.
**Статус:** ⬜

### TC-PM-5.6 Часткове повернення коштів (50%)
**Передумова:** Увійти як `admin@test.com` (роль: admin), `/admin/orders/1004`. Замовлення #1004 оплачено, сума 1000 грн.
**Тестові дані:**
- Замовлення: #1004, сума 1000 грн, статус: "paid"
- Сума повернення: 500 грн (50%)
**Кроки:**
1. Перейти на `/admin/orders/1004`
2. Натиснути "Повернення коштів"
3. Обрати "Часткове повернення"
4. Ввести суму: 500 грн
5. Вказати причину: "Повернення частини товарів"
6. Підтвердити
**Очікуваний результат:** Статус замовлення: "partially_refunded". В деталях: "Повернено 500 грн з 1000 грн". Webhook refund відправлено провайдеру. Бали лояльності перераховані пропорційно (списано 50% нарахованих балів).
**Статус:** ⬜

### TC-PM-5.7 Повне повернення коштів
**Передумова:** Увійти як `admin@test.com` (роль: admin), `/admin/orders/1005`. Замовлення #1005 оплачено, сума 800 грн. За замовлення нараховано 80 балів лояльності.
**Тестові дані:**
- Замовлення: #1005, сума 800 грн, статус: "paid"
- Бали лояльності за замовлення: 80
**Кроки:**
1. Перейти на `/admin/orders/1005`
2. Натиснути "Повернення коштів"
3. Обрати "Повне повернення"
4. Підтвердити
5. Перевірити баланс лояльності клієнта
**Очікуваний результат:** Статус замовлення: "refunded". Кошти повернені через платіжний провайдер. Бали лояльності (80) списані з балансу клієнта. В історії транзакцій: "Списано 80 балів — повернення замовлення #1005".
**Статус:** ⬜

### TC-PM-5.8 Webhook з невалідним підписом
**Передумова:** Тестове середовище з доступом до curl або Postman
**Тестові дані:**
- Endpoint: `/api/v1/webhooks/liqpay`
- Body: валідний JSON payload
- Signature: `invalid_signature_value`
**Кроки:**
1. Скопіювати body успішного webhook з `/admin/webhook-logs`
2. Відправити POST на `/api/v1/webhooks/liqpay` з правильним body, але невалідним підписом
3. Перевірити відповідь
4. Перевірити що статус замовлення не змінився
**Очікуваний результат:** HTTP 401 Unauthorized. Відповідь: `{ "error": "Invalid webhook signature" }`. Замовлення не підтверджено. В логах: "Webhook signature verification failed".
**Статус:** ⬜

### TC-PM-5.9 Оплата при maintenance mode
**Передумова:** Увімкнути maintenance mode в `/admin/settings`. Замовлення створене до увімкнення maintenance.
**Тестові дані:**
- Замовлення: #1006, сума 400 грн
**Кроки:**
1. Створити замовлення з онлайн-оплатою
2. В адмінці увімкнути maintenance mode
3. Спробувати перейти до оплати (натиснути "Оплатити" в кабінеті)
**Очікуваний результат:** Відображається сторінка maintenance: "Сайт на технічному обслуговуванні. Спробуйте пізніше." Redirect на платіжну систему не виконується. Замовлення залишається у статусі "payment_pending".
**Статус:** ⬜

### TC-PM-5.10 Callback URL manipulation
**Передумова:** Тестове середовище з доступом до curl або interceptor
**Тестові дані:**
- Оригінальний callback URL: `https://shop.com/api/v1/webhooks/wayforpay`
- Маніпульований: змінити параметри orderReference або amount
**Кроки:**
1. Перехопити redirect з WayForPay (через proxy або DevTools)
2. Змінити параметр `amount` з `500` на `1` в callback URL
3. Відправити модифікований callback на сервер
4. Перевірити відповідь та статус замовлення
**Очікуваний результат:** Сервер перевіряє підпис (HMAC-MD5). Підпис не збігається через змінені параметри. HTTP 401: "Signature check failed". Замовлення не підтверджено. В логах: "Callback signature mismatch".
**Статус:** ⬜
