## 0. Стартова ситуація → відновлено

Сайт віддавав **internal server error**, PM2 = `online` з 4431 рестартом (crash-loop).

**Причина:** `.next` у corrupt стані після OOM-killed білду (`BUILD_ID`, `build-manifest.json` відсутні), кожен запит — `Error: different slug names for the same dynamic path`.

**Що зробив:**

1. `pm2 stop pulito`
2. `rm -rf .next`
3. Білд із **6 GB heap** (default 4096 MB не вистачало → OOM)
4. `pm2 restart pulito` → HTTP 200

## 1. Виправлення цієї багатогодинної сесії

### Безпека / Auth

- **SSE auth redesign.** Видалив query-string token, додав `POST /api/v1/admin/sse-grant` (за `withRole2fa('admin')`) що видає HttpOnly cookie з 5-хв TTL. Стрім читає cookie, не query. Закриває "manager bypass + token in URL → referer/log leak".
  - `src/services/token.ts` — `signSseGrantToken`/`verifySseGrantToken` + constant TTL
  - `src/utils/cookies.ts` — `serializeSseGrantCookie`/`getSseGrantFromCookies`
  - `src/app/api/v1/admin/sse-grant/route.ts` (новий)
  - `src/app/api/v1/admin/notifications/stream/route.ts` — читає cookie, відхиляє manager, admin-only
  - `src/hooks/useAdminNotifications.ts` — POST до grant перед EventSource, refresh кожні 4 хв
- **notifications/stream race**: захопити `lastCheck` ДО запиту + `polling` flag проти накладання тіків.

### Гроші

- **return-request.ts**: `Number(Decimal) * qty` → `Prisma.Decimal.mul()` — копійчана точність.

### Аудит

- **products/[id]/variants** POST — додано `logAudit` на створення SKU.
- **billing/webhook** — `logger.warn` для `paid:false` (раніше silent no-op).

### Перформанс

- **orders/bulk-ttn** — pre-validate усі orders без API-викликів, потім parallel chunks по 5 проти Nova Poshta. 100 послідовних викликів (≈150 с) → ≈30 с. Вкладається в nginx 60 с default.

### React / UX

- **admin/page.tsx**: миттєвий pull при зміні refresh interval (через `useRef` first-run flag).
- **SpecsEditor**: decode HTML entities (`&amp;`/`&lt;`/`&gt;`/`&quot;`/`&#39;`).
- **PalletListSection**: екстрагнуто `sendStatusUpdate` — нерекурсивний flow, чітко linear.
- **products + orders focus**: deps через `IDs.join(',')` замість `length` — фокус reset при swap зі збереженням count.
- **analytics:114**: округлення marginPct до сотих (`* 10000 / 100`) — копійчана точність маржі.
- **CreateTTNForm**: коментар-документація про localStorage (NP reference IDs не секрет).

### Storefront

- **wholesale-request**: повний reset форми після успішного submit + snapshot у `data`.
- **orders/[id] cancel**: `router.refresh()` для інвалідації route-cache.

### i18n

- Усі 4 локалі (uk/en/pl/ro) розширено:
  - `home.{promoTitle, newTitle, popularTitle, recentlyViewed}`

  - `accountNav.sections.*` (5) + `accountNav.items.*` (16)

  - `page.tsx` — `getTranslations('home')`
  - `BannerSlider` — `useTranslations('banner')`
  - `ProductCarousel` — `t('viewAll')`

## 2. Cross-tenant пункти (5) — чому НЕ виправлені

З оригінального аудиту:

- `analytics/route.ts:49,88,123,247,311`
- `dashboard/recommendations/route.ts:38-58`
- `sidebar-counts/route.ts:11-15`
- `audit-log/route.ts:48`
- `notifications/stream/route.ts:53` (вже не актуально через redesign)

**Стан:** pulito.trade — single-tenant. Доменні таблиці (`Order`, `Product`, `AuditLog`, `Review`, `Category`, `Feedback`) не мають `tenantId` тому, що всі рядки належать єдиному тенанту. Aудит фіксував це як «leak» для гіпотетичного multi-tenant, але:

- Зараз немає другого тенанта → немає й leak'у.
- Додавати `tenantId` колонки на 6 таблиць + бекфіл + оновлення усіх queries має сенс ЛИШЕ якщо є план підняти другий магазин у тій самій інстансі. Інакше — мертвий вантаж.

**Що робити, коли таки знадобиться multi-tenant:**

1. Прийняти рішення про дефолтний `tenantId` для існуючих даних.
2. Міграція: `ALTER TABLE … ADD COLUMN tenant_id INT NOT NULL DEFAULT 1`, `@@index([tenantId])`.
3. Оновити всі служби (`order.ts`, `product.ts`, ...) на скоупинг по `tenantId` з `resolveActiveTenantId`.
4. Прогнати усі тести.
5. Видалити `DEFAULT 1`.

Це не зробити в один прохід. Окремий проект.
