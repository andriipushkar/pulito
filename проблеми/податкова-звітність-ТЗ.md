# ТЗ: Модуль податкової звітності ФОП (генератор даних доходу)

**Статус:** в роботі (старт 2026-06-01). Будуємо MVP пунктами 1→3, XML — окремий етап.

## Мета

На основі продажів (оплат) у БД формувати **дохід ФОП за касовим методом** (дата надходження грошей) за звітні періоди, показувати превʼю й давати експорт. Дані вводяться вручну в Електронний кабінет / Дію (або в XML — коли буде XSD).

## Чому без власного XML/підпису зараз

- Крипто-підпис (КЕП) + відправка через API ДПС = дорого й ризиковано. Кабінет/Дія підтягують КЕП самі.
- **XSD форми отримати наразі не можемо** → валідний XML не гарантувати. Вигадані теги ДПС відхиляє. Тому MVP = **точні цифри доходу**, які вводяться вручну (для єдиного податку це кілька чисел).

## Фундамент у БД (вже є)

- `Payment.paidAt`, `Order.paidAt` — дата надходження грошей (касовий метод) ✅
- `Payment.amount`, `Payment.refundedAmount`, `Payment.paymentStatus` ('paid'), `Payment.paymentMethod`
- `Order.source` (`OrderSource`: web / marketplace…) — розрізняє сайт і маркетплейси ✅
- `marketplace-return.refundAmount` — повернення на маркетплейсі

## Чого нема (обмеження, чесно)

- **Виплати маркетплейсів** (нетто після комісії + дата виплати) — НЕ трекаються. Для маркетплейс-продажів можемо дати лише **валовий продаж** за період; фактичне надходження звіряти з банком/бухгалтером.
- Налаштувань ФОП (ІПН, група) — нема → додаємо.

## MVP — пункти

### Пункт 1 — Налаштування ФОП (мультигрупа)

- Зберігати в `SiteSetting` (БЕЗ міграції прод-БД): `fop_tin` (ІПН), `fop_name` (ПІБ), `fop_default_group` (1/2/3), `fop_rate_percent` (для 3-ї гр., напр. 5).
- Група **обирається при формуванні звіту** (як просив — «для різних груп, котрі обиратимемо»).

### Пункт 2 — Сервіс `src/services/tax-report.ts`

- `getTaxReport({ year, quarter?, group, ratePercent, fixedTax? })`:
  - Дохід = Σ `Payment` де `paymentStatus='paid'` та `paidAt` у періоді − `refundedAmount`.
  - Розбивка по `Order.source`: **web** (точно) / **marketplace** (валове, з прапорцем «оцінка»).
  - Прапорець COD (надходження ≠ дата доставки).
  - Періоди: за квартал І **наростаючим підсумком з початку року** (декларація вимагає YTD).
  - Орієнтовний податок: гр.3 → `income × rate%`; гр.1/2 → фікс-сума (вводиться).
  - `warnings[]`: маркетплейс-нетто не враховано, валюта НБУ для інвалюти тощо.

### Пункт 3 — Адмін-сторінка «Звітність» `/admin/tax-report`

- Вибір: рік, квартал (або весь рік), **група ФОП**, ставка/фікс.
- Превʼю-таблиця: дохід web / маркетплейс / повернення / чистий дохід / податок, + YTD.
- Кнопка «Завантажити CSV» (звірити й перенести в Кабінет/Дію).
- Пункт у меню адмінки. i18n: namespace `admin.taxReport` в uk/en/pl/ro.

### Пункт 4 (відкладено) — XML-генератор

- Коли буде XSD конкретної форми (напр. 3-тя гр. F0103xxx) → `xmlbuilder2` поверх готових цифр.

## ⚠️ Звірити з бухгалтером (не податкова порада)

- Правило визнання доходу маркетплейс-продажів (валове vs нетто, дата).
- Дата доходу для COD (день виплати НП).
- Перерахунок інвалюти за курсом НБУ на дату надходження.
