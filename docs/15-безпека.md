# 15. Безпека

Цей розділ описує модель безпеки Pulito: автентифікацію (JWT + refresh-ротація + 2FA + Google OAuth), авторизацію (ролі, impersonation), rate limiting, HTTP-заголовки безпеки (CSP з nonce), CSRF/XSS/SQL-injection захист, шифрування чутливих даних, idempotency, cron-lock, IP-обмеження адмінки, GDPR та maintenance mode. Усі описи звірені з кодом (`src/proxy.ts`, `src/services/auth.ts`, `token.ts`, `totp.ts`, `src/lib/encryption.ts`, `src/middleware/*`).

> Принцип: захист багатошаровий (defense-in-depth). Жоден окремий шар не є єдиною лінією оборони — Edge-проксі, обгортки маршрутів, сервісний шар і БД дублюють перевірки.

---

## 1. Автентифікація

### 1.1 JWT-токени

Підпис і верифікація — `src/services/token.ts`. Алгоритм керується `JWT_ALGORITHM` (`HS256` дефолт, або `RS256` з парою ключів `JWT_PRIVATE_KEY_PATH`/`JWT_PUBLIC_KEY_PATH`).

| Токен     | Тип         | TTL (env)               | Де живе                           |
| --------- | ----------- | ----------------------- | --------------------------------- |
| Access    | `access`    | `JWT_ACCESS_TTL` (15m)  | заголовок `Authorization: Bearer` |
| Refresh   | `refresh`   | `JWT_REFRESH_TTL` (30d) | httpOnly cookie `refresh_token`   |
| 2FA temp  | `2fa`       | 5 хв (зашито)           | повертається тілом при вході      |
| SSE-grant | `sse_grant` | 5 хв                    | httpOnly cookie `admin_sse_grant` |

- `verifyAccessToken`/`verifyRefreshToken`/`verify2faToken` перевіряють **і підпис, і поле `type`** — токен одного призначення не можна підсунути в інше.
- Refresh-токен має `jti` (UUID), тож швидкі переви́дачі дають різні хеші (колонка `token_hash` UNIQUE).
- `JWT_SECRET` валідовується на ентропію (16+ унікальних символів) у `src/config/env.ts`.

### 1.2 Cookies

`src/utils/cookies.ts`. Refresh-cookie `refresh_token`:

- `httpOnly: true` — недоступний з JS (захист від XSS-крадіжки);
- `secure` — у проді / під HTTPS;
- `sameSite: 'lax'` (або `'none'` для dev-tunnel-ів);
- `path: '/'`, `maxAge = JWT_REFRESH_TTL`.

`getAllRefreshTokensFromCookies` повертає **всі** cookie з цим іменем (браузер може надіслати кілька за різними path), і `/auth/refresh` пробує кожен — інакше можна було б застрягнути на застарілому токені після деплою.

### 1.3 Паролі

`src/services/auth.ts`: bcrypt (`SALT_ROUNDS = 10`). При вході для неіснуючого email виконується **dummy bcrypt-compare** проти попередньо обчисленого хешу (`DUMMY_BCRYPT_HASH`) — щоб час відповіді не видавав, чи існує email (захист від user-enumeration через тайминг).

### 1.4 Refresh-ротація + reuse detection

`refreshTokens()`:

1. Перевіряє підпис refresh-токена і знаходить рядок `RefreshToken` за хешем (SHA-256).
2. Якщо рядка немає → токен відкликано (`401`).
3. **Reuse detection:** якщо знайдений рядок уже має `revokedAt` — це ознака, що вкрадений токен використали після ротації. Тоді **відкликаються ВСІ токени родини користувача** (`updateMany revokedAt = null → now`), пишеться критичний аудит-запис (`refresh_token_reuse_detected`), повертається `401`.
4. Інакше — стара версія відкликається, видається нова пара (rotation).
5. Якщо користувача заблокували після входу (`isBlocked`) — refresh повертає `403`, обриваючи активні сесії.

Конкурентні refresh-и серіалізуються auth-refresh-локом (запобігає гонці подвійної ротації).

### 1.5 Logout / blacklist

`logoutUser()`: access-токен додається в Redis-blacklist (`bl:<hash>`) на час, що йому лишився; refresh-токен відкликається в БД. `withAuth` перевіряє blacklist (`isAccessTokenBlacklisted`). Якщо Redis недоступний — fail-open (токен ще валідується підписом; blacklist — додатковий шар).

### 1.6 Google OAuth

`loginWithGoogle()` (`/auth/google` + `/auth/google/callback`):

- Знаходить за `googleId`, інакше за `email`.
- **Account-takeover guard:** авто-прив'язка Google до існуючого локального акаунту лише якщо Google **підтвердив email** (`emailVerified`); інакше `403`. Без цього зловмисник із Google-акаунтом, що містить непідтверджену копію чужого email, міг би прив'язатись і ввійти як жертва.
- Race на паралельному створенні ловиться через unique-constraint (`P2002`) з повторним fetch.
- Google-вхід теж проходить 2FA, якщо ввімкнено.

### 1.7 Двофакторна автентифікація (TOTP)

`src/services/totp.ts` — власна реалізація RFC 6238, без зовнішніх залежностей:

- Секрет: 20 випадкових байтів, Base32 (RFC 4648). Період 30 с, 6 цифр, SHA-1, вікно ±1 крок (clock skew).
- Верифікація кодів — **`timingSafeEqual`** (захист від тайминг-атак).
- **Backup-коди:** 10 одноразових кодів, зберігаються як SHA-256-хеші; використаний код видаляється зі списку.
- **Шифрування секрету at-rest:** `encryptSecret`/`decryptStoredSecret` через AES-256-GCM (`encryption.ts`). Витік БД не дає сід для генерації валідних кодів. Легасі-плейнтекст-рядки толеруються і шифруються при наступному setup.

Потік входу з 2FA: `/auth/login` → `tempToken` (тип `2fa`, прив'язаний до SHA-256-хешу IP) → `/auth/2fa/verify-login` з кодом. Зміна IP між кроками → `401`.

## 2. Авторизація

### 2.1 Ролі

`UserRole` (`prisma/schema/user.prisma`): `client`, `wholesaler`, `manager`, `admin`. (У документації/коді інколи згадуються концептуальні рівні OWNER/ADMIN/MANAGER/CUSTOMER — у схемі їм відповідають значення вище.)

Для SaaS-тенантів — `TenantUserRole` (`prisma/schema/tenant.prisma`): `owner`, `admin`, `member`.

### 2.2 Обгортки доступу (`src/middleware/auth.ts`)

| Обгортка                | Поведінка                                                                                                                                              |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `withAuth`              | вимагає валідний access-токен; інакше `401`. Перевіряє blacklist. Кладе `user` у контекст                                                              |
| `withOptionalAuth`      | токен необов'язковий; `user` або об'єкт, або `null`                                                                                                    |
| `withRole(...roles)`    | `withAuth` + перевірка ролі; інакше `403`                                                                                                              |
| `withRole2fa(...roles)` | `withRole` + перевірка, що у користувача **увімкнено 2FA** (1 запит до БД). Для чутливих адмін-маршрутів (платежі, користувачі, білінг, impersonation) |
| `withApiKey(perms)`     | автентифікація за API-ключем для інтеграцій                                                                                                            |

`withRole2fa` закриває діру, де вкрадений адмін-токен обходив би «layout-only» перевірку 2FA в UI.

### 2.3 Impersonation (admin → user)

`/admin/users/{id}/impersonate` (`src/services/impersonation.ts`): захищено `withRole2fa('admin')` + rate-limit `sensitive` (3/15хв на адміна) — щоб вкрадена сесія не енумерувала ID користувачів через side-channel повідомлень помилок. Кожна успішна імперсонація лишає аудит-запис.

### 2.4 API-ключі (`src/middleware/api-key-auth.ts`)

Для зовнішніх інтеграцій (1С/BAS). Ключ формату `csk_<hex>`, передається як `Authorization: Bearer` або `X-API-Key`. Зберігається SHA-256-хешем у таблиці `ApiKey`. Перевіряється: активність (`isActive`), термін (`expiresAt`), гранульовані `permissions`, per-key rate-limit (`integration1c` 100/хв). `lastUsedAt` оновлюється fire-and-forget. Сирий ключ показується лише раз при створенні.

## 3. Rate limiting

Три рівні (`src/services/rate-limit.ts`, `src/proxy.ts`, `src/lib/action-rate-limit.ts`):

1. **Edge / proxy.ts** — in-memory per-instance ліміт per-route для всіх `/api/*` (`checkApiRateLimit` + `getRouteLimit`). Edge Runtime не може в ioredis, тому in-memory. Для мульти-інстансу первинний захист — Cloudflare / nginx `limit_req`.
2. **Redis sliding-window** (`checkRateLimit`) у самих хендлерах — кластеро-безпечний. Атомарний пайплайн (`ZREMRANGEBYSCORE` + `ZADD` + `ZCARD` + `EXPIRE`). Десятки пресетів `RATE_LIMITS` (`auth` 10/хв, `sensitive` 3/15хв, `orders` 10/хв, `couponValidate` 20/5хв, `adminAiGenerate` 60/год, `adminPaymentTest` 5/хв тощо).
3. **Server Actions** (`action-rate-limit.ts`) — оскільки Server Actions обходять middleware: `checkout` 5/хв, `cart` 30/хв, `review` 5/15хв.

Окремо: **логін** — `checkLoginRateLimit` рахує 5 спроб / 15 хв на ключ `IP:email`; невдала спроба → `recordFailedLogin`, успіх → `clearLoginAttempts`.

При перевищенні — `429` + `Retry-After` + `X-RateLimit-Limit`/`X-RateLimit-Remaining`. **Fail-open:** якщо Redis недоступний, запит дозволяється (доступність важливіша; JWT досі захищає). Вимикач `DISABLE_RATE_LIMIT=1` (тільки для тестів).

## 4. HTTP-заголовки безпеки

Встановлюються в `src/proxy.ts` на кожній відповіді:

| Заголовок                           | Значення                                       |
| ----------------------------------- | ---------------------------------------------- |
| `X-Content-Type-Options`            | `nosniff`                                      |
| `X-Frame-Options`                   | `DENY`                                         |
| `Referrer-Policy`                   | `strict-origin-when-cross-origin`              |
| `Permissions-Policy`                | `camera=(), microphone=(), geolocation=()`     |
| `X-XSS-Protection`                  | `1; mode=block`                                |
| `X-Permitted-Cross-Domain-Policies` | `none`                                         |
| `Strict-Transport-Security` (HSTS)  | `max-age=63072000; includeSubDomains; preload` |
| `X-DNS-Prefetch-Control`            | `on`                                           |
| `X-Download-Options`                | `noopen`                                       |
| `X-Request-Id`                      | `crypto.randomUUID()` (кореляція в логах)      |
| `X-Nonce`                           | per-request nonce для CSP                      |
| `Content-Security-Policy`           | див. нижче                                     |

> **COEP / COOP свідомо НЕ виставляються.** Сторонні iframe (OpenStreetMap, Google Maps, 3-D Secure платіжок) не надсилають CORP-заголовків — будь-яке значення COEP/COOP змушує Firefox відмовляти у завантаженні iframe. Вмикати лише після видалення/проксювання всіх сторонніх вбудувань.

### 4.1 CSP з nonce

Per-request nonce (`crypto.randomUUID()`) — основний захист від XSS (інлайн-скрипт-ін'єкція блокується). Ключові директиви:

```
default-src 'self'
script-src 'self' 'unsafe-eval' 'nonce-<nonce>' https://www.googletagmanager.com
           https://www.google-analytics.com https://connect.facebook.net
style-src-elem 'self' 'unsafe-inline' https://fonts.googleapis.com
style-src-attr 'unsafe-inline'
img-src 'self' data: blob: https:
font-src 'self' https://fonts.gstatic.com
connect-src 'self' https://www.google-analytics.com https://api.telegram.org https://api.novaposhta.ua
frame-src https://www.google.com https://maps.google.com https://www.openstreetmap.org
frame-ancestors 'none'
base-uri 'self'
form-action 'self'
object-src 'none'
upgrade-insecure-requests
```

> **`script-src` потребує `'unsafe-eval'` — НЕ прибирати.** Бандлені клієнтські залежності використовують `new Function("return this")()` (globalThis-shim) та `@firebase/util`-проб (`eval("__FIREBASE_DEFAULTS__")`). CSP, що блокує eval, кидав помилку на module-init і ламав кабінет (`/account`). Nonce лишається первинним XSS-захистом; `unsafe-eval` дозволяє лише `eval`/`Function` — значно вужчий вектор, ніж `unsafe-inline`.
>
> `style-src` використовує `'unsafe-inline'` (а не nonce): сторонні бібліотеки інжектять `<style>` без нашого nonce; CSP3 ігнорує `unsafe-inline` за наявності nonce, тож nonce тут опущено свідомо (інжекція стилів не виконує код).

За наявності `SENTRY_CSP_REPORT_URI` додаються `report-uri` + `report-to` + заголовок `Report-To`.

## 5. CSRF

`checkCsrf` у `src/proxy.ts` для мутуючих API-запитів (`POST/PUT/PATCH/DELETE` під `/api`):

1. **Origin/Referer** мають збігатися з довіреним хостом (`getTrustedHosts`: ефективний host + host з `APP_URL` + localhost). Невідповідність → `403`.
2. Обов'язковий заголовок **`X-Requested-With`** (блокує прості HTML-form-сабміти). Відсутній → `403`.

**Виключення** (`CSRF_EXEMPT_PREFIXES`): `/api/webhooks/*` (зовнішні POST), `/api/v1/cron/*` (Bearer-auth), `/api/v1/metrics` та `/api/v1/events` (sendBeacon не вміє ставити заголовки; пишуть лише валідовану аналітику), `/api/v1/log-client-error` (діагностичний маяк, пише лише обрізані рядки в лог).

## 6. XSS та SQL-injection

- **XSS:** користувацький HTML (відгуки з зображеннями, контент) санітизується через `isomorphic-dompurify` (DOMPurify). CSP-nonce — другий шар. httpOnly-cookie не дає вкрасти сесію навіть при XSS.
- **SQL-injection:** уся робота з БД — через Prisma (параметризовані запити). Сирий SQL (`$queryRaw`) використовується точково для повнотекстового пошуку/аналітики з параметрами, не конкатенацією.
- **Ліміти розміру тіла:** вебхуки обрізають за `Content-Length` (напр. 64 КБ для платіжок → `413`); інпути обмежені Zod (`max(...)`), напр. `sessionId` cookie-consent ≤128 символів.

## 7. Шифрування чутливих даних

`src/lib/encryption.ts` — **AES-256-GCM**:

- Ключ: `scryptSync(APP_SECRET, 'salt', 32)`.
- Формат шифротексту: `ivHex:tagHex:encryptedHex` (16-байтовий IV, GCM auth-tag).
- `isEncrypted()` розпізнає формат (для толерування легасі-плейнтексту).

Шифруються at-rest:

- **2FA-секрети** (`encryptSecret`/`decryptStoredSecret`);
- **платіжні ключі** (LiqPay/Mono/WayForPay) — лише в `/admin/payment-settings`, без env-фолбеку;
- **бот-токени** (Telegram), SMTP-паролі, ключі каналів/інтеграцій.

> `APP_SECRET` — критичний секрет: його втрата/зміна робить нечитабельними всі зашифровані поля. Зберігати поза репозиторієм.

Хешування (одностороннє, SHA-256): refresh-токени, access-blacklist, API-ключі, backup-коди, idempotency-ключі.

## 8. Idempotency

`src/services/idempotency.ts` — для `POST /api/v1/orders` (заголовок `X-Idempotency-Key`):

- `reserveIdempotencyKey` атомарно (`SET NX`) ставить in-flight маркер (TTL 60 с);
- дубль під час обробки → `409`;
- дубль після завершення → закешована відповідь (TTL 24 год), статус `201`;
- закриває гонку подвійного кліку, коли два запити обидва бачать «немає кешу» і обидва створюють замовлення.

---

## 9. Cron-lock (fail-closed)

`src/lib/cron-lock.ts` — `withCronLock(name, ttl, fn)` для cron-задач, що не повинні перетинатися (синки маркетплейсів/цін/залишків). Лок — Redis `SET NX EX`; авто-звільнення по TTL при краху процесу; реліз лише якщо токен ще наш.

> **Fail-closed:** якщо Redis недоступний — запуск **пропускається** (повертає `acquired: false`), а не виконується без локу. Краще пропустити тік (наступний ретрайне), ніж ризикнути подвійним виконанням неідемпотентної задачі.

Самі cron-маршрути захищені `CRON_SECRET` (фолбек `APP_SECRET`) з порівнянням через `timingSafeCompare` (`src/utils/timing-safe.ts`, постійний час).

## 10. IP-обмеження доступу до адмінки

`src/proxy.ts`: якщо задано `admin_allowed_ips` (у `/admin → System`) або env `ADMIN_ALLOWED_IPS`, доступ до `/admin` **і** `/api/v1/admin` обмежується списком. Локальні IP (`127.0.0.1`, `::1`, `localhost`) дозволені завжди (cron + on-box дебаг). Поза списком: UI → HTML-сторінка «Доступ заборонено» з IP відвідувача; API → JSON `403`. Налаштування читається з БД (через `getSettings`, кеш ≤60с), тож власник міняє без редеплою.

---

## 11. Cookie consent / GDPR

- `POST /api/v1/cookie-consent` записує згоду (`analyticsAccepted`/`marketingAccepted`) у `CookieConsent` за `sessionId` (≤128 символів, rate-limited).
- `GET /api/v1/me/gdpr-export` та `/me/export` — експорт персональних даних користувача.
- Видалення/блокування акаунту через `/me/account` та адмін-інструменти.
- Аналітика/маркетинг-скрипти не вантажаться без згоди.

---

## 12. Audit log

`src/services/audit.ts` (`logAudit`) — durable-журнал чутливих подій у БД (на відміну від rate-limit-лічильників у Redis, що спливають):

- невдалі/успішні логіни (зокрема `2fa_required`, `blocked`);
- reuse-detection refresh-токенів (критично);
- impersonation;
- зміни у платежах/користувачах/налаштуваннях.

Перегляд та експорт — `/admin/audit-log`, `/admin/audit-log/export`. Очищення — cron `cleanup-audit-log`.

---

## 13. Maintenance mode

`src/proxy.ts`: прапор `maintenance_mode` (у `/admin → Maintenance`, фолбек env `MAINTENANCE_MODE=true`). Коли ввімкнено, усі запити редіректяться на `/maintenance`, **окрім** дозволених префіксів: `/maintenance`, `/admin`, `/api/v1/admin`, `/api/v1/auth` — щоб власник міг увійти й вимкнути режим. Поширення обмежене TTL кешу налаштувань (≤60 с).
