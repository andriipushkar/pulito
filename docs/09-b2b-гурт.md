# 09. B2B / гуртова торгівля

Pulito працює одночасно як роздрібний магазин і як **B2B-платформа** для гуртових клієнтів. Цей розділ описує гуртову модель: три рівні гуртових цін, персональні ціни, рушій знижок за обсягом, правила замовлення, статус/підтвердження гуртового клієнта, bulk-замовлення, комерційні пропозиції, кредитний ліміт, палетну доставку та прайс-листи.

Основні сервіси:

```
src/services/b2b.ts
src/lib/wholesale-price.ts
src/services/volume-pricing.ts
src/services/personal-price.ts
src/services/pricelist.ts
src/services/commercial-proposal.ts
src/services/pallet-delivery.ts
src/services/pallet.ts
```

Компоненти: `src/components/wholesale/`. Кабінетні сторінки: `account/wholesale-request/`, `account/wholesale/bulk-order/`, `account/status/`, `account/pricelist/` (див. [08](08-особистий-кабінет.md)).

---

## Ключові принципи (важливо)

1. **Гуртова ціна — це стеля (ceiling), а не похідна величина.** Поля `priceWholesale*` товару **вводяться продавцем** (seller-submitted), не обчислюються з собівартості чи роздробу. На checkout гуртова ціна виступає верхньою межею для гуртового клієнта — навіть якщо роздрібна знижка дала б більше, B2B-клієнта не «перевищать» (фікс у checkout, щоб volume-off-retail не завищував ціну гурту).
2. **3 рівні дискретні:** 1 = дрібний опт, 2 = середній опт, 3 = великий опт. Без проміжних градацій.
3. **Запобігання перетину діапазонів:** `VolumeDiscount` і `WholesaleRule` перевіряють перетин дат/діапазонів ще на створенні.
4. **Скоупінг менеджера:** операції з `PersonalPrice` перевіряють `assignedManagerId` — менеджер не чіпає чужих клієнтів.
5. **Атомарне підтвердження:** апрув заявки робиться через `updateMany { where: { status: 'pending' } }` (race-safe, 409 при подвійному).

## Три рівні гуртових цін

Поля на моделі `Product` (`prisma/schema/product.prisma`):

| Поле              | Тип              | Рівень                  |
| ----------------- | ---------------- | ----------------------- |
| `priceRetail`     | `Decimal(10,2)`  | роздрібна ціна          |
| `priceWholesale`  | `Decimal(10,2)?` | рівень 1 — дрібний опт  |
| `priceWholesale2` | `Decimal(10,2)?` | рівень 2 — середній опт |
| `priceWholesale3` | `Decimal(10,2)?` | рівень 3 — великий опт  |

### Резолвер ціни

`src/lib/wholesale-price.ts → resolveWholesalePrice(product, wholesaleGroup)`:

- `wholesaleGroup === 1` → `priceWholesale`
- `wholesaleGroup === 2` → `priceWholesale2`
- `wholesaleGroup === 3` → `priceWholesale3`
- повертає `null`, якщо значення відсутнє або ≤ 0 (нуль трактується як «не задано», а не «безкоштовно»).

При генерації прайсу/замовлення, якщо тарифна ціна `null`, відбувається відкат до наступного рівня, далі до `priceRetail`.

## Персональні ціни (per-client)

Сервіс: `src/services/personal-price.ts`. Модель: `PersonalPrice` (`prisma/schema/b2b.prisma`).

| Поле                              | Опис                                               |
| --------------------------------- | -------------------------------------------------- |
| `userId`                          | клієнт, до якого прив'язана ціна                   |
| `productId?`                      | прив'язка до конкретного товару (XOR з категорією) |
| `categoryId?`                     | прив'язка до категорії (скаляр, без relation)      |
| `discountPercent?` `Decimal(5,2)` | знижка у % (XOR з `fixedPrice`)                    |
| `fixedPrice?` `Decimal(10,2)`     | фіксована ціна                                     |
| `validFrom`, `validUntil?`        | вікно дії                                          |
| `stackableWith[]`                 | з якими типами знижок можна комбінувати            |
| `createdBy`                       | автор правила                                      |

Функції:

- `getEffectivePrice(userId, productId, categoryId)` — резолвить активне правило: спочатку по товару, потім по категорії; враховує вікно дат; повертає `{ discountPercent, fixedPrice, stackableWith }` або `null`.
- `getPersonalPrices(filters, managerScopeId?)` — список з пагінацією; менеджер бачить лише своїх клієнтів.
- `createPersonalPrice(...)` — ідемпотентне (блокує дублікат `(userId, productId, categoryId)` → 409); вимагає `productId` XOR `categoryId` і `discountPercent` XOR `fixedPrice`.
- `updatePersonalPrice` / `deletePersonalPrice` — зі скоупом менеджера.

Адмін-UI: `/admin/personal-prices/`. Валідатори: `src/validators/personal-price.ts`.

## Рушій знижок за обсягом (Volume pricing)

Сервіс: `src/services/volume-pricing.ts`. Модель: `VolumeDiscount` (`prisma/schema/volume-discount.prisma`).

| Поле                             | Опис                                             |
| -------------------------------- | ------------------------------------------------ |
| `productId?` / `categoryId?`     | область дії (товар має пріоритет над категорією) |
| `minQuantity`, `maxQuantity?`    | діапазон кількості                               |
| `discountPercent` `Decimal(5,2)` | розмір знижки (деф. 0)                           |
| `discountType`                   | `percentage` або `fixed_amount`                  |
| `isActive`, `priority`           | активність і пріоритет (tiebreak)                |
| `startsAt`, `endsAt?`            | планування у часі                                |
| `stackableWith[]`                | сумісність з іншими знижками                     |

Логіка:

- `getVolumeDiscount(productId, categoryId, quantity)` — обирає найкраще правило: товар > категорія; вищий `priority` перемагає, при рівних — більший `discountPercent`; враховує `isActive`, діапазон кількості та дат. Повертає `null`, якщо нема збігу.
- `applyVolumeDiscounts(cartItems[])` — застосовує знижки до кошика. `percentage`: `price × (1 − percent/100)`; `fixed_amount`: `price − amount` (не нижче 0).
- `getVolumeDiscountsForProduct(...)` — для показу на картці товару (бейдж `VolumeDiscountBadge.tsx`).
- Адмін-CRUD: `getVolumeDiscounts`, `createVolumeDiscount` (відхиляє перетин діапазонів у тій самій області), `createVolumeDiscountsBulk` (рядки незалежні), `updateVolumeDiscount`, `deleteVolumeDiscount`.

Адмін-UI: `/admin/volume-discounts/`. Валідатори: `src/validators/volume-discount.ts`.

## Гуртові правила замовлення (WholesaleRule)

Модель: `WholesaleRule` (`prisma/schema/b2b.prisma`). Перевіряються на checkout (всі активні мають пройти, логіка AND).

| `ruleType`         | Зміст                                               |
| ------------------ | --------------------------------------------------- |
| `min_order_amount` | мінімальна сума кошика (напр. 5000 грн)             |
| `min_quantity`     | мінімальна кількість одиниць                        |
| `multiplicity`     | кількість має бути кратною N (напр. упаковки по 10) |

Поля: `productId?` (null = на всі товари), `value` `Decimal(10,2)`, `isActive`, `validFrom`/`validUntil?`. Два правила з однаковими `(ruleType, productId)` не можуть мати перетин дат. Клієнт бачить активні обмеження у вкладці `account/status/`. Адмін-UI: `/admin/wholesale-rules/`.

---

## Статус гуртового клієнта та підтвердження

Поля на `User` (`prisma/schema/user.prisma`): `wholesaleStatus` (`none`/`pending`/`approved`/`rejected`), `wholesaleGroup` (1/2/3 або null), `wholesaleRequestDate`, `wholesaleApprovedDate`, `wholesaleMonthlyVol`, `creditLimit`, `creditUsed`, `assignedManagerId`, `role`.

### Потік (3 етапи)

1. **Подача заявки** — `account/wholesale-request/page.tsx`, 3-кроковий майстер:
   - Крок 1: компанія/ФОП, ЄДРПОУ/ІПН, форма власності (ФОП/ТОВ/ПП/інше), система оподаткування, юр. адреса.
   - Крок 2: контактна особа, телефон (мін. 10 цифр; передзаповнення з профілю).
   - Крок 3: очікуваний місячний обсяг, коментар.
   - `POST /api/v1/me/wholesale-request` → `wholesaleStatus = 'pending'`, нотифікація менеджерів у Telegram (non-blocking), rate-limit 3/15 хв, блок повторної подачі якщо вже pending/wholesaler.
2. **Підтвердження менеджером/адміном** — `PUT /api/v1/admin/users/[id]/wholesale`, body `{ action: 'approve'|'reject', wholesaleGroup? }`:
   - approve (атомарно лише якщо `status === 'pending'`): `role='wholesaler'`, `status='approved'`, `wholesaleGroup`, дата апруву, опційно `assignedManagerId`; 409 при повторі.
   - reject: `status='rejected'` (можна подати повторно), аудит `wholesale_reject`.
3. **Поточний статус** — `GET /api/v1/me/wholesale-request`: pending-картка з даними / форма для повторної подачі (rejected) / success (вже гурт). Статистика: `GET /api/v1/me/wholesale-stats` (кількість замовлень, lifetime-сума, дата апруву).

Валідатор: `src/validators/wholesale-request.ts`.

## Bulk / швидке замовлення (CSV)

Сервіс: `src/services/b2b.ts → resolveBulkOrder(items, wholesaleGroup)`. Компонент: `src/components/wholesale/BulkOrderForm.tsx`. Сторінка: `account/wholesale/bulk-order/page.tsx`.

- Введення `код, кількість` (роздільники: кома/крапка-з-комою/таб), 1–500 позицій.
- `resolveBulkOrder` шукає товари за кодом (без регістру, trim), перевіряє склад (warnings при нестачі), застосовує тарифну ціну за `wholesaleGroup`. Повертає `{ items, totalAmount, errors }`.
- Ендпоінт `POST /api/v1/wholesale/bulk-order` (схема `{ code, quantity }`, rate-limit `RATE_LIMITS.wholesale`).
- Результат: таблиця з кодом, назвою, кількістю, ціною, сумою, попередженнями про наявність.

Також у кабінеті є простіший `account/quick-order/` з підтримкою Excel/CSV (див. [08](08-особистий-кабінет.md)).

## Комерційна пропозиція (PDF)

Сервіс: `src/services/commercial-proposal.ts → generateCommercialProposal(options)`. Ендпоінт: `POST /api/v1/wholesale/commercial-proposal`.

- Вхід: `{ items: [{ code, quantity }], comment?, validDays? }`; ціни резолвляться через `resolveBulkOrder`.
- PDF (брендований, multi-page з header/footer): колонки №, Код, Назва, К-сть, Ціна, Сума; дата дійсності (`validDays`, деф. 14), коментар.
- Зберігається у `UPLOAD_DIR/proposals/proposal_{timestamp}.pdf`, повертає публічний URL.
- Rate-limit `RATE_LIMITS.adminPdfExport` (50/добу) проти вичерпання диска.

## Кредитний ліміт (відтермінована оплата)

Сервіс: `src/services/b2b.ts`.

- `checkCreditLimit(userId, amount)` — `true`, якщо `(creditLimit − creditUsed) >= amount`.
- `useCreditForOrder(userId, amount)` — атомарно інкрементує `creditUsed` після перевірки; кидає `B2BError` при нестачі.

Поля `creditLimit` / `creditUsed` (`Decimal(10,2)`) на `User`.

---

## Палетна доставка

Сервіси: `src/services/pallet-delivery.ts`, `src/services/pallet.ts`. Форма на checkout: `src/components/checkout/PalletDeliveryForm.tsx` (метод доставки `pallet`).

- `calculatePalletDeliveryCost(weightKg, region?)` — базова ціна + ціна за кг × вага × регіональний множник; безкоштовно понад поріг (деф. 50 000 грн); межі ваги з конфігу (деф. 100–5000 кг). Повертає `{ cost, estimatedDays, isFreeDelivery }`.
- `getPalletConfig()` / `updatePalletConfig(partial)` — конфіг у `SiteSettings`.
- `Pallet` — агрегує замовлення для регіонального відвантаження; статуси `forming`/`in_transit`/`delivered`/`cancelled`, пагінація вкладених замовлень (до 100).

## Прайс-листи (PDF)

Сервіс: `src/services/pricelist.ts → generatePricelist(type, wholesaleGroup?)`. Кабінет: `account/pricelist/`.

- `type = 'retail'` — роздрібний; `type = 'wholesale'` — тарифна ціна через `resolveWholesalePrice`, фолбек `priceWholesale → priceRetail`.
- Брендований A4: фото (80×80), код, назва, ціна, бейдж наявності; секції за категоріями з лічильником; шапка з реквізитами і футер. `PricelistError`, якщо немає активних товарів.
- Завантаження: `GET /api/v1/pricelist?type=...`, метадані: `GET /api/v1/pricelist/meta`.

---

## Доступ за роллю

Ендпоінти bulk-order, гуртових цін і комерційних пропозицій перевіряють `role in ['wholesaler', 'admin', 'manager']`. У вітрині/кошику ціни автоматично враховують гуртову групу залогіненого користувача.

---

## Стекування знижок

Кожна знижка має `stackableWith[]`. На checkout враховується перетин: усі застосовані знижки повинні взаємно дозволяти одна одну. Порядок застосування: бандл-знижка → volume/персональна → купон (проти суми після бандлів) → бонуси лояльності, з гуртовою ціною як стелею.
