# 19. Деплой та експлуатація

Цей розділ описує, як **Pulito** деплоїться на production і як його експлуатувати: atomic-деплой через `./deploy.sh`, PM2, Nginx, обмеження збірки (heap/OOM), cron-задачі, бекапи, моніторинг і Docker-деплой.

Платформа працює на одному VPS під користувачем `pulitotrade`. Застосунок — Next.js 16, запущений через PM2 (`next start`), за reverse-proxy Nginx, з PostgreSQL/PgBouncer/Redis/Typesense у Docker. Live-сайт `pulito.trade` стоїть за Cloudflare.

---

## 1. Production-деплой через `./deploy.sh`

> ⚠️ **Завжди деплойте через `./deploy.sh`, а не `npm run build` напряму.** In-place `next build` перезаписує `.next/` поки live-`next start` ще роздає сторінки — на півдорозі старий сервер читає недописаний `prerender-manifest.json` / `client-reference-manifest` і кидає `InvariantError` + **HTTP 500** живим відвідувачам.

`deploy.sh` робить **atomic zero-downtime деплой**. Логіка скрипта:

1. **Очищає** стейл-стейджинг (`rm -rf .next-staging`).
2. **Gate 0 — typecheck:** `npx tsc --noEmit` (включно з тестами; `next build` перевіряє лише app-код). Падіння → деплой зупинено.
3. **Генерує slug-redirects:** `node scripts/gen-slug-redirects.cjs` — знімок таблиці `slugRedirect` у статичну мапу `src/generated/slug-redirects.ts`, яку імпортує middleware (`proxy.ts`). Не фатально — при помилці лишається старий файл.
4. **Збирає в `.next-staging`** (не в live `.next`):
   ```
   NEXT_DIST_DIR=.next-staging BUILD_ID=$(date +%s) \
   NODE_OPTIONS='--max-old-space-size=3072' npx next build --webpack
   ```
   `BUILD_ID` (timestamp) скоупить service-worker кеші під цей деплой → браузери відвідувачів перевстановлюють воркер. `tsconfig.json` тимчасово знімається й відновлюється (next build його переписує).
5. **Gate — успіх збірки:** у логу має бути `Compiled successfully` або `Compiled with warnings` (бенінні webpack-варнінги не блокують), **не** має бути `Failed to compile`/`Failed to type check`, і має існувати `.next-staging/BUILD_ID`.
6. **Atomic swap:** два миттєвих `mv` на одній ФС — `.next → .next-prev`, `.next-staging → .next`. Старий сервер тримає відкриті дескриптори `.next` валідними через rename (Linux), тому in-flight запити не бачать зниклих файлів.
7. **Reload PM2:** `pm2 reload pulito --update-env` (фолбек `pm2 restart`).
8. **Health-gate:** до 12 спроб × 3 с `curl localhost:3000/api/v1/health`. Якщо жодна не дала `200` → **авто-роллбек** (`.next-prev → .next`, reload PM2), вихід 1.
9. **Admin smoke (non-blocking):** створює тимчасового адміна, прогоняє `tests/e2e/admin-smoke.spec.ts` у реальному браузері — ловить client-runtime краші, яких health-check не бачить. Сайт уже live, тому падіння **не** роллбекає авто; друкує биті сторінки й виходить 3. Пропуск: `SKIP_SMOKE=1`.

**Запуск:**

```bash
cd /home/pulitotrade/pulito
./deploy.sh
# екстрений хотфікс без smoke:
SKIP_SMOKE=1 ./deploy.sh
```

Логи деплою: `/tmp/pulito-deploy.log`, typecheck: `/tmp/pulito-typecheck.log`, smoke: `/tmp/pulito-smoke.log`. Попередня збірка лишається в `.next-prev` (видалити, коли впевнені).

## 2. Обмеження збірки (heap / OOM)

> ⚠️ **Ніколи не запускайте дві збірки одночасно** — це викликає OOM, який убиває live-сайт.

- `build` обмежено heap'ом **3072 МБ** (`NODE_OPTIONS='--max-old-space-size=3072'`).
- Сервер має **swap 8 ГБ** як страховку.
- `next build --webpack` працює в 2 CPU (узгоджено з ресурсами VPS).
- PM2-процес застосунку має окремий, набагато менший ліміт heap (512 МБ) — це **runtime**, не білд (див. нижче).

## 3. PM2

Конфіг — `ecosystem.config.js`:

```js
{
  name: 'pulito',
  cwd: '/home/pulitotrade/pulito',
  script: 'node_modules/next/dist/bin/next',
  args: 'start',
  exec_mode: 'fork',
  instances: 1,
  node_args: '--max-old-space-size=512',   // clean OOM замість swap-thrashing
  max_memory_restart: '600M',
  env: { NODE_ENV: 'production' },
  out_file: '/home/pulitotrade/.pm2/logs/pulito-out.log',
  error_file: '/home/pulitotrade/.pm2/logs/pulito-error.log',
  merge_logs: true,
  time: true,
}
```

**Команди:**

```bash
pm2 startOrReload ecosystem.config.js   # застосувати конфіг
pm2 restart pulito                       # після зміни server-only ENV
pm2 reload pulito --update-env           # zero-downtime (deploy.sh)
pm2 status                               # стан процесів
pm2 logs pulito                          # логи
pm2 monit                                # реалтайм-моніторинг
pm2 save && pm2 startup                  # автозапуск після ребуту
```

**Ротація логів PM2** (bare-metal):

```bash
pm2 install pm2-logrotate
pm2 set pm2-logrotate:max_size 50M
pm2 set pm2-logrotate:retain 7
pm2 set pm2-logrotate:compress true
pm2 set pm2-logrotate:rotateInterval '0 0 * * *'
```

## 4. Nginx (reverse proxy)

Приклад — `nginx.conf.example`. Ключове:

- HTTP→HTTPS редирект (301), TLS через Let's Encrypt.
- **Rate-limit зони:** `auth` (10 req/min), `webhooks` (30 req/min), `api` (120 req/min) з burst'ами.
- Маршрути: `/api/v1/auth/`, `/api/webhooks/`, `/api/`, `/` → `proxy_pass http://localhost:3000`.
- Статика `/_next/static/` кешується на 365 днів (`immutable`).
- **`/uploads/`** — захист від виконання скриптів: `X-Content-Type-Options nosniff`, жорсткий CSP (`default-src 'none'`), і **403 на небезпечні розширення** (`php|asp|jsp|sh|py|exe|html|svg|js` тощо).
- `client_max_body_size 10m`.

**SSL (certbot):**

```bash
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d pulito.trade
```

**Firewall:**

```bash
sudo ufw allow 22/tcp && sudo ufw allow 80/tcp && sudo ufw allow 443/tcp && sudo ufw enable
```

## 5. Cron-задачі

Усі cron-маршрути живуть у `src/app/api/v1/cron/` (понад 50 routes) і вимагають заголовка `Authorization: Bearer <CRON_SECRET|APP_SECRET>`.

### Генеричний раннер — `/home/pulitotrade/cron-run.sh`

```bash
cron-run.sh <route-name> [query-string]
# напр.:  cron-run.sh cleanup-tokens
#         cron-run.sh safety-stock-alert "threshold=5"
```

Що робить:

- **POST на `localhost:3000/api/v1/cron/<route>`** (через localhost, щоб Cloudflare не 403-нув server-to-server).
- Читає Bearer-токен із `.env`: спершу `CRON_SECRET`, інакше `APP_SECRET` — секрет ніколи не лежить у crontab.
- При падінні шле **Telegram-алерт менеджеру** (`TELEGRAM_BOT_TOKEN` + `TELEGRAM_MANAGER_CHAT_ID`), щоб битий cron не падав тихо.

### Спеціалізовані обгортки

| Скрипт                 | Маршрут                  | Розклад                                                |
| ---------------------- | ------------------------ | ------------------------------------------------------ |
| `cron-cleanup-nfl.sh`  | `cleanup-not-found-logs` | щоденно (чистка 404-логів > 90 днів)                   |
| `cron-safety-stock.sh` | `safety-stock-alert`     | щогодини (Telegram-алерт менеджеру про низькі залишки) |
| `backup.sh`            | (не route — `pg_dump`)   | щоденно 03:00                                          |

### Поточний crontab (вибірка)

```cron
0 3 * * * /home/pulitotrade/backup.sh
0 4 * * * /home/pulitotrade/cron-cleanup-nfl.sh   >> .../cron-cleanup-nfl.log 2>&1
15 * * * * /home/pulitotrade/cron-safety-stock.sh >> .../cron-safety-stock.log 2>&1

# через cron-run.sh (приклади):
0 2 * * *     cron-run.sh cleanup-tokens
20 1 * * *    cron-run.sh cleanup-carts
10 2 * * *    cron-run.sh payment-reconciliation
30 6 * * *    cron-run.sh precompute-analytics
*/30 * * * *  cron-run.sh process-subscriptions      # без авто-списання
30 9 * * *    cron-run.sh subscription-reminders      # нагадування
45 * * * *    cron-run.sh health-check
0 * * * *     cron-run.sh promo-autopost
5,15,25,35,45,55 * * * * cron-run.sh publish-scheduled
```

Усі рядки логуються в `/home/pulitotrade/cron-run.log`. Маркетплейс-cron'и (`sync-marketplace-*`, `marketplace-token-refresh`) — no-op доки Rozetka/Prom/OLX не налаштовані.

> ~59 cron-маршрутів існує; реально заплановано ~30+ безпечних/потрібних. Heavy-задачі винесені на нічне вікно 01:00–05:00. Дати/бакети рахуються за Київським часом (`kyivMidnightUtc` тощо).

## 6. Бекапи

### `backup.sh` (щоденно 03:00)

- `pg_dump` БД через `docker compose exec -T postgres` → gzip у `/home/pulitotrade/backups/pulito_<timestamp>.sql.gz`.
- **Архівує `uploads/`** (фото/документи живуть лише на диску) → `uploads_<timestamp>.tar.gz`.
- **Offsite (опційно, no-op доки не налаштовано):**
  - S3 — `scripts/backup-s3-upload.mjs` (активний, коли заданий `BACKUP_S3_BUCKET`);
  - Telegram (GPG-encrypted) — `scripts/backup-telegram-upload.mjs` (активний при `BACKUP_PASSPHRASE`; вимикається, якщо налаштовано S3).
- **Retention:** видаляє бекапи старші за **14 днів**. Лог: `backups/backup.log`.

> ⚠️ **Offsite-бекапи досі тільки локальні** — Telegram-скрипт готовий, але активація чекає рішення власника (S3 чи Telegram).

### `backup-verify.sh` (щотижня, напр. `30 5 * * 1`)

Перевіряє, що бекап реально відновлюється:

- Відновлює останній дамп у тимчасову БД `pulito_verify_<ts>`.
- Перевіряє, що `products`/`users` непорожні, таблиця `orders` існує.
- Дропає тестову БД. Пише `backups/verify.log` + `backups/verify-status.json` (читається `/admin/health`).

**Ручний бекап / відновлення:**

```bash
docker compose exec -T postgres pg_dump -U pulito_user pulito_trade | gzip > backup.sql.gz
gunzip -c backup.sql.gz | docker compose exec -T postgres psql -U pulito_user -d pulito_trade
```

> Redis **не бекапиться** — дані ефемерні (кеш/сесії), відновлюються самі.

## 7. Моніторинг

| Інструмент          | Призначення                                                                                   |
| ------------------- | --------------------------------------------------------------------------------------------- |
| **Health-check**    | `GET /api/v1/health` (200 = OK). Використовується deploy.sh, Dockerfile HEALTHCHECK, cron'ом. |
| **`/admin/health`** | Адмін-дашборд стану: БД, бекапи (читає `verify-status.json`), інтеграції.                     |
| **Sentry**          | Серверні + клієнтські помилки (`SENTRY_DSN` / `NEXT_PUBLIC_SENTRY_DSN`).                      |
| **Axiom**           | Хмарне логування (`AXIOM_TOKEN`, free 500 МБ/міс).                                            |
| **UptimeRobot**     | Зовнішній моніторинг доступності (пінгує сайт).                                               |
| **PostgreSQL**      | `pg_stat_statements` + логування запитів > 500 мс (в `docker-compose.yml`).                   |

**Логи:**

- PM2: `~/.pm2/logs/pulito-{out,error}.log`
- Docker (з ротацією): `docker compose logs -f <service>`
- Nginx: `/var/log/nginx/`
- Cron: `/home/pulitotrade/cron-run.log` та `cron-*.log`

## 8. Docker-деплой

Альтернатива bare-metal+PM2 — повний контейнер. `Dockerfile` — 3 стадії:

1. **`deps`** (`node:20-alpine`) — `npm ci`.
2. **`builder`** — `NEXT_BUILD_STANDALONE=1`, `npx prisma generate`, `npm run build` → standalone-вивід.
3. **`runner`** — non-root юзер `nextjs`, копіює `.next/standalone` + `static` + `public`, `CMD ["node", "server.js"]`. HEALTHCHECK кожні 30 с пінгує `/api/v1/health`.

`docker-compose.yml` сервіс `app` будується з цього Dockerfile, ходить у БД через **PgBouncer (6432)**, вимагає `APP_SECRET`/`JWT_SECRET`.

```bash
docker compose up -d --build app
```

## 9. CI/CD (GitHub Actions)

`.github/workflows/`:

- **`ci.yml`** — на push/PR у `main`/`develop`: lint+typecheck, security-scan (npm audit + сканер секретів), unit-tests+coverage, build, docker-build, e2e (Postgres+Redis services), deploy-staging (на `develop`).
- **`deploy.yml`** — деплой на сервер (rsync через SSH).
- **`backup.yml`** — бекап-пайплайн.
- **`cron-payments-tracking.yml`** — payments/tracking cron через Actions-розклад.

CI використовує Node 24, кешує `node_modules`, `.next/cache`, Playwright-браузери.

## 10. Застосування змін ENV на проді

- **Server-only** (`TELEGRAM_*`, `NOVA_POSHTA_API_KEY`, `SMTP_*`, платіжні ключі) → `pm2 restart pulito`.
- **`NEXT_PUBLIC_*` / структурні** → rebuild: **`./deploy.sh`** (не in-place build!).

> Багато налаштувань тепер у `/admin → Налаштування` (DB-first), і їх зміна **не потребує** редеплою. Платіжні ключі та боти — лише через адмінку (шифрування).
