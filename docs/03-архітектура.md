# 03. Архітектура

Цей документ описує внутрішню будову Pulito: шарову архітектуру, структуру папок `src/` та `prisma/`, роутинг, де живе бізнес-логіка, як рухаються дані (data flow), фонові задачі та подієву систему. Усе звірено з реальними файлами репозиторію репозиторію.

> Що це за платформа — у [01-огляд-проєкту.md](01-огляд-проєкту.md). Конкретні версії технологій — у [02-технології.md](02-технології.md).

## Шарова архітектура

Pulito побудовано на Next.js 16 (App Router) за класичною **шаровою (layered)** моделлю. Запит проходить зверху вниз; кожен шар має чітку відповідальність і не «перестрибує» сусіда.

```
┌────────────────────────────────────────────────────────────────┐
│  1. FRONTEND  (React 19 + Tailwind 4)                          │
│     • Server Components (RSC) — рендер на сервері               │
│     • Client Components — інтерактив, hooks, SWR                │
│     • Server Actions (src/app/actions) — мутації без API        │
│     ├── app/[locale]/(shop)  — публічна вітрина                 │
│     ├── app/(admin)/admin    — адмін-панель                     │
│     └── providers/           — Auth, Cart, Settings, Theme      │
├────────────────────────────────────────────────────────────────┤
│  2. EDGE / MIDDLEWARE  (src/proxy.ts)                          │
│     • next-intl: визначення локалі, rewrite [locale]            │
│     • in-memory rate-limit (перша лінія)                        │
│     • security headers (CSP+nonce, HSTS, COOP/COEP)             │
│     • slug-redirects (308), maintenance-mode                    │
├────────────────────────────────────────────────────────────────┤
│  3. API ROUTE HANDLERS  (src/app/api/v1/* — 516 роутів)        │
│     • REST-ендпоінти, версіоновані під /api/v1                  │
│     • валідація входу через Zod (src/validators)                │
│     • обгортки: withAuth / withRole / withRateLimit             │
│     • стандартна відповідь: successResponse / errorResponse     │
├────────────────────────────────────────────────────────────────┤
│  4. MIDDLEWARE-ДЕКОРАТОРИ  (src/middleware)                    │
│     • auth.ts — withAuth() / withRole() (перевірка JWT, ролей)  │
│     • api-rate-limit.ts — Redis sliding window (друга лінія)    │
│     • api-key-auth.ts, api-version.ts                          │
├────────────────────────────────────────────────────────────────┤
│  5. SERVICES  (src/services — ~277 файлів)  ← БІЗНЕС-ЛОГІКА    │
│     • order, payment, cart, product, loyalty, referral...       │
│     • payment-providers/, jobs/ (фонові)                        │
│     • емітять доменні події (event-bus)                         │
├────────────────────────────────────────────────────────────────┤
│  6. DATA LAYER  (src/lib)                                      │
│     • prisma.ts → PostgreSQL (через PgBouncer)                  │
│     • redis.ts → кеш / сесії / rate-limit / pub-sub             │
│     • Typesense → пошук;  storage.ts → R2/S3                    │
│     • queue/ → BullMQ (email, push, pdf, marketplace-sync)      │
└────────────────────────────────────────────────────────────────┘
            ▲                                          │
            │   event-bus (Redis pub/sub)   cron (~60) │
            └──────── фонові процеси ◄─────────────────┘
```

### Відповідальність кожного шару

| #   | Шар                       | Відповідальність                                                | Що НЕ робить                                  |
| --- | ------------------------- | --------------------------------------------------------------- | --------------------------------------------- |
| 1   | **Frontend**              | Рендер UI, інтерактив, виклик API/Server Actions                | Не містить бізнес-правил (ціни, статуси)      |
| 2   | **Middleware (proxy)**    | Локаль, headers, rate-limit, redirects — на кожному запиті      | Не лізе в БД важко                            |
| 3   | **API route handlers**    | Прийняти запит, провалідувати, викликати сервіс, повернути JSON | Не містить складної логіки — делегує сервісам |
| 4   | **Middleware-декоратори** | Auth/ролі/rate-limit як обгортки навколо хендлерів              | Не знають про конкретну бізнес-задачу         |
| 5   | **Services**              | Уся бізнес-логіка: транзакції, правила, оркестрація             | Не парсять HTTP напряму                       |
| 6   | **Data layer**            | Доступ до PostgreSQL / Redis / Typesense / черг / storage       | Не містить бізнес-правил                      |

## Структура папок `src/`

```
src/
├── app/                  # Next.js App Router: сторінки, API, server actions
│   ├── [locale]/         # Локалізований сегмент (uk/en)
│   │   └── (shop)/       # Публічна вітрина (route group)
│   ├── (admin)/          # Адмін-панель (route group, без локалі)
│   ├── api/              # Route handlers
│   │   ├── v1/           # Версіонований REST API (~516 route.ts)
│   │   └── webhooks/     # Вебхуки платіжних провайдерів
│   ├── actions/          # Server Actions
│   ├── api-docs/         # Swagger UI
│   ├── blog/, feed/, sitemap*, robots.ts, manifest.ts, opengraph-image.tsx ...
│   └── layout.tsx, error.tsx, not-found.tsx, offline/, maintenance/
├── components/           # ~441 React-компонентів (ui/, layout/, admin/, checkout/, common/)
├── services/             # ~277 файлів бізнес-логіки  ← серце системи
│   ├── payment-providers/  # liqpay, monobank, wayforpay
│   └── jobs/             # ~50+ фонових job-ів (cart-abandonment, price-sync, ...)
├── lib/                  # Інфраструктурні клієнти та утиліти
│   ├── prisma.ts, prisma-tenant.ts   # Prisma singleton + tenant-scoped
│   ├── redis.ts          # Redis + TTL-константи
│   ├── event-bus.ts, event-handlers.ts  # подієва система
│   ├── queue/            # BullMQ (connection, queues, workers)
│   ├── api-handler.ts, api-client.ts    # обгортки серверного/клієнтського HTTP
│   ├── encryption.ts, logger.ts, sentry.ts, storage.ts, swagger.ts ...
├── validators/           # ~62 Zod-схеми валідації входу
├── middleware/           # auth.ts, api-rate-limit.ts, api-key-auth.ts, api-version.ts, rate-limit-config.ts
├── hooks/                # ~42 React-хуки
├── providers/            # AuthProvider, CartProvider, SettingsProvider, ThemeProvider, WishlistBulkProvider
├── utils/                # ~34 чисті утиліти (money.ts, api-response.ts, kyiv-date ...)
├── i18n/                 # routing.ts, request.ts, navigation.ts (next-intl)
├── messages/             # uk.json (default), en.json — переклади
├── config/               # env.ts (Zod-валідований), locale-config.ts, admin-constants.ts
├── types/                # TypeScript-типи доменів (order, payment, loyalty, ...)
├── generated/            # Згенеровані артефакти (slug-redirects.ts тощо)
├── assets/, test/        # статика, тестові helpers
├── proxy.ts              # Edge middleware (локаль, headers, rate-limit, redirects)
└── instrumentation.ts    # Sentry/telemetry init
```

### Призначення ключових папок

- **`app/`** — сторінки (Server/Client Components), route handlers (`route.ts`) і Server Actions. Тут немає бізнес-логіки: сторінки рендерять, хендлери делегують у `services/`.
- **`components/`** — переиспользуваний UI. `ui/` — примітиви (Button, Input, Modal); `layout/` — Header/Footer/Sidebar; `admin/` — адмінські; `checkout/` — кроки checkout; `common/` — спільні (WebVitalsReporter).
- **`services/`** — **єдине місце бізнес-логіки**. Сервіс інкапсулює правила (наприклад, матрицю переходів статусів замовлення, нарахування балів, ефективну ціну). Кілька route-ів можуть викликати один сервіс.
- **`lib/`** — інфраструктура: клієнти БД/Redis/storage, event-bus, черги, обгортки, логер. Без бізнес-правил.
- **`validators/`** — Zod-схеми; кожен мутаційний route валідує тіло запиту перед викликом сервісу.
- **`middleware/`** — декоратори `withAuth`/`withRole` та rate-limit, які обгортають route handlers.
- **`utils/`** — чисті функції без побічних ефектів (`money.ts` — int-копійки, kyiv-date helpers для коректних добових меж у часовому поясі Києва, `api-response.ts`).
- **`config/env.ts`** — централізована Zod-валідована конфігурація середовища (див. [05-конфігурація-env.md](05-конфігурація-env.md)).
- **`providers/`** — React context-провайдери (Auth, Cart, Settings, Theme, WishlistBulk).

## Роутинг

Pulito використовує App Router із **route groups** і локалізованим сегментом.

| Шаблон URL                                                                       | Файлова локація           | Призначення                               |
| -------------------------------------------------------------------------------- | ------------------------- | ----------------------------------------- |
| `/` , `/catalog`, `/product/...`, `/cart`, `/checkout`, `/account/...`           | `app/[locale]/(shop)/...` | Публічна вітрина + кабінет (локалізовані) |
| `/uk/...` (за потреби), без префіксу для дефолтної `uk`                          | `app/[locale]/...`        | i18n стратегія `as-needed`                |
| `/admin/...`                                                                     | `app/(admin)/admin/...`   | Адмін-панель (без локалі)                 |
| `/api/v1/...`                                                                    | `app/api/v1/.../route.ts` | Версіонований REST API                    |
| `/api/webhooks/...`                                                              | `app/api/webhooks/...`    | Вебхуки (LiqPay, Monobank)                |
| `/order/...`, `/r/...`, `/sitemap.xml`, `/robots.txt`, `/feed`, `/blog/feed.xml` | поза `[locale]`           | Трекінг, редіректи, SEO-артефакти         |

**Ключові деталі з `src/proxy.ts`:**

- `next-intl` обробляє визначення локалі та rewrite сегмента `[locale]`. Префікси у `I18N_EXEMPT_PREFIXES` (`/api`, `/admin`, `/sitemap`, `/feed`, `/r/`, `/uploads`, `/order`, `/manifest`, `/robots`, `/api-docs`, `/offline`, `/maintenance`) **не** несуть локалі.
- `/auth` та `/account` живуть **усередині** `app/[locale]/(shop)`, тому **мусять** локалізуватися — їх не можна додавати до exempt-списку (інакше `/auth/callback` дає 404).
- Статичні-подібні файли (`/sw.js`, `/sitemap.xml`, `/robots.txt`, `/llms.txt`) завжди пропускають i18n.
- Тут же — slug-redirects (308 old→new) із `src/generated/slug-redirects.ts` та security headers.

## Де живе бізнес-логіка

**Правило:** уся бізнес-логіка — у `src/services/`. Route handlers — тонкі.

Типовий route handler:

```typescript
// app/api/v1/orders/route.ts (схематично)
export const POST = withAuth(async (request) => {
  const body = await request.json();
  const data = createOrderSchema.parse(body); // 1. валідація (validators/)
  const order = await createOrder(userId, data); // 2. сервіс (services/)
  return successResponse(order); // 3. стандартна відповідь (utils/)
});
```

Приклади сервісів та їх обов'язків:

- **`order.ts`** — `createOrder` (валідація кошика, застосування правил ціноутворення, купонів, балів), `updateOrderStatus` із матрицею переходів `new_order → processing → confirmed → paid → shipped → completed`; на `completed` — нарахування балів і оновлення рефералів; на `cancelled/returned` — списання балів.
- **`payment.ts`** — `initiatePayment` / `handlePaymentCallback` (перевірка підпису, оновлення статусу).
- **`loyalty.ts`** — нарахування/списання балів із множником рівня, перерахунок рівня.
- **`personal-price.ts`** — `getEffectivePrice` (товарна ціна має пріоритет над категорійною).
- **`analytics.ts`** — воронка, когорти, ABC-аналіз.

## Data flow (рух даних)

**Читання (GET):**

```
Browser → proxy.ts → RSC/route handler → service → lib/prisma (PG) / redis (cache) / Typesense → дані → рендер/JSON
```

**Запис (мутація):**

```
Browser → proxy.ts → route handler (withAuth/withRole)
        → Zod-валідація → service (транзакція в PG)
        → emit(domain-event) → response (successResponse)
                    │
                    ├── event-bus локальні handler-и
                    └── Redis pub/sub → інші процеси
```

- **Кешування:** «гарячі» дані (налаштування, каталог-фрагменти) кешуються в Redis із TTL-константами (`src/lib/redis.ts`). Динамічні сторінки (головна) рендеряться `no-store`; SEO-сторінки — через ISR.
- **Стандартна відповідь:** `successResponse()` / `errorResponse()` / `paginatedResponse()` з `utils/api-response.ts` — єдиний формат `{ success, data | error }`.
- **Гроші:** усі грошові обчислення — через `@/utils/money` (int-копійки), щоб уникнути float-помилок.

## Фонові задачі: BullMQ + cron

Два механізми фонової роботи:

### 1. BullMQ-черги (`src/lib/queue/`)

Надійні черги з retry/backoff для задач, що можуть впасти або бути повільними:

| Черга              | Призначення                        | attempts / backoff  |
| ------------------ | ---------------------------------- | ------------------- |
| `email`            | Розсилка листів                    | 3 / exponential 5s  |
| `push`             | Web Push                           | 2 / exponential 3s  |
| `pdf`              | Генерація PDF (рахунки/каталог/КП) | 2 / fixed 10s       |
| `marketplace-sync` | Синхронізація з маркетплейсами     | 3 / exponential 15s |

Воркери — у `queue/workers.ts`, з'єднання — `queue/connection.ts` (ioredis). Черги — lazy-singletons.

### 2. Cron-роути (`/api/v1/cron/*`, ~60 шт.)

Заплановані задачі реалізовані як **захищені HTTP-ендпоінти** (перевірка `CRON_SECRET`), які тригерить зовнішній планувальник (системний cron через `cron-run.sh`). Логіка кожної задачі — у `src/services/jobs/*`. Приклади: `cart-abandonment`, `price-sync`, `payment-reconciliation`, `health-check`, `promo-autopost`, `expire-loyalty`, `process-subscriptions`, `funnel-aggregate`, `db-backup`, `partition-rotate`, `marketplace-token-refresh`.

> Не всі ~60 роутів увімкнено — лише потрібні/безпечні заплановано в системному cron. `cron-lock.ts` забезпечує fail-closed блокування, щоб задача не виконувалась паралельно. Деталі планування — у [19-деплой-операції.md](19-деплой-операції.md).

## Подієва система (event-bus)

Легка доменна шина подій на Redis pub/sub (`src/lib/event-bus.ts`):

```typescript
emit({ type: 'order.completed', payload: { orderId, userId } });
```

- **`emit(event)`** публікує подію в Redis-канал `domain-events` (для крос-процесної комунікації між PM2-воркерами) **і** виконує локальні handler-и через `Promise.allSettled` (помилки одного handler-а не валять інших).
- **`on(type, handler)`** реєструє обробник. Усі обробники реєструються одноразово у `registerEventHandlers()` (`src/lib/event-handlers.ts`).

**Типи доменних подій:** `order.created`, `order.completed`, `order.cancelled`, `product.updated`, `product.stock_changed`, `user.registered`, `review.created`, `subscription.renewed`.

Приклади реакцій: на `order.completed` оновлюються loyalty-стріки користувача; `product.stock_changed` — seam для marketplace-sync / low-stock alert. Event-bus навмисно «м'який» (fire-and-forget): якщо Redis недоступний, подія обробляється лише локально, основний потік не блокується.

## i18n (інтернаціоналізація)

- **Бібліотека:** `next-intl` 4.8.x. **Локалі:** `uk` (default), `en`. **Стратегія:** `as-needed` (без `/uk`-префікса для дефолтної локалі).
- **Конфіг:** `src/i18n/routing.ts` (локалі), `request.ts` (серверний resolver), `navigation.ts` (клієнтська навігація).
- **Переклади:** `src/messages/uk.json`, `en.json`. У клієнтських компонентах — `useTranslations('namespace')`.
