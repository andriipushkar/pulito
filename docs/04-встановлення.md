# 04. Встановлення та локальна розробка

Цей розділ описує, як підняти платформу **Pulito** (інтернет-магазин побутової хімії на Next.js 16) на локальній машині для розробки: від клонування репозиторію до запущеного dev-сервера з повним набором інфраструктури (PostgreSQL, PgBouncer, Redis, Typesense).

Платформа — це монолітний застосунок Next.js 16 (App Router) із Prisma 7 (ORM), Redis (кеш/черги BullMQ), Typesense (повнотекстовий пошук) і PostgreSQL 16 як основною БД. Усі залежності інфраструктури підіймаються одним `docker compose`.

---

## Передумови

| Інструмент                         | Версія                             | Навіщо                                                               |
| ---------------------------------- | ---------------------------------- | -------------------------------------------------------------------- |
| **Node.js**                        | ≥ 20 LTS (CI використовує Node 24) | Виконання застосунку, npm-скриптів, Prisma CLI                       |
| **npm**                            | йде в комплекті з Node             | Менеджер пакетів (репозиторій має `package-lock.json` → `npm ci`)    |
| **Docker** + **Docker Compose v2** | актуальна                          | PostgreSQL, PgBouncer, Redis, Typesense                              |
| **Git**                            | будь-яка                           | Клонування репозиторію                                               |
| **openssl**                        | будь-яка                           | Генерація секретів (`JWT_SECRET`, `APP_SECRET`, `TYPESENSE_API_KEY`) |

> Бінарного встановлення PostgreSQL/Redis на хост **не потрібно** — усе піднімається в Docker через `docker-compose.yml`. Якщо ви все ж тримаєте PostgreSQL/Redis локально, узгодьте порти з `.env`.

## Швидкий старт (Quick Start)

```bash
# 1. Клонувати репозиторій
git clone <repo-url> pulito
cd pulito

# 2. Встановити залежності (детермінований інстал з lock-файлу)
npm install            # або: npm ci

# 3. Підготувати конфігурацію оточення
cp .env.example .env
#   Відредагуйте .env — щонайменше згенеруйте секрети:
#   JWT_SECRET, APP_SECRET, TYPESENSE_API_KEY (див. розділ 05)
#   openssl rand -hex 32

# 4. Підняти інфраструктуру (Postgres + PgBouncer + Redis + Typesense)
docker compose up -d
#   або npm-скрипт:
npm run docker:up

# 5. Згенерувати Prisma-клієнт і прогнати міграції
npm run db:generate
npm run db:migrate

# 6. (Опціонально) Залити демо-дані
npm run db:seed

# 7. Запустити dev-сервер
npm run dev
#   → http://localhost:3000
```

Після `npm run dev` застосунок доступний на `http://localhost:3000`. Адмінка — на `/admin`.

> **Порядок важливий:** Prisma-клієнт (`db:generate`) має бути згенерований до першого запуску `dev`/`build`, інакше імпорт `@prisma/client` впаде. `db:migrate` створює схему в БД, яка вже має бути піднята через `docker compose up -d`.

## Опис усіх npm-скриптів

Усі скрипти беруться з `package.json` (секція `scripts`). Запускайте з кореня проєкту: `npm run <скрипт>`.

### Розробка та збірка

| Скрипт    | Команда                                                         | Призначення                                                                                        |
| --------- | --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `dev`     | `next dev`                                                      | Dev-сервер із hot-reload на `localhost:3000`. Основний режим розробки.                             |
| `build`   | `NODE_OPTIONS='--max-old-space-size=3072' next build --webpack` | Production-збірка. Heap обмежено до **3072 МБ** (захист від OOM). Використовує webpack-бекенд.     |
| `start`   | `next start`                                                    | Запуск **уже зібраного** застосунку (`.next/`) у production-режимі. PM2 на проді запускає саме це. |
| `analyze` | `ANALYZE=true next build`                                       | Збірка з аналізатором бандлів (`@next/bundle-analyzer`) — показує розмір чанків.                   |

### Якість коду

| Скрипт      | Команда                                                   | Призначення                                                                                                                      |
| ----------- | --------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `lint`      | `eslint . --fix`                                          | ESLint з автофіксом по всьому проєкту.                                                                                           |
| `typecheck` | `tsc --noEmit`                                            | Перевірка типів TypeScript без емісії. Той самий gate, що в `deploy.sh`.                                                         |
| `format`    | `prettier --write "src/**/*.{ts,tsx,js,jsx,json,css,md}"` | Форматування Prettier по всьому `src/`.                                                                                          |
| `prepare`   | `husky`                                                   | Інсталяція git-хуків (через Husky) — автозапуск при `npm install`. `lint-staged` фіксує/форматує застейджені файли в pre-commit. |

### База даних (Prisma)

| Скрипт            | Команда                 | Призначення                                                                                                                               |
| ----------------- | ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `db:generate`     | `prisma generate`       | Генерує Prisma-клієнт зі схеми (`prisma/schema.prisma`). Запускати після кожної зміни схеми.                                              |
| `db:migrate`      | `prisma migrate dev`    | **Dev**: створює нову міграцію з diff-у схеми, застосовує її, регенерує клієнт.                                                           |
| `db:migrate:prod` | `prisma migrate deploy` | **Prod**: застосовує наявні міграції без створення нових і без інтерактивності.                                                           |
| `db:push`         | `prisma db push`        | Пушить схему в БД без створення файлів міграцій. ⚠️ **НЕ запускати на проді** — ламає `search_vector` (GENERATED-колонку) та GIN-індекси. |
| `db:seed`         | `tsx prisma/seed.ts`    | Заливає демо/довідкові дані (`prisma/seed.ts`).                                                                                           |
| `db:studio`       | `prisma studio`         | GUI Prisma Studio для перегляду/редагування БД (`localhost:5555`).                                                                        |
| `db:reset`        | `prisma migrate reset`  | ⚠️ Дропає БД, перестворює зі скретчу, прогоняє всі міграції + seed. Знищує всі дані.                                                      |

## Сервіси docker-compose

Файл `docker-compose.yml` визначає п'ять сервісів. У dev зазвичай потрібні чотири інфраструктурні (`postgres`, `pgbouncer`, `redis`, `typesense`); сервіс `app` — для повного контейнеризованого запуску.

### `postgres` — основна БД

- **Образ:** `postgres:16-alpine`
- **Порт:** `5432`
- **Дані:** volume `postgres_data`
- **Користувач/БД:** `POSTGRES_USER` (деф. `pulito_user`), `POSTGRES_DB` (деф. `pulito_trade`), пароль `POSTGRES_PASSWORD` (обов'язковий).
- **Тюнінг:** увімкнено `pg_stat_statements` (трекінг усіх запитів), логування запитів довших за **500 мс**, `max_connections=50`.
- **Healthcheck:** `pg_isready` кожні 10 с.

> У production-конфігурації застосунок ходить у БД **через PgBouncer (порт 6432)**, а не напряму в Postgres.

### `pgbouncer` — пул з'єднань

- **Образ:** `edoburu/pgbouncer:v1.25.1-p0`
- **Порт:** `6432`
- **Режим пулу:** `transaction`
- **Розміри пулу:** `DEFAULT_POOL_SIZE=10`, `MIN_POOL_SIZE=2`, `RESERVE_POOL_SIZE=5`, `MAX_CLIENT_CONN=200`, idle-таймаут сервера 600 с.
- **Auth:** `scram-sha-256`.
- Розв'язує проблему вичерпання конектів Postgres за великої кількості клієнтів.

### `redis` — кеш та черги

- **Образ:** `redis:7-alpine`
- **Порт:** `6380` на хості → `6379` у контейнері (тому `REDIS_URL` локально вказує на `6380`).
- **Дані:** volume `redis_data`
- **Ліміти:** `maxmemory 256mb`, політика витіснення `allkeys-lru`.
- Використовується для кешу, сесій і черг BullMQ.

### `typesense` — повнотекстовий пошук

- **Образ:** `typesense/typesense:27.1`
- **Порт:** `8108`
- **Дані:** volume `typesense_data`
- **API-ключ:** `TYPESENSE_API_KEY` (обов'язковий).

### `app` — застосунок (production-контейнер)

- Збирається з локального `Dockerfile`.
- **Порт:** `3000`
- Підключається до БД **через PgBouncer** (`pgbouncer:6432`), до Redis (`redis:6379`).
- Вимагає `APP_SECRET` і `JWT_SECRET` (compose падає, якщо їх не задано).
- Залежить від healthcheck'ів `pgbouncer` і `redis`.

> Логування у кожного сервісу обмежене через `json-file` драйвер з ротацією (`app`: 7×50 МБ, `postgres`: 5×20 МБ, `redis`/`typesense`: 3×10 МБ).

## Workflow роботи з Prisma

```bash
# 1. Змінили prisma/schema.prisma → створіть міграцію (dev)
npm run db:migrate          # prisma migrate dev — питає назву міграції

# 2. Перегенеруйте клієнт (db:migrate робить це автоматично, але вручну:)
npm run db:generate

# 3. Перегляньте/відредагуйте дані вручну
npm run db:studio           # GUI на localhost:5555

# 4. Залийте seed-дані
npm run db:seed

# 5. На production застосовуйте міграції без створення нових
npm run db:migrate:prod     # prisma migrate deploy

# 6. Повний reset (ТІЛЬКИ dev — знищує дані)
npm run db:reset
```

**Правила безпеки:**

- ⚠️ **Ніколи не запускайте `npm run db:push` (`prisma db push`) на production.** Це руйнує `search_vector` (GENERATED-колонку повнотекстового пошуку) і GIN-індекси — пошук по каталогу перестає працювати на всьому сайті. На проді тільки `db:migrate:prod`.
- ⚠️ `db:reset` дропає базу — лише для локальної розробки.
- Після зміни схеми завжди `db:generate`, інакше TypeScript бачить старий клієнт.

## Типові проблеми при встановленні

| Симптом                               | Причина / рішення                                                            |
| ------------------------------------- | ---------------------------------------------------------------------------- |
| `Cannot find module '@prisma/client'` | Не виконано `npm run db:generate`.                                           |
| Помилка з'єднання з БД                | Не піднято `docker compose up -d`, або `DATABASE_URL` вказує не на той порт. |
| Redis ECONNREFUSED                    | Локально Redis на порту **6380**, не 6379 — перевірте `REDIS_URL`.           |
| `JWT_SECRET / APP_SECRET must be set` | Не заповнені секрети в `.env` (див. розділ 05).                              |
| OOM під час `build`                   | Не запускайте дві збірки одночасно; heap обмежено 3072 МБ (див. розділ 19).  |
