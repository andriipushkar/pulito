# 12. Маркетплейси

Pulito інтегрується з чотирма українськими маркетплейсами для автоматичної публікації товарів, імпорту замовлень/відгуків, синхронізації цін і залишків, листування з покупцями та обробки повернень. Адмінка — `/admin/marketplaces`.

## Архітектура

```
Admin UI  →  API routes  →  Services layer  →  External APIs / YML-фіди
                                ↓
                          Prisma (DB)
```

- **UI**: `src/app/(admin)/admin/marketplaces/*`
- **API**: `src/app/api/v1/admin/marketplaces/*`, `src/app/api/v1/cron/*`, фіди `src/app/api/v1/feeds/*`
- **Сервіси**: `marketplaces.ts` (диспетчер + платформенні handler'и), `marketplace-sync.ts` (фонова синхронізація), `channel-config.ts` (зашифровані credentials), `marketplace-rozetka.ts`/`marketplace-prom.ts` (специфічні клієнти), плюс модулі repricing/categories/rate-limit/quota/health/disputes/analytics/auto-crosslist.

## Підтримувані маркетплейси

| Платформа      | Спосіб           | Авторизація                 | Публікація | Замовлення | Залишок |  Повідомлення  | Повернення |
| -------------- | ---------------- | --------------------------- | :--------: | :--------: | :-----: | :------------: | :--------: |
| **OLX**        | API (Partner v2) | OAuth 2.0 + refresh         |     ✅     |     ✅     |   ✅    |       ✅       | ⚠️ (read)  |
| **Rozetka**    | API (Seller)     | login:password → 24h Bearer |     ✅     |     ✅     |   ✅    |       ✅       |     ✅     |
| **Prom.ua**    | YML-фід + API    | Bearer token                | через фід  |     ✅     |   ✅    |       ✅       |     ✅     |
| **Epicentr K** | YML-фід + API    | Bearer (API key)            | через фід  |     ✅     |   ✅    | ❌ (немає API) | ⚠️ (read)  |

**Ключова різниця:** OLX і Rozetka підтримують **створення товарів через API**. Prom.ua і Epicentr K **не мають API створення** — нові товари потрапляють на них **лише через YML-фід** (див. нижче). Усі чотири мають API для замовлень/цін/залишків.

## Конфігурація та шифрування

Credentials зберігаються в `SiteSetting` під ключем `channel_<platform>` у зашифрованому вигляді (`channel-config.ts`). `SENSITIVE_FIELDS` (`apiKey`, `apiSecret`, `apiToken`, `accessToken`, `refreshToken`, `clientSecret`, `password`, …) шифруються AES-256-GCM через `@/lib/encryption`. Plaintext-credentials авто-мігрують у шифровані при читанні.

При збереженні — **merge**: порожні чутливі поля **не перезаписують** збережені значення (захист від стирання токенів). У відповідях API — masked (`maskChannelConfig` → `•••...last4`).

## Платформи

### OLX

API Partner v2.0, OAuth 2.0 з авто-refresh.

- **Поля конфігу:** `clientId`, `clientSecret`, `accessToken`, `refreshToken`, `accessTokenExpiresAt`, `defaultCategoryId`, `contactName`, `contactPhone`, `cityId`.
- **Токен:** `refreshOlxToken()` → `POST olx.ua/api/open/oauth/token` (`grant_type=refresh_token`); авто-refresh на 401 через `olxFetchWithAuthRetry()`; cron `marketplace-token-refresh` оновлює до закінчення.
- **Публікація:** `POST /api/partner/adverts` (header `Version: 2.0`); title ≤70, description ≤9000, до 8 фото, `advertiser_type=business`, `state=new`, `price.value` (цілі грн); код товару дописується в опис. `externalId` зберігається в `PublicationChannel`.
- **Замовлення:** `GET /api/partner/orders?limit=50`.
- **Повідомлення:** `GET /threads` → upsert у `MarketplaceMessage`; відповідь `POST /threads/{id}/messages`.
- **Повернення:** `GET /returns` (read-only, без API рішення).
- **Health:** `GET /api/partner/users/me`. **Rate-limit:** 6000/год.

### Rozetka

Seller API, файл `marketplace-rozetka.ts`, клас `RozetkaClient`.

- **Поля конфігу:** `apiKey` (формат `login:password`), `sellerId`.
- **Токен:** `authenticate()` → `POST /sites` з `{username, password: base64(password)}` → Bearer ~24 год; авто-refresh при простроченні.
- **Публікація:** `POST /items` — `name`, `price`, `currency=UAH`, `quantity`, `status=active`, `article` (SKU); опційно `barcode`, `category_id`, `images[]` (до 10), двомовні `name_ua`/`description_ua`.
- **Замовлення:** `GET /orders/search?per_page=50` (пагінація, PAGE_CAP=40).
- **Повідомлення:** `GET /orders/messages`; відповідь `POST /orders/messages`.
- **Повернення:** `GET /returns/search`; рішення `POST /returns/{id}/{accept|reject|complete}`.
- **Залишок:** `PUT /items/{id}` `{quantity}`. **Rate-limit:** 1000/год.

### Prom.ua

Файл `marketplace-prom.ts`, клас `PromClient`. Bearer token. **Без API створення товарів.**

- **Замовлення:** `GET /orders/list?limit=50&offset` (пагінація).
- **Повідомлення:** `GET /messages/list`; відповідь `POST /messages/reply`.
- **Ціна/залишок:** `POST /products/edit` `{products:[{id, price, quantity_in_stock, presence}]}` (`presence` авто `available`/`not_available`).
- **Повернення:** `GET /returns/list`; рішення `POST /returns/set_status`.
- **Публікація:** через YML-фід `/api/v1/feeds/prom.xml`. **Rate-limit:** 1000/год.

### Epicentr K

Файл `marketplaces.ts`. Bearer (API key). **Без API створення товарів і без API повідомлень.**

- **Замовлення:** `GET merchant-api.epicentrm.com.ua/v1/orders?limit=50`.
- **Ціна/залишок:** `PUT /v1/products/{id}` `{name, description, price, stock}`.
- **Повернення:** `GET /v1/returns` (read-only).
- **Публікація:** через YML-фід `/api/v1/feeds/epicentr.xml`. **Rate-limit:** 1000/год.

---

## YML-фіди (Prom + Epicentr)

Для платформ без API створення товарів генеруються динамічні YML-каталоги:

- `GET /api/v1/feeds/prom.xml` — формат YML за специфікацією Prom; підключення в кабінеті Prom: **Імпорт → за посиланням → вставити URL фіду**.
- `GET /api/v1/feeds/epicentr.xml` — той самий формат, інший хост.

Фіди динамічні (запит активних товарів → `<yml_catalog>`), кешуються `Cache-Control: public, max-age=1800`. Код товару, фото, ціна присутні в кожному offer.

## Синхронізація (cron)

Усі cron-роути захищені `Authorization: Bearer ${APP_SECRET}` + `withCronLock(1800)`.

| Endpoint                     | Що робить                                                                    | Періодичність |
| ---------------------------- | ---------------------------------------------------------------------------- | ------------- |
| `sync-marketplace-orders`    | Імпорт замовлень з усіх увімкнених → `Order`+`OrderItem` + декремент залишку | ~30 хв        |
| `sync-marketplace-stock`     | Push залишку (з урахуванням allocation)                                      | щоцикл        |
| `sync-marketplace-prices`    | Push цін (markup + repricing)                                                | N годин       |
| `sync-marketplace-messages`  | Пул повідомлень покупців                                                     | за конфігом   |
| `import-marketplace-reviews` | Імпорт відгуків (OLX/Rozetka)                                                | N годин       |
| `marketplace-token-refresh`  | Оновлення OLX access_token                                                   | щогодини      |
| `marketplace-health-check`   | Пінг API, uptime                                                             | ~15 хв        |
| `marketplace-auto-crosslist` | Авто-публікація нових товарів на всі платформи                               | щодня         |

### Цикл життя товару

1. Адмін публікує: `POST /api/v1/admin/products/:id/marketplaces {channel}`.
2. `publishToMarketplace()` → handler платформи → зберігає `externalId` у `PublicationChannel`.
3. Cron `sync-marketplace-orders`: тягне замовлення, атомарно (Prisma-транзакція) декрементує `product.quantity` по `productCode`. Якщо залишку не вистачає — `quantity := 0` + Telegram-алерт `notifyManagerOversoldAlert`. Після імпорту викликає `syncProductsStockToMarketplaces()`, щоб усі інші маркетплейси побачили новий залишок.
4. Cron `sync-marketplace-prices`: пушить актуальну ціну (з націнкою + repricing).

## Ціноутворення (repricing)

Файли `marketplaces.ts` (`getMarketplacePrice`), `marketplace-repricing.ts`.

1. **Націнка каналу** — поле `priceMarkupPercent` (clamp ±50%): `price × (1 + markup/100)`.
2. **Repricing-правила** — `marketplace_repricing_{platform}` (JSON-масив). Умови: `stock_below`/`above`, `date_between`, `category_in`, `product_in`, `always`. Перше правило, що збіглося, додає свій `markupPercent` (clamp ±50%) **поверх** націнки каналу. Best-effort — ніколи не блокує публікацію.

**Приклад:** база 100 ₴ → націнка каналу +5% → 105 ₴ → правило «stock < 10» +10% → **115.5 ₴**.

---

## Залишки та захист від overselu

Файл `marketplaces.ts` (`getMarketplaceStock`).

- **Прив'язка до складу** (`warehouseId`): якщо задано — залишок з конкретного складу (`WarehouseStock.quantity − reserved`), інакше загальний `quantity`.
- **stockAllocationPercent** (0–100, деф. 100): який відсоток залишку показувати маркетплейсу. Для високообертових товарів — 50–70%. Allocated = `floor(baseQty × allocationPercent/100)`.
- **fulfilmentMode**: `fbs` (деф., продавець тримає товар — локальний декремент на замовлення) / `fbo` (маркетплейс тримає товар — локальний декремент пропускається).

---

## Повідомлення покупців

Файл `marketplace-messages-sync.ts`. Storage `MarketplaceMessage` (unique `[platform, externalThreadId]`).

- Cron `sync-marketplace-messages` тягне нові, upsert по `externalThreadId`.
- Сповіщення: Web Push адмінам (`sendPushToAdmins`) + Telegram (`notifyManagerMarketplaceMessages`).
- `POST /api/v1/admin/marketplaces/messages/reply` → `replyToMarketplaceMessage()` → API платформи (на успіх ставить `firstRespondedAt`). OLX `/threads/{id}/messages`, Rozetka/Prom — свої ендпоінти; **Epicentr — без API** (редірект у кабінет).
- `PATCH /messages/:id` — призначення менеджера (`assigneeId`), `isRead`. Шаблони відповідей — `MarketplaceReplyTemplate`, CRUD `/reply-templates`.

---

## Повернення та спори

Файл `marketplace-sync.ts` (returns), `marketplace-disputes.ts` (disputes).

- **Повернення:** cron + кнопка `POST /api/v1/admin/marketplaces/returns`. Storage `MarketplaceReturn` (unique `[connectionId, externalReturnId]`). Маппінг статусів: `pending/new`→pending, `approved/accepted`→approved, `rejected/declined`→rejected, `completed/refunded/closed`→completed.
- `PATCH /returns/:id {status}` оновлює локально ТА викликає `pushReturnDecision()`: Rozetka `POST /returns/{id}/{action}`, Prom `POST /returns/set_status`; для OLX/Epicentr — no-op (рішення в кабінеті). Прапорець `skipPush: true` оновлює статус без push (реконсиляція stale-даних).
- **Спори:** Rozetka `GET /disputes`, OLX `GET /complaints` → нормалізація → агрегація (`aggregateMarketplaceDisputes`). Prom/Epicentr — не експонують через API.

---

## Rate-limiting та квоти

- **Rate-limit** (`marketplace-rate-limit.ts`): Redis sorted-set, 5-хв sliding window на платформу. Ліміти/год: OLX 6000, решта 1000. `recordMarketplaceCall(platform)` (fire-and-forget); `getRateUsage` → `{count, limit5min, percent, warning: ≥80%}`. Fallback — in-memory Map якщо Redis недоступний.
- **Квоти** (`marketplace-quota.ts`): зчитує `X-RateLimit-Remaining/Limit/Reset` із відповідей API → `marketplace_quota_{platform}`; показ у дашборді.
- **Retry** (`marketplace-retry.ts`): retryable 429/5xx/мережеві (ECONNRESET/ETIMEDOUT); non-retryable 4xx/валідація/auth; експоненційний backoff (`base × 2^attempt + jitter`, до 3).

---

## Категорії та атрибути

Файл `marketplace-categories.ts`. Маппінг `marketplace_category_map_{platform}` (JSON `{localCategoryId: {externalId, externalName}}`). `resolveExternalCategory()` → `externalId` або fallback на `defaultCategoryId` каналу.

Валідація перед публікацією (`validateForMarketplace`): title 50–200 (OLX ≤70), description 30–50000 (OLX ≤9000), ≥1 фото (OLX ≤8, решта ≤10–12), ціна >0, код (опц. для OLX, обов'язк. для Rozetka/Epicentr). OLX: `attributes:[{code,value}]`; Rozetka: двомовні `name_ua`/`description_ua`.

---

## Health-моніторинг

Файл `marketplace-health.ts`. `runHealthCheck(platform)` → `testChannelConnection()`; пише latency/status/account у `marketplace_health_{platform}` + історія (rolling, до 200). Статуси: `ok`/`error`/`disabled`/`unconfigured`. Telegram-алерт `notifyManagerMarketplaceAlert` лише **на зміну стану** (не щопінгу). Дашборд `/admin/marketplaces` показує статус + час останньої синхронізації.

---

## Аналітика лістингів та авто-крослістинг

- **Аналітика** (`marketplace-listing-analytics.ts`): імпорт переглядів/кліків по лістингах; revenue по платформах (`/api/v1/admin/marketplaces/revenue`), price-parity (`/pricing-parity`).
- **Auto-crosslist** (`marketplace-auto-crosslist.ts`): `marketplace_auto_crosslist` (`{enabled, windowDays, excludePlatforms}`). Для активних товарів за останні N днів авто-публікує на всі ввімкнені платформи (пропускає вже опубліковані + виключені).

---

## Безпека та розширення

- Webhooks `src/app/api/webhooks/marketplaces/[platform]/route.ts` — HMAC `verifySignature()`. Admin-ендпоінти — `withRole('admin','manager')`. Чутливі поля — masked.
- **Sandbox / dry-run** у `/admin/marketplaces` → `publishToMarketplace()` повертає фейковий `externalId` без виклику API.
- **Додати маркетплейс:** додати `'newmp'` до `MARKETPLACE_CHANNELS`, написати handler'и (publish/update/delete/orders/messages), додати case у диспетчери, метадані в `_shared.ts`, тести.

## Адмінка `/admin/marketplaces`

Огляд (статус + health + квоти), `[id]` (конфіг платформи), а також підрозділи: `listings`, `orders`, `messages`, `returns`, `disputes`, `categories`, `repricing`, `pricing-parity`, `analytics`, `audit`, `buyer`, `help`.
