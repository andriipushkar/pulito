# 06. База даних та Prisma

Цей розділ — повний довідник по схемі бази даних платформи **Pulito** (інтернет-магазин побутової хімії, Next.js 16). База — **PostgreSQL**, доступ через ORM **Prisma 7.4**. Документ покриває організацію схеми, ~120 моделей, згрупованих за доменами, перелік усіх enum-типів, правила роботи з грошима та порядок міграцій.

---

## Загальний огляд

### ORM та підключення

- **ORM:** Prisma 7.4 (`prisma`, `@prisma/client`, `@prisma/adapter-pg` — усі `^7.4.0`).
- **Драйвер:** не вбудований Prisma-движок, а нативний `pg` (`Pool`) через адаптер `PrismaPg` (`@prisma/adapter-pg`). Клієнт ініціалізується в `src/lib/prisma.ts`.
- **Згенерований клієнт** лежить у `generated/prisma` (`output = "../../generated/prisma"` у `base.prisma`), а не в `node_modules`. Імпорт типів — з `../../generated/prisma`.
- **Сінглтон:** `prisma` зберігається у `globalThis` у dev-режимі, щоб HMR не плодив зайвих з'єднань.

### Пул з'єднань і PgBouncer

```ts
// src/lib/prisma.ts (скорочено)
const poolMax = Number(process.env.DATABASE_POOL_MAX) || 5;
const poolMin = Number(process.env.DATABASE_POOL_MIN) || 1;
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: poolMax,
  min: poolMin,
  idleTimeoutMillis: 30_000,
  connectionTimeoutMillis: 5_000,
});
const adapter = new PrismaPg(pool);
```

- App-side пул навмисно **малий** (`max=5`), бо сервер — дешевий VPS (1–2 ГБ RAM).
- У production **PgBouncer** виконує мультиплексування з'єднань; маленький app-pool не вичерпує пам'ять сервера.
- Розмір пулу налаштовується через `DATABASE_POOL_MAX` / `DATABASE_POOL_MIN` (див. [05. Конфігурація (ENV)](05-конфігурація-env.md)).

### Конвенції схеми

- **Іменування:** моделі — у PascalCase (`OrderItem`), а реальні таблиці/колонки — у snake_case через `@map`/`@@map` (наприклад, `@@map("order_items")`, `@map("user_id")`).
- **Soft-delete:** ключові сутності мають `deletedAt DateTime?` (User, Product, Brand, Category, Order, Payment, BlogPost) — рядок не видаляється фізично, а позначається; списки фільтрують `deletedAt: null`.
- **Оптимістична конкурентність:** `version Int @default(0)` на Product / Brand / Category — PUT відхиляється з 409, якщо версія не збігається (захист від одночасного редагування двома адмінами).
- **i18n:** україномовні поля — основні; англійські дублюються з суфіксом `En` (`nameEn`, `descriptionEn`, `seoTitleEn`) і є nullable — код робить fallback на uk. Slug завжди uk (URL-ідентичність).
- **Гроші** зберігаються у `Decimal(10,2)` (детально — нижче).
- **Аудит-поля:** майже всюди `createdAt @default(now())` та `updatedAt @updatedAt`.

### Multi-tenancy (tenantId)

Платформа має заготовку під **SaaS multi-tenant** режим: моделі `Tenant`, `TenantUser`, `TenantBilling`, `Plan`, `BillingInvoice`, `UsageRecord`. Прив'язка користувача до орендаря — через зв'язкову таблицю `TenantUser` (`@@unique([tenantId, userId])`), а не через колонку `tenantId` на кожній доменній моделі. Тобто більшість каталог/замовлення-таблиць наразі **спільні** (single-tenant за фактом), а tenant-шар — окрема надбудова для білінгу/доменів. Див. домен «Tenant / Multi-tenancy» та «Білінг / SaaS» нижче.

## Гроші: зберігання та арифметика

Усі грошові суми зберігаються як **`Decimal`** з фіксованою точністю — **`@db.Decimal(10,2)`** (UAH, 2 знаки після коми; для накопичувальних сум — `Decimal(12,2)`, для відсотків — `Decimal(5,2)`). **Ніколи `Float`** — IEEE-754 «з'їдає» копійки (`10.005` → `10.00`/`10.01`), що згодом вилазить як розбіжності в аудиті.

Колонки `Decimal(10,2)` клампають значення при записі, тож БД-рівень коректний за дизайном. Але щоб **зробити коректність структурною** (а не випадковою — залежною від того, чи кожен call-site не забув округлити), уся грошова математика йде через утиліту **`@/utils/money`** (`src/utils/money.ts`):

```ts
import {
  toCents,
  fromCents,
  round2,
  lineTotal,
  sumMoney,
  addMoney,
  subtractMoney,
  percentOf,
} from '@/utils/money';

// Принцип: конвертуй у цілі копійки на межі → рахуй на int (завжди точно)
// → повертай 2-dp UAH число тільки при віддачі в БД / провайдеру / UI.
toCents(100.55); // → 10055  (копійки)
fromCents(10055); // → 100.55 (UAH)
lineTotal(99.9, 3); // ціна × кількість, точно
sumMoney([10.1, 0.2]); // сума без float-дрейфу
percentOf(100.55, 10); // → 10.06 (1005.5 коп → 1006)
```

Правило: **будь-яка грошова арифметика — через `@/utils/money`**; «голий» `number * quantity` у коді заборонений. Деталі — пам'ять проєкту «Money float fix».

> Колонки з датами для фільтрів/бакетів аналітики використовують київські межі доби (`kyivMidnightUtc`/`kyivNextDayUtc`/`kyivDateIso`) — див. пам'ять «Kyiv date boundaries».

| `db:generate`     | `prisma generate`       | Згенерувати клієнт у `generated/prisma`                               |
| `db:migrate`      | `prisma migrate dev`    | Створити + застосувати міграцію **локально** (dev)                    |
| `db:migrate:prod` | `prisma migrate deploy` | Застосувати наявні міграції на **проді** (безпечно, без diff)         |
| `db:push`         | `prisma db push`        | ⚠️ Синхронізувати схему без міграції — **тільки локально/одноразово** |
| `db:seed`         | `tsx prisma/seed.ts`    | Наповнити демо-даними                                                 |
| `db:studio`       | `prisma studio`         | GUI для перегляду БД                                                  |
| `db:reset`        | `prisma migrate reset`  | ⚠️ Дроп + перестворення + seed (тільки dev)                           |

На проді деплой застосовує **`prisma migrate deploy`** (див. [19. Деплой та експлуатація](19-деплой-операції.md)).

### ⚠️ КРИТИЧНО: ніколи не `prisma db push` на проді

`Product.searchVector` оголошено як `Unsupported("tsvector")?` — це **GENERATED-колонка** для повнотекстового пошуку, разом із **GIN-індексами**. Prisma не знає, як її відтворити (тип unsupported), тож:

- **`prisma db push` на production** перезаписує колонку як звичайну (не-generated, all-NULL) і **знищує GIN-індекси** → повнотекстовий пошук по всьому сайту вмирає (запити повертають порожньо).
- Це вже траплялося один раз і лагодилося live (recreate GENERATED-колонки + індексів вручну SQL'ем). Деталі — пам'ять «Full-text search fix».

**Правило:** на проді — лише `prisma migrate deploy`. Зміни `searchVector`/GIN — тільки через явний SQL у міграції. `db push`/`db:reset` — заборонені на production.
